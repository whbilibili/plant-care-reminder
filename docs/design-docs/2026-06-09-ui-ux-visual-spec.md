## 1. 文档目标

这份文档不是“灵感说明”，而是前端视觉改造的交付规范。  
目标是让前端架构师拿到后，可以继续拆成页面任务、组件任务、状态任务和视觉回归任务。

文档解决 4 件事：

1. 明确产品在 UI 层到底应该长什么样。
2. 明确每个核心页面的信息层级、交互路径和状态反馈。
3. 明确视觉语言、组件风格和页面之间的一致性规则。
4. 明确后续任务拆解建议，避免前端只做“局部修补”。

---

## 2. 产品体验核心判断

## 2.1 这不是展示型产品

这是一个家庭内部高频使用的养护工具，不是品牌展示站，不是社区内容产品，也不是以“浏览植物资料”为主的相册应用。

因此 UI 的第一原则不是“好看得像海报”，而是：

- 一眼看懂今天该做什么
- 一步完成最常见动作
- 图片帮助识别植物
- 家庭协作动作稳定可信

## 2.2 首页必须是任务首页

根据 moodboard 的主方向 `01-task-first-dashboard.svg`，登录后的第一屏应该优先显示：

1. 今日到期任务
2. 已逾期任务
3. 一键完成
4. 与任务直接相关的植物识别信息

植物列表不是第一入口，而是第二入口。  
设置页是辅助页，不是主流程页。

## 2.3 图片是识别锚点，不是装饰

植物照片承担“快速认出是哪一盆植物”的职责。  
因此植物卡片、详情头图、上传预览都应围绕“识别效率”设计，而不是只把图片当作可有可无的点缀。

## 2.4 表单体验必须安静

植物录入、家庭创建、加入家庭都属于低频但重要动作。  
这些页面应该是“短路径、低干扰、少解释、多确认感”的 quiet flow，而不是堆很多说明文字。

## 3. 用户体验原则

## 3.1 三步原则

高频动作必须尽量在 3 步内完成：

- 从提醒进入后完成一个任务
- 登录后看到待办
- 创建家庭后拿到邀请码
- 添加植物后继续添加任务

## 3.2 首屏原则

每个页面首屏只允许出现一个主要动作。

示例：

- 待办页：完成任务
- 植物列表页：添加植物
- 创建家庭页：创建家庭
- 设置页：复制邀请码

## 3.3 家庭语言原则

文案要像中国家庭里会自然使用的话，不要像后台系统或国际化占位文案。

推荐语气：

- 温和、直接、少修辞
- 少用“shared household board”这类翻译腔
- 多用“家人”“家庭”“植物看板”“养护任务”“邀请码”

## 3.4 状态不打断原则

完成任务、复制邀请码、开启通知等动作，反馈应短、明确、可继续当前流程。

避免：

- 大面积阻断式成功页
- 看不懂下一步的错误提示
- 只提示失败，不给替代路径

## 4. 视觉系统规范

## 4.1 视觉关键词

整体风格关键词固定为：

- Quiet botanical
- Task-first
- Family utility
- Photo-led
- Mobile compact

展开后的视觉含义：

- 安静，不吵闹
- 自然，但不做装饰性插画堆砌
- 功能优先，不做营销式 Hero
- 图片承担识别职责
- 适合 iPhone 单手快速操作

## 4.2 色彩策略

> **实施基准**：色彩 Token 的具体色值和变量名以 `2026-06-09-frontend-refresh-plan.md §3.2.1` 为准。该文档已将旧的蓝色系（`--color-primary` / `--color-cta`）替换为 moodboard 叶绿系。以下为语义级使用规则。

Token 语义映射：

- `--color-leaf`：主色，用于主按钮填充、导航高亮、关键状态锚点
- `--color-gold`：CTA 强调色，仅给当前屏幕最重要的动作（完成任务、复制邀请码）
- `--color-surface`：主要卡片与表单底色
- `--color-paper`：页面整体底色
- `--color-mist`：次级区块底色、搜索栏底色
- `--color-ink`：主文字
- `--color-muted`：辅助文字
- `--color-error`：错误和危险操作
- `--color-warning`：逾期、即将到期
- `--color-success`：完成、已启用、复制成功

使用约束：

- 同一屏不要同时出现多个强饱和主按钮色块
- 逾期状态优先用轻量底色（`rgba(221,107,32,.08)`）+ 明确文案，不要全屏红色
- 页面背景保持 `--color-paper`（暖白），不改成深色工业风
- 装饰性渐变背景仅限登录页，认证后操作页面使用纯色底

## 4.3 字体策略

- 标题：保留 `Lora`
- 正文：保留 `Raleway`

使用规则：

- 登录前页面可以允许更有气质的大标题
- 登录后主流程页面标题必须收敛，避免过大
- 中文正文优先 14px 到 16px 的可读区间
- 少用全大写英文 eyebrow，中文界面里应降到最低

## 4.4 圆角与卡片

- 页面主卡片：16px 到 20px 圆角
- 图片容器：12px 到 16px 圆角
- 输入框：12px 到 14px 圆角
- 浮动主按钮或 FAB：24px 到 28px 圆角

卡片策略：

- 任务卡：信息卡，不做厚重投影
- 植物卡：图片卡，照片占比高
- 设置卡：操作卡，分区清晰
- 表单卡：安静容器，弱化装饰

## 4.5 阴影和分隔

优先级：

1. 间距
2. 分组
3. 字体层级
4. 轻背景差
5. 细边框
6. 轻阴影

不要默认所有卡片都靠大阴影分层。

## 5. 信息架构与导航规范

## 5.1 主导航

底部导航建议顺序调整为：

1. `待办`
2. `植物`
3. `设置`

理由：

- 待办是高频入口
- 植物是浏览与维护入口
- 设置是低频辅助入口

## 5.2 默认落点

已加入家庭的用户登录后，默认落到 `/todo`。

理由：

- 产品的核心价值是“提醒与完成”
- 待办页最符合“打开就知道该做什么”

## 5.3 页面层级

推荐主路径：

1. 登录
2. 称呼设置
3. 创建家庭 / 加入家庭
4. 待办首页
5. 植物列表
6. 植物详情
7. 任务编辑 / 植物编辑
8. 设置

## 6. 页面详细规范

## 6.1 登录页 `/login`

### 页面目标

- 让用户快速进入应用
- 减少解释成本
- 建立“共享家庭植物提醒工具”的第一印象

### 信息结构

从上到下：

1. 品牌或轻量标题区
2. 登录 / 注册切换
3. 邮箱
4. 密码
5. 错误提示
6. 主按钮
7. 简短辅助说明

### UI 规则

- 标题不应超过两行
- 描述不超过两行
- 表单区应尽量进入首屏
- 按钮固定为当前屏幕最强视觉动作

### 交互规则

- 点击登录后进入 loading 状态，按钮文案变更
- 错误提示出现在表单区内，不应把按钮推出视口
- 登录成功后立即进入下一流程，不停留在当前页讲故事

### 异常状态

- 账号不存在 / 密码错误：直接给可理解错误
- 网络失败：给“稍后再试”

---

## 6.2 称呼设置页 `/onboarding/profile`

### 页面目标

- 收集展示名
- 让后续任务和家庭成员列表有自然称呼

### 信息结构

1. 标题
2. 一行说明
3. 称呼输入框
4. 提交按钮

### UI 规则

- 只保留一个核心输入项
- 说明文案要简短
- 不需要额外插图

### 交互规则

- 输入后可直接提交
- 成功后进入下一步，不展示冗余成功页

---

## 6.3 家庭接入选择页 `/onboarding`

### 页面目标

- 让用户明确在两个路径中选择一个

### 信息结构

1. 标题
2. 一句说明
3. `创建家庭` 卡片
4. `加入家庭` 卡片

### UI 规则

- 两个卡片都要足够大，适合拇指点击
- 每个卡片只保留：
  - 角标
  - 标题
  - 一行描述
  - CTA

### 交互规则

- 点击卡片或按钮都可进入
- 主次明显，但不要把次路径做得太弱

---

## 6.4 创建家庭页 `/onboarding/create-family`

### 页面目标

- 创建家庭
- 让用户明确获得邀请码

### 页面状态

1. 表单态
2. 创建成功态

### 表单态 UI

- 家庭名称输入框为页面核心
- 页面底部主按钮固定为 `创建家庭`

### 成功态 UI

- 邀请码必须是视觉中心
- 文案控制在一句：`把这串邀请码发给家人即可加入`
- 下方给次一级动作：`进入植物看板`

### 交互规则

- 成功后不立即自动跳走
- 用户需要先看到邀请码，再决定下一步

---

## 6.5 加入家庭页 `/onboarding/join-family`

### 页面目标

- 让成员快速加入家庭

### 页面结构

1. 标题
2. 简短说明
3. 邀请码输入框
4. 主按钮

### 中间状态

加入成功后，可短暂显示“正在接入家庭植物看板”，然后跳转。

### 错误处理

- 邀请码错误时在输入框附近提示
- 不要把错误埋到页面底部

---

## 6.6 待办页 `/todo`

### 页面目标

- 成为真正首页
- 帮用户在最短路径内完成任务

### 信息结构

1. 页头：简短标题 + 可选一句说明
2. 已逾期分组
3. 今日到期分组
4. 即将到期分组

### 任务卡结构

每张卡片必须包含：

1. 植物名称
2. 任务动作
3. 状态标签
4. 完成按钮

可选包含：

- 小图或识别 icon
- 上次完成时间

### 视觉规则

- 第一张最紧急任务卡应最显眼
- 完成按钮必须容易点击
- 不要让一大段说明文字出现在第一屏任务上方

### 交互规则

- 点击完成后立即给反馈
- 成功后任务状态应更新，不强制页面跳转
- 如果有多个任务，用户能顺着继续完成下一项

---

## 6.7 植物列表页 `/plants`

### 页面目标

- 通过图片快速识别植物
- 进入详情或新增植物

### 页面结构

1. 标题区
2. 搜索
3. 植物列表
4. 浮动新增入口或明确主按钮

### 植物卡结构

1. 图片区
2. 植物名称
3. 位置
4. 下一任务摘要
5. 操作入口

### 视觉规则

- 图片占卡片上半部分主要面积
- 植物名要强于位置
- 任务状态用 pill 或短标签表示

### 无图与坏图规则

- 无图：显示有意设计的植物占位态
- 坏图：先尝试刷新 URL，再回落到占位态
- 不允许出现浏览器默认坏图图标

### 交互规则

- 点击卡片主体进入详情
- 编辑入口应与主点击区区分

---

## 6.8 植物详情页 `/plants/:plantId`

### 页面目标

- 查看植物资料
- 直接管理任务

### 页面结构

1. 顶部返回
2. 头图 / 植物 Hero
3. 植物核心信息
4. 任务列表
5. 编辑相关操作

### Hero 区规则

- 显示图片、植物名、位置、简短状态
- 不再堆长描述
- 描述、备注等静态信息应折叠或下置

### 任务区规则

- 任务区应尽量上移
- 每条任务要清晰展示：
  - 类型
  - 间隔
  - 下次提醒
  - 启用状态
  - 可编辑入口

---

## 6.9 设置页 `/settings`

### 页面目标

- 展示家庭协作状态
- 分享邀请码
- 管理通知
- 退出登录

### 页面结构

1. 家庭摘要
2. 邀请码主卡
3. 通知状态
4. 家庭成员
5. 退出登录

### 邀请码卡片规则

必须包含：

1. 邀请码
2. `复制邀请码`
3. 失败 fallback

### 复制失败 fallback 规范

如果系统剪贴板失败，必须立刻提供继续完成任务的路径：

- 自动展开备用区域
- 可选中邀请码文本
- 提示 `长按可复制`
- 告知可以直接转发给家人

### 成员列表规则

- 成员项展示头像占位、名字、状态
- 管理员身份可用次级文案标注
- 不要做复杂权限系统 UI

### 退出登录规则

- 页面底部独立区块
- 按钮风格与普通设置项区分
- 不隐藏，不埋深层

## 7. 关键交互与状态规范

## 7.1 Loading

文案规范：

- 简短、可理解
- 不使用过长句子

示例：

- `正在加载待办任务`
- `正在同步家庭信息`
- `正在保存植物资料`

## 7.2 Success

成功反馈不做长停留，不单开成功页，优先采用：

- 按钮文案变化
- 卡片内提示
- 列表数据即时更新

## 7.3 Error

错误提示必须包含下一步方向：

- 再试一次
- 检查邀请码
- 手动复制邀请码

禁止只显示抽象报错。

## 7.4 Empty State

空状态必须告诉用户做什么：

- 没有植物：去添加第一株植物
- 没有任务：去添加第一条养护任务
- 没有待办：说明未来几天都无需处理，并给去植物列表的入口

---

## 8. 组件级设计要求

## 8.1 按钮

### 主按钮

- 当前页面唯一最重要动作
- 使用 CTA 色
- 高度保证拇指易点击

### 次按钮

- 边框态或浅色态
- 只承担辅助动作

### 危险按钮

- 与普通按钮分区
- 不与主按钮并列争夺注意力

## 8.2 输入框

- 标签始终可见
- placeholder 不承担标签职责
- 错误提示紧贴字段

## 8.3 卡片

不要卡中套卡。  
普通列表尽量是“一个分组面 + 多行项”，而不是每行一个重卡片。

## 8.4 图片容器

- 默认优先真实照片
- 占位图需风格统一
- 图片失败态是设计态，不是浏览器默认态

---

## 9. 动效与微交互

本项目动效应少而准。

允许使用：

- 页面进入的轻微淡入
- 按钮按下反馈
- 卡片状态切换
- 复制成功的轻提示

不要使用：

- 大幅位移动画
- 长时间 loading 动画
- 装饰性转场

必须尊重 `prefers-reduced-motion`。

---

## 10. 无障碍与可用性要求

- 所有表单项都有明确中文 label
- 底部导航有准确 `aria-current`
- 所有按钮 accessible name 与可见文案一致
- 状态反馈不能只靠颜色表达
- iPhone 单手操作区优先考虑下半屏

---

## 11. 前端架构师可继续拆解的任务维度

建议后续从 4 个维度拆任务，而不是按“页面随手改”推进。

## 11.1 信息架构层

- 默认路由落点调整
- 底部导航顺序与语义调整
- 页面头部密度统一

## 11.2 组件系统层

- 主按钮 / 次按钮 / 危险按钮统一
- 表单输入与错误态统一
- 图片容器与占位态统一
- 任务卡 / 植物卡 / 设置卡分型

## 11.3 页面层

- 登录与 onboarding
- 待办页
- 植物列表与详情
- 设置页

## 11.4 状态与交互层

- 邀请码复制 fallback
- 完成任务即时反馈
- 上传图片失败回退
- 空状态与错误状态统一

## 12. 推荐拆解顺序

> **仲裁规则**：实际执行顺序以 `docs/exec-plans/modules/frontend-refresh.json` 各任务的 `priority` 字段为准。本节为设计侧建议，供架构师参考依赖关系，不作为排期依据。

1. 全量中文文案与 accessible name 清理（对应 UX-001）
2. 默认首页与底部导航信息架构调整（对应 UX-003）
3. 待办页 task-first 改造（对应 UX-003）
4. 植物列表 photo-led 改造（对应 UX-004）
5. 植物详情与任务区重排（对应 UX-004）
6. 设置页邀请码分享与退出登录优化（对应 UX-002 + UX-005）
7. onboarding / 表单流 quiet redesign（对应 UX-005）

## 13. 验收标准

前端视觉改造完成后，至少满足以下结果：

1. 登录后第一屏优先看到待办任务，而不是植物浏览。
2. 植物列表中的图片成为主要识别锚点。
3. 邀请码复制失败时，用户依然能顺利拿到邀请码并发给家人。
4. 主要页面不再残留英文主文案。
5. onboarding、待办、植物、设置四条主路径在视觉上属于同一套产品语言。
6. 任务完成、图片加载、空状态、错误状态都有清晰的统一反馈。

## 14. 与现有执行计划的关系

本方案可直接映射到现有 `frontend-refresh` 模块：

- `UX-001`：文案体系与中文语境
- `UX-002`：邀请码复制与 fallback
- `UX-003`：首页、导航、task-first 结构
- `UX-004`：植物列表、详情、图片识别

如果前端架构师继续细拆，建议每个 `UX-*` 再拆成：

- layout task
- components task
- state-feedback task
- test / visual regression task
