## 2. 改造目标

本次不是继续增加功能，而是把现有页面统一到一套可执行的视觉系统：

1. 去掉顶部白色横条贴片感，采用轻量原生导航。
2. 认证后页面回到工具型界面，不做营销式 Hero。
3. 任务优先，完成动作清晰、靠右、容易点。
4. 植物图片成为识别锚点，不再只是小装饰。
5. 设置页突出家庭协作，邀请码和邀请链接都能快速复制。
6. 登录注册保留同页双模式，不拆成两个独立页面。

## 3. 95% 还原标准

### 3.1 必须还原

- **页面结构**：每张设计稿的主要区块顺序、主操作位置、底部导航状态必须一致。
- **顶部导航**：认证后详情/表单页使用轻量导航行，不允许回到全宽白色硬条。
- **分组方式**：列表类内容使用 grouped surface + row divider，不允许卡片套卡片。
- **按钮层级**：每屏只有一个最强主按钮；设置页邀请卡中 `复制邀请链接` 为主按钮，`复制邀请码` 为次级快捷按钮。
- **登录注册模式**：`/login` 是同页 `登录 / 注册` 双模式表单，不新增独立注册路由。
- **植物识别**：植物列表和详情必须优先保留图片区域，坏图/无图要有有意设计的占位态。
- **任务动作**：待办和详情中的完成按钮保持右侧、圆形、可单手点击。

### 3.2 允许微调

- 实际中文文案可以比设计稿更准确，但不可改变主信息层级。
- 数据为空、数量不同、长文本时允许按现有数据能力收敛布局。
- 受现有组件和浏览器限制影响，像素级阴影和字体渲染可有轻微差异。
- 图片素材可用真实上传图或现有 fallback，不要求与设计稿照片一致。

### 3.3 不可接受偏差

- 顶部导航再次出现突兀白色矩形横条。
- 页面首屏被大标题和说明文案占满，主任务/主表单不可见。
- 主要列表项被拆成大量独立浮卡，导致信息密度过低。
- 设置页只提供邀请链接复制，缺少快捷复制邀请码。
- 登录注册拆成两个页面或引入额外跳转。

## 4. 共享组件规划

### 4.1 `ScreenNav`

用途：详情页、表单页和设置子流程的轻量顶部导航。

结构：

- 左侧：44px icon-only back button。
- 中间：单行标题，最大宽度约 55%-60%，超长省略。
- 右侧：可选 icon button / text action / spacer。

视觉：

- 背景默认透明或 `var(--color-paper)`，不画全宽白色卡片。
- 高度 48-56px。
- 无底部强边框；必要时使用极浅分割线或滚动后弱阴影。

验收：

- 在植物详情和任务编辑页不出现“白条贴片”。
- 返回按钮可访问名为 `返回`。

### 4.2 `GroupedSurface`

用途：待办分组、养护计划、设置分组、家庭接入选项。

结构：

- 外层一个 surface。
- 标题行 + 可选 action。
- 子项使用 row divider。

视觉：

- `background: var(--color-surface)`。
- `border: 1px solid var(--color-line)`。
- `border-radius: var(--radius-card)` 或 18-20px。
- 优先使用分割线，不给每一行再套卡片。

验收：

- 同一 surface 内行项目不再各自有大阴影。
- 不出现 card inside card。

### 4.3 `ObjectSummaryBand`

用途：植物详情顶部对象摘要、任务编辑表单上下文、家庭摘要。

结构：

- 左侧缩略图/头像。
- 中间主标题 + 次要信息 + 状态。
- 右侧可选编辑/操作按钮。

验收：

- 植物详情的页面标题和摘要标题不重复制造层级冲突。
- 摘要区高度紧凑，不能挤压下面任务区。

### 4.4 `TaskActionRow`

用途：待办页、详情页需要处理区、计划列表可编辑行。

结构：

- 左侧任务类型图标或植物缩略图。
- 中间任务名 + 植物名/周期/到期文案。
- 右侧状态文案 + 完成按钮或 chevron。

验收：

- 完成按钮在高频任务场景可直接点击。
- 行内长文本不撑宽，不遮挡按钮。

### 4.5 `InviteCard`

用途：设置页邀请家人加入。

结构：

- 标题 `邀请家人加入`。
- 邀请码大号 monospace 展示。
- 次级按钮 `复制邀请码`。
- 主按钮 `复制邀请链接`。
- 失败/手动面板 `手动查看邀请码`。

验收：

- 两个复制动作同时存在，视觉层级不同。
- 复制失败时不只显示错误，必须提供手动复制路径。

## 5. 页面改造方案

## 5.1 登录/注册 `/login`

目标：同一页面完成登录和注册，不拆路由。

结构：

1. 轻氛围背景，只在 auth 页使用渐变。
2. 顶部品牌区：`植物养护` + 一行说明。
3. 表单 surface：`登录 / 注册` segmented control。
4. 登录模式：邮箱、密码、登录按钮、切换注册提示。
5. 注册模式：同一组邮箱/密码字段，主按钮变为 `创建账号`，必要提示内联展示。
6. 邀请链接上下文提示：收到邀请时登录或注册后继续加入家庭。

验收：

- 375px 宽度首屏内必须看到输入框和主按钮。
- 登录/注册切换不发生路由跳转。
- 不出现两套并列表单。

## 5.2 家庭接入 `/onboarding`

目标：让用户只做一个判断：创建家庭或加入家庭。

结构：

1. 标题 `加入家庭空间`。
2. 简短说明。
3. `创建新家庭` 选项 surface。
4. `加入家人的家庭` 选项 surface。
5. 可选显示当前称呼。

验收：

- 两个选择权重清晰，但不做厚重卡片。
- 主路径按钮不超过两个。

## 5.3 待办首页 `/todo`

目标：登录后的任务首页，首屏可直接完成任务。

结构：

1. 顶部标题 `今日待办` + 日期。
2. 紧凑状态条：待照顾植物数 + 完成进度。
3. `逾期` 分组。
4. `今天` 分组。
5. `即将到期` 轻列表。
6. 底部导航：`待办` active。

验收：

- 任务行右侧完成按钮清晰。
- 首屏不被说明文案占据。
- 空状态仍保留下一步动作。

## 5.4 植物看板 `/plants`

目标：照片驱动的植物识别看板。

结构：

1. 顶部标题 `植物看板` + `+ 添加`。
2. 搜索栏。
3. 植物卡列表：图片、名称、位置、下次养护状态。
4. 归档区保持低层级。
5. 底部导航：`植物` active。

验收：

- 有图时图片是卡片视觉锚点。
- 无图/坏图有占位态，不是空白块。
- 植物卡支持整卡进入详情。

## 5.5 植物详情 `/plants/:plantId`

目标：解决顶部贴片感，并把任务操作上移。

结构：

1. `ScreenNav`：返回、标题、编辑入口。
2. `ObjectSummaryBand`：照片、名称、位置、状态。
3. `需要处理` grouped surface。
4. `养护计划` grouped surface + `+ 添加`。
5. 植物档案/归档/删除等低频动作放低层级。

验收：

- 顶部没有全宽白色硬条。
- `需要处理` 在首屏内可见。
- 编辑植物入口存在但不抢完成任务层级。

## 5.6 安静表单模板

覆盖：

- `/plants/new`
- `/plants/:id/edit`
- `/plants/:id/tasks/new`
- `/plants/:id/tasks/:taskId/edit`

目标：表单不再使用巨大标题卡片，改成轻导航 + 上下文摘要 + 紧凑字段。

结构：

1. `ScreenNav`：返回、标题、保存动作。
2. 上下文摘要：植物/任务名称。
3. 表单字段 grouped surface。
4. 预览/状态 section。
5. 危险操作独立放底部。

验收：

- 表单首屏优先显示字段，而不是大标题说明。
- 保存动作可见但不压迫。
- 删除任务/删除植物与普通字段视觉分离。

## 5.7 设置页 `/settings`

目标：家庭协作操作页。

结构：

1. 顶部 `设置` + 当前用户角色。
2. 家庭摘要。
3. `InviteCard`：邀请链接和邀请码复制。
4. 通知权限。
5. 家庭成员。
6. 退出家庭/退出登录。
7. 底部导航：`设置` active。

验收：

- `复制邀请链接` 为主按钮。
- `复制邀请码` 快捷按钮可见。
- 复制失败显示手动查看/复制备用路径。
- 退出动作与普通设置项分离。

## 6. 开发顺序

推荐按依赖顺序执行：

1. `UI-SYS-001`：建立共享布局基元与视觉 QA 基线。
2. `UI-SYS-002`：登录/注册 v2。
3. `UI-SYS-003`：家庭接入。
4. `UI-SYS-004`：待办首页。
5. `UI-SYS-005`：植物看板。
6. `UI-SYS-006`：植物详情。
7. `UI-SYS-007`：安静表单模板。
8. `UI-SYS-008`：设置页 v2。
9. `UI-SYS-009`：跨页面截图 QA 与 95% 还原修正。

顺序理由：

- 先做共享基元，避免每页各自写一套。
- 先做登录和家庭接入，验证 auth/onboarding 基础视觉。
- 再做待办和植物主路径，验证任务和照片系统。
- 详情与表单依赖 `ScreenNav` 和 `ObjectSummaryBand`。
- 设置页依赖 `GroupedSurface` 和 `InviteCard`。

### 还原验收

对照设计稿检查：

- 页面区块顺序一致。
- 主操作位置一致。
- 顶部导航符合轻量原生导航。
- grouped surface 不出现卡片套卡片。
- 任务完成按钮、植物图片、复制邀请码按钮均可见。

## 8. 风险与处理

- **设计稿文字不是代码事实**：实现以当前业务文案和数据字段为准，但层级不变。
- **部分页面已有复杂状态**：先完成主态，再补空/错/加载态。
- **现有测试可能绑定旧文案**：UI 改造时同步更新 accessible name 与测试。
- **图片加载不稳定**：实现时必须保留 StorageImage/fallback 现有可靠性能力。
- **一次性改动过大**：每个任务独立提交，避免跨页面大爆炸。

## 9. Definition of Done

本轮 UI 改造完成时应满足：

1. 七张主设计稿对应页面均有实现。
2. `ScreenNav` / `GroupedSurface` / `ObjectSummaryBand` / `InviteCard` 至少完成代码级复用。
3. 主路径 375px 截图与设计稿结构 95% 对齐。
4. 登录/注册仍为同页双模式。
5. 设置页同时支持复制邀请链接和复制邀请码。
6. 植物详情顶部无白条贴片感。
7. 验证命令执行并记录结果。
