# 设置页产品设计细化（迭代 v0.3）

**版本**：v0.3（迭代设计）
**日期**：2026-06-15
**关联文档**：承接 [2026-06-12-settings-page-iteration-spec.md](./2026-06-12-settings-page-iteration-spec.md)（v0.2，已落地 SET2-001~012），基于其确立的「个人 / 家庭 / 通知与应用」三组信息架构继续扩展
**文档定位**：v0.2 把设置页从「家庭信息展示页」升级为「设置中心」，但仍有四块能力缺口。本次 v0.3 聚焦补齐这四项**真实闭环能力**：个人头像、家庭改名、邀请链接拉新、成员主动退出。本文档只做需求评估与产品设计拍板，不重复 v0.2 已落地内容，UI 细节由配套 UI Spec 承接。

---

## 0. 本次迭代要解决的真实问题（JTBD 视角）

v0.2 之后，用户在设置页仍会撞到四堵墙：

> - **"我想换个头像，让家人一眼认出我"** —— 现在成员列表全是首字母色块，分不清谁是谁。
> - **"家庭名当初随手起的，现在想改"** —— 改不了，创建时定死。
> - **"我想把家人拉进来，但让对方手输一串码太麻烦"** —— 只能口头报 6 位邀请码，转化路径长、易出错。
> - **"我误加入了别人的家庭 / 不想待在这个家庭了"** —— 只能等管理员移除，自己走不了。

这四件事分别对应设置页的两大核心价值——**自我掌控**（头像、退出）与**协作入口**（改名、邀请链接）。补齐后，设置页才算真正闭环。

## 1. 需求评估总览

### 1.1 四项需求一览与优先级

| # | 需求 | 用户价值 | 工程量 | 复用度 | 优先级 | 风险 |
|---|------|---------|--------|--------|--------|------|
| ① | 个人头像上传 / 更换 | 中高（成员识别度） | 中 | 高（植物图片 storage 链路现成） | **P1** | 低 |
| ② | 管理员修改家庭名称 | 中（纠错预期） | 低 | 极高（照搬现有权限+弹窗模式） | **P0** | 极低 |
| ③ | 邀请链接注册即加入 | 高（拉新转化） | **高** | 后端高 / 前端低（自研路由需改造） | **P1** | 中高 |
| ④ | 成员主动退出家庭 | 中高（自我掌控兜底） | 中 | 中（参考移除成员） | **P0** | 中（家庭无主问题） |

**优先级说明**：
- ② 改家庭名、④ 退出家庭定为 **P0**：前者是最基础的纠错预期且改动极小（性价比最高），后者补齐"自我掌控"链条上的关键缺口（v0.2 只做了"管理员移除别人"，缺"自己能走"），且涉及数据一致性决策，越早拍板越好。
- ① 头像、③ 邀请链接定为 **P1**：① 是体验增强、非阻塞性；③ 价值最高但工程量也最大（涉及自研路由改造 + 跨多跳状态保留），需要单独排期，建议在 ②④ 落地后启动。

### 1.2 与 v0.2 信息架构的衔接

四项需求全部挂载在 v0.2 已确立的三组结构内，**不改动信息架构骨架**：

```
设置（h1）
│
├── 个人
│     ├── 我的头像              ← ① 新增（可上传/更换，圆形预览）
│     ├── 我的称呼（已有，可编辑）
│     ├── 我的账号（邮箱，只读）
│     ├── 退出家庭              ← ④ 新增（破坏性，二次确认）
│     └── 退出登录（已有）
│
├── 家庭
│     ├── 家庭摘要（名称 + 成员数）  ← ② 名称行变为管理员可编辑
│     ├── 邀请分享                ← ③ 新增「复制邀请链接」（与现有「复制邀请码」并存）
│     │     ├── 邀请码（已有）
│     │     └── 重置邀请码（已有）
│     └── 家庭成员管理（已有）
│
└── 通知与应用（沿用 v0.2，本次不动）
```

## 2. 需求①｜个人头像上传 / 更换（P1）

### 2.1 产品语义

头像和称呼一样，是家庭成员之间相互识别的标签。当前成员列表用「首字母 + hash 取色块」（`MemberAvatar`）兜底，多人时辨识度低。允许用户上传真实头像，提升家庭空间的"人味"和识别效率。

### 2.2 能力边界

| 项 | 规则 |
|----|------|
| 入口 | 「个人」组首项「我的头像」，点击进入选择/上传 |
| 上传方式 | 从相册选图（沿用植物图片的文件选择），支持拍照取决于浏览器 file input 能力 |
| 预览 | 圆形预览（区别于植物封面的圆角矩形） |
| 格式兼容 | 复用 `normalizeImageFile.ts`：自动把 iOS 的 HEIC/HEIF 转 JPEG |
| 上传中 | 复用现有"上传中 / 已保存 / 错误"三态反馈 |
| 同步 | 保存后，成员列表中"我"的头像、待办页等所有展示头像处实时更新 |
| 删除/恢复 | 支持"移除头像"回退到首字母色块（MVP 可选，见 §2.5） |

### 2.3 技术评估（复用现成 storage 链路）

植物图片已是一套成熟的 Convex storage 链路，可大比例复用：

- **可直接复用**：`uploadPlantImage.ts`（取直传 URL → 直传 → 拿 storageId）、`normalizeImageFile.ts`（HEIC 转码）、`StorageImage.tsx`（按 storageId 渲染，含 HEIC 解码失败兜底）——这些与"家庭"无耦合。
- **需要新增**：
  1. users 表增加 `imageStorageId: v.optional(v.id("_storage"))` 字段（注意：authTables 自带的 `image` 字段语义是 URL，会过期，**不复用它**，统一走 storageId 方案与植物保持一致）。
  2. 后端新增**用户级**接口：`users.generateAvatarUploadUrl`（仅校验登录，**不带** `requireCurrentFamilyMember` 家庭门槛）、`users.getAvatarUrl`（同理）。原因：头像是用户级能力，不应依赖家庭归属（用户在 no-family 状态也该能设头像）。
  3. `updateMyProfile` 增加 `imageStorageId` 参数（或新增 `updateMyAvatar` mutation）。
  4. 改造 `MemberAvatar.tsx`：有 `imageStorageId` 时渲染真实头像，否则回退现有首字母色块。
  5. `getFamilySettingsSummary` 的成员对象补充 `imageStorageId`，供成员列表渲染头像。

> **决策点 D1**：`StorageImage` 当前硬编码调 `api.plants.getPlantImageUrl`（带家庭门槛）。头像复用时需把它**参数化**（传入取 URL 的查询）或新增 `getAvatarUrl` 走通用渲染。建议参数化，避免重复组件。

### 2.4 边界与异常

| 场景 | 处理 |
|------|------|
| 大图 / 超大文件 | 建议加客户端尺寸/体积上限（如长边 ≤1024px、≤5MB），超限给提示。MVP 可先只限体积 |
| HEIC | 自动转 JPEG（现成能力） |
| 上传失败 | 行内错误态，不静默；保留原头像 |
| 解码失败（坏图） | `StorageImage` 已有 naturalWidth===0 兜底，回退首字母色块 |

### 2.5 MVP 取舍

- **做**：上传 / 更换头像、圆形预览、HEIC 兼容、成员列表实时同步。
- **本次可不做（记入后续）**：头像裁剪框、移除头像回退色块、上传进度条百分比。若工期紧，"移除头像"可放到下一迭代。

## 3. 需求②｜管理员修改家庭名称（P0）

### 3.1 产品语义

家庭名当前在 `createFamily` 时定死，之后无任何修改入口。这是最基础的纠错预期落空。本次让**管理员**可改家庭名。

### 3.2 能力边界

| 项 | 规则 |
|----|------|
| 谁能操作 | 仅 `role === "admin"` 成员；普通成员看到的家庭名为只读，**不渲染**编辑入口（不置灰） |
| 入口 | 「家庭」组家庭摘要卡，家庭名行变为可点击（管理员视角），点击弹出底部编辑 sheet |
| 约束 | 非空、trim、长度上限（建议 ≤20 字，与创建时一致） |
| 反馈 | 保存成功即时更新页面所有家庭名展示处；失败行内错误态 |
| 二次确认 | 改名非破坏性，**无需**二次确认，直接编辑保存（与改称呼一致） |

### 3.3 技术评估（纯增量，最稳）

这是四项里最低成本、最低风险的：

- **后端**：新增 `families.renameFamily` mutation，**照搬 `resetInviteCode` 的 admin 权限校验模式**（校验登录 → 校验有家庭 → 取 currentMembership 判 `role !== "admin"` 抛错），再 trim + 非空校验后 `ctx.db.patch(familyId, { name })`。
- **前端**：**照搬 `NicknameEditSheet` 模式**（可点击 SettingRow + ConfirmSheet 内嵌 InputField），克隆为 `FamilyNameEditSheet`，把 `updateMyProfile` 换成 `renameFamily`。仅管理员渲染该可点击行。

### 3.4 与测试基线的衔接

v0.2 的 smoke 测试断言"家庭名用 `getByText` 而非 heading"（家庭名已从 h1 降级为头图卡内信息）。改名功能若把家庭名做成可点击编辑，需确认不破坏该断言（可点击元素内文本仍可被 `getByText` 命中，预计无影响，落地时验证）。

## 4. 需求③｜邀请链接注册即加入（P1，工程量最大）

### 4.1 产品语义

当前拉新链路：管理员口头/转发 6 位邀请码 → 被邀人自己注册 → 进 onboarding 选"加入家庭" → 手输邀请码。**链路长、易输错、转化漏斗大**。目标：管理员生成一条**邀请链接**，被邀人点链接 → 完成注册 → **自动加入该家庭**，跳过手输邀请码这一步。

> **范围约束（用户明确）**：本次仍坚持「**一个人只加入一个家庭**」。邀请链接只是"加入家庭"的便捷入口，不改变单一家庭约束。

### 4.2 用户流程（被邀人视角）

```
点击邀请链接（带邀请码，如 /join/QWNAE3 或 ?invite=QWNAE3）
   │
   ├─ 已登录且已有家庭 → 提示"你已在某个家庭中"，不重复加入（单一家庭约束）
   │
   ├─ 已登录、有称呼、无家庭 → 直接用链接里的邀请码加入 → 进 /todo
   │
   └─ 未登录 → 暂存邀请码 → 引导注册/登录 → 填称呼 → 自动用暂存邀请码加入 → 进 /todo
                （全程无需手输邀请码）
```

### 4.3 技术评估（后端可复用，前端路由是瓶颈）

- **后端可直接复用**：`joinFamilyByInviteCode` mutation 已完整（normalize、查家庭、校验未重复加入、校验 `familyId === null` 单一家庭约束）。**不需要新后端逻辑**，邀请链接只是换一种"把邀请码喂给这个 mutation"的方式。
- **前端是主要工程量**，三个改造点：
  1. **路由层**：当前是自研极简路由（`router.tsx` 的 `normalizePath` 用硬编码白名单），**不解析 query 参数、也无 `/join/:code` 动态段**。需要扩展路由以承载邀请链接（方案见 §4.4）。
  2. **跨多跳保留邀请码**：未登录用户点链接后会被 `RouteGate` 重定向到 `/login`，经历 登录/注册 → 填称呼 → onboarding 多跳，**邀请码必须跨这几跳存活**。建议用 `sessionStorage` 暂存（标签页级、关闭即清，避免污染）。
  3. **onboarding 自动加入**：到达 onboarding 阶段时，若 sessionStorage 有暂存邀请码，自动调 `joinFamilyByInviteCode` 并清除暂存，跳过"创建/加入家庭"二选一页。
- **邀请链接生成与分享**：在 `InviteCodeCard` 现有"复制邀请码"旁新增"复制邀请链接"按钮，拼接 `window.location.origin + /join/CODE`（或 `?invite=CODE`）。沿用现有 clipboard 复制 + 长按兜底逻辑。

### 4.4 待拍板决策

> **决策点 D2 — 链接格式**：`/join/QWNAE3`（path 段，更友好可读）vs `/login?invite=QWNAE3`（query 参数，路由改造更小）。
> **推荐**：`/join/:code` 路径段。语义清晰、可读性好，且能作为独立"邀请落地页"承载（未登录展示"XX 邀请你加入『家庭名』"的引导，提升转化）。代价是 `normalizePath` 要新增一条动态段解析。

> **决策点 D3 — 已有家庭者点链接的处理**：单一家庭约束下，已在家庭中的人点了别人的邀请链接，应给明确提示（"你已加入『当前家庭名』，需先退出才能加入新家庭"），而非静默失败或报底层错。结合需求④（退出家庭），可引导其退出后再加入。

> **决策点 D4 — 邀请码安全性**：当前邀请码永久有效（v0.2 已提供"重置邀请码"作撤销手段）。链接化后转发更容易，是否需要给链接加有效期/一次性？**MVP 建议不做**（沿用"可重置"作为兜底），记入后续候选。

### 4.5 边界与异常

| 场景 | 处理 |
|------|------|
| 链接邀请码无效/已重置 | 落地页/加入时提示"邀请链接已失效，请向家人索取新链接" |
| 已登录已有家庭 | 提示已在家庭中，不重复加入（D3） |
| 注册中途放弃 | sessionStorage 暂存随标签页关闭清除，不残留 |
| 链接被多人使用 | 允许（邀请码本就可多人加入），单一家庭约束在 `joinFamilyByInviteCode` 内部保证 |

## 5. 需求④｜成员主动退出家庭（P0）

### 5.1 产品语义

v0.2 只做了"管理员移除别人"，且 `removeMember` **显式禁止移除自己**。缺口：成员（误加入 / 不想待了）无法主动离开。本次补齐"自己退出家庭"。

### 5.2 能力边界

| 项 | 规则 |
|----|------|
| 入口 | 「个人」组「退出家庭」行（破坏性视觉，区别于普通行） |
| 谁能操作 | 所有成员都能退出**自己**（含普通成员、管理员） |
| 二次确认 | 破坏性操作，**必须二次确认**，文案点名后果："退出后你将离开『家庭名』，养护数据仍归这个家庭所有，需重新邀请才能回来" |
| 退出效果 | 删除自己的 `familyMembers` 记录 → 回到"创建/加入家庭"状态；级联删除自己在该家庭的 `pushSubscriptions`；**保留** `taskCompletionLogs`（养护历史属于家庭资产，参考 v0.2 §7.1） |
| 退出后体验 | 下次进应用回到 onboarding 加入/创建家庭流程，不报错崩溃 |

### 5.3 技术评估

- **后端**：新增 `families.leaveFamily` mutation。级联逻辑**参考 `removeMember`**：删自己的 pushSubscriptions（`by_familyId_and_userId` 索引 collect 后逐条删）、保留 taskCompletionLogs、删自己的 familyMembers 记录。
- **前端**：「个人」组新增「退出家庭」破坏性行 + 复用 `ConfirmSheet`（`variant="danger-solid"`）二次确认。

### 5.4 关键决策：创建者 / 唯一管理员退出怎么办（必须拍板）

现有数据模型**没有任何机制保证家庭始终有至少一个 admin**，三种危险情形：

1. **创建者退出** → `family.createdBy` 仍指向已退出用户，"创建者"标记失效、家庭无主。
2. **唯一管理员退出** → 家庭无 admin，所有 admin-only 操作（改名、移除、重置邀请码）永久无人可执行。
3. **最后一名成员退出** → families/plants/tasks 全部残留为无人可见的空壳。

> **决策点 D5 — 退出策略（推荐方案，请确认）**：
>
> | 情形 | 推荐处理 |
> |------|---------|
> | 普通成员退出 | 直接退出，无额外限制 |
> | 管理员退出，但家庭还有其他 admin | 直接退出 |
> | 管理员退出，但 TA 是**唯一管理员**且家庭还有其他成员 | **阻止退出，提示"请先指定其他成员为管理员"**——但 v0.2/v0.3 暂无"转让管理员"功能 → **降级方案：自动把最早加入的其他成员提升为 admin 后再放行退出** |
> | 任何人退出后家庭**只剩 ta 自己**（即最后一人退出） | **连同删除整个家庭及其 plants/plantTasks/taskCompletionLogs/pushSubscriptions**（避免空壳脏数据） |
> | 创建者退出但家庭还有其他人 | 退出放行，但需更新 `createdBy` 或解除"创建者不可移除"对已退出者的引用（建议把 createdBy 转移给被提升的 admin，或弱化 createdBy 的语义） |
>
> **最简 MVP 兜底（若想压缩工期）**：(a) 唯一管理员/创建者**禁止直接退出**，提示"作为家庭管理员，暂不能直接退出（后续版本将支持转让）"——把"转让管理员"推到后续；(b) 最后一人退出则删除整个家庭。这套最简单、无需新增转让能力，但牺牲了创建者的退出自由。
>
> **✅ 已拍板（2026-06-15）**：采用"最简兜底"——**唯一管理员/创建者禁止直接退出**（提示"作为家庭管理员，暂不能直接退出，后续版本将支持转让"）+ **最后一人退出则删除整个家庭及其全部数据**。"自动提升 admin / 转让管理员"推迟到 v0.4。

### 5.5 与单一家庭约束的联动

退出家庭后用户回到 `familyId === null` 状态，可重新加入/创建。这与需求③（邀请链接）天然衔接：**"误入家庭"的完整闭环 = 退出当前家庭（④）→ 点新邀请链接加入（③）**。

## 6. 全局约束与红线（四项需求都要遵守）

### 6.1 单一家庭硬约束（架构级红线）

`lib/auth.ts` 的 `getCurrentUserContext` 用 `.unique()` 取用户的 familyMembers 记录——**一旦某用户有 ≥2 条 membership，该函数直接抛异常，会导致这个用户几乎所有功能瘫痪**（路由、植物、任务、家庭设置全挂）。因此：

- 所有**写入 membership 的新路径**（需求③邀请链接加入）必须先校验 `familyId === null`（`joinFamilyByInviteCode` 已有此校验，复用即可，不得绕过）。
- 需求④退出后再加入也必须保证退出时干净删除旧 membership，避免残留制造脏数据。
- **建议附带加固**（可记入技术债或本次顺带）：把 `.unique()` 改为 `.first()` 做防御，避免脏数据"一炸炸全站"。

### 6.2 设计令牌纪律

所有新增 UI（头像卡、改名 sheet、退出家庭行、邀请链接按钮、邀请落地页）的色值/间距/圆角/阴影**一律引用 `tokens.css` 令牌，禁止硬编码 #hex**。

## 9. 优先级建议与拆分

建议分两批落地，先稳后大：

- **第一批（P0，性价比高、风险低）**：② 家庭改名 + ④ 退出家庭。两者都改动可控，且 ④ 的退出能力是 ③ 邀请链接闭环的前置（误入家庭→退出→重新加入）。
- **第二批（P1）**：① 头像（体验增强，独立）+ ③ 邀请链接（价值最高但工程量最大，含自研路由改造，单独排期）。

## 11. 决策汇总（最终结论，已确认 2026-06-15）

本节为开发与 UI Spec 的唯一依据。D1~D6 全部确认如下：

| 编号 | 最终结论 |
|------|---------|
| **D1** | `StorageImage` 取 URL 查询参数化复用，不重复造组件 |
| **D2** | 邀请链接采用 `/join/:code` 路径段格式，可承载独立邀请落地页 |
| **D3** | 已有家庭者点邀请链接 → 明确提示"已在『当前家庭名』中，需先退出才能加入新家庭"，引导走需求④ |
| **D4** | 邀请链接有效期 / 一次性链接本次不做，沿用"重置邀请码"作撤销兜底，记入后续候选 |
| **D5** | **唯一管理员/创建者禁止直接退出**（提示后续支持转让）；**最后一人退出则删除整个家庭及其 plants/plantTasks/taskCompletionLogs/pushSubscriptions**；普通成员、非唯一管理员可直接退出。"转让管理员/自动提升 admin"放 v0.4 |
| **D6** | users 表新增 `imageStorageId: v.optional(v.id("_storage"))`，**不复用** authTables 自带的 `image`（URL 语义、会过期），统一走 storageId 方案与植物保持一致 |

**落地拆分（已定）**：第一批 P0 = ②家庭改名 + ④退出家庭；第二批 P1 = ①头像 + ③邀请链接（含自研路由改造，单独排期）。
