# 数据导出

**版本**：v1.0
**日期**：2026-06-22
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（§6 数据模型）、[养护历史时间线](./2026-06-17-care-history-timeline-spec.md)（导出日志依赖完成记录）
**文档定位**：为用户提供家庭养护数据的完整导出能力，支持 CSV 格式的养护日志导出和 JSON 格式的全量数据备份，满足数据分析和数据主权需求。

---

## 0. 本次迭代要解决的核心问题

用户的养护数据全部存储在云端（Convex），没有任何方式将数据取出。这导致两个体验缺口：

- **"养了一年的数据，万一服务挂了就全没了"** —— 数据安全焦虑：专业植物爱好者积累了大量养护记录，但无法备份，对数据持久性缺乏信心。
- **"想分析一下哪盆植物浇水最频繁，但没法导出数据做表格"** —— 数据分析需求：进阶用户想用 Excel 或其他工具分析养护规律，但数据被锁在应用内。

本次通过提供**结构化数据导出**能力，让用户拥有自己数据的完整控制权。

---

## 1. 产品目标

让用户能够一键导出家庭的全部养护数据：养护日志以 CSV 格式导出（方便 Excel 分析），完整数据以 JSON 格式导出（方便备份和迁移）。导出过程简单、安全、隐私合规。

---

## 2. 用户场景

### 场景 1：导出养护日志做分析

用户是一位植物爱好者，养了 15 盆植物半年了。她想知道哪盆植物最"费水"、施肥频率是否合理。她在设置页点击"导出数据"，选择"养护日志（CSV）"，下载后用 Excel 打开，按植物名筛选、按月份统计，得到清晰的养护频率分析。

### 场景 2：全量备份

用户担心数据丢失，每月定期在设置页点击"导出数据"，选择"完整备份（JSON）"，下载一份包含所有植物信息、任务配置和完成记录的 JSON 文件，存到自己的网盘里。

### 场景 3：家庭迁移

用户的家庭要从一个账号体系迁移到另一个（未来可能的场景）。通过 JSON 全量导出，保留了所有历史数据的结构化记录，为未来的数据导入提供基础。

## 3. 功能设计

### 3.1 导出入口

**位置**：设置页 → "应用与通知"分组 → "导出数据"选项。

**交互流程**：

1. 用户点击"导出数据"，弹出 ActionSheet，展示两个选项：
   - "养护日志（CSV）" —— 适合用 Excel 分析
   - "完整备份（JSON）" —— 适合数据备份
2. 用户选择一种格式后，按钮变为 loading 态，文案"正在生成..."。
3. 生成完成后，自动触发浏览器下载（或 PWA 的文件保存）。
4. 下载成功后 toast 提示"导出成功"。

### 3.2 CSV 导出 —— 养护日志

**文件名**：`plant-care-log-{familyName}-{YYYY-MM-DD}.csv`

**列定义**：

| 列名 | 说明 | 示例 |
|------|------|------|
| 完成时间 | 本地时区格式化 | 2026-06-15 08:30 |
| 植物名称 | 植物名 | 绿萝 |
| 植物位置 | 植物 location | 客厅窗台 |
| 任务类型 | 中文标签 | 浇水 |
| 自定义任务名 | 仅 custom 类型有值 | 擦叶片 |
| 间隔天数 | 完成时的间隔配置 | 3 |
| 完成人 | 成员昵称 | 妈妈 |

**排序**：按完成时间降序（最新在前）。

**编码**：UTF-8 with BOM（确保 Excel 正确识别中文）。

**数据范围**：当前家庭的全部 `taskCompletionLogs`，不限时间范围。

### 3.3 JSON 导出 —— 完整备份

**文件名**：`plant-care-backup-{familyName}-{YYYY-MM-DD}.json`

**数据结构**：

```json
{
  "exportVersion": "1.0",
  "exportedAt": "2026-06-22T10:30:00.000Z",
  "family": {
    "name": "王家",
    "createdAt": "2026-01-15T08:00:00.000Z"
  },
  "members": [
    {
      "displayName": "妈妈",
      "role": "owner",
      "joinedAt": "2026-01-15T08:00:00.000Z"
    }
  ],
  "plants": [
    {
      "name": "绿萝",
      "location": "客厅窗台",
      "description": "水培绿萝",
      "notes": "喜欢散射光",
      "isArchived": false,
      "createdAt": "2026-01-20T10:00:00.000Z",
      "tasks": [
        {
          "taskType": "watering",
          "intervalDays": 3,
          "enabled": true,
          "lastCompletedAt": "2026-06-20T08:00:00.000Z",
          "nextDueAt": "2026-06-23T08:00:00.000Z"
        }
      ],
      "completionLogs": [
        {
          "taskType": "watering",
          "completedBy": "妈妈",
          "completedAt": "2026-06-20T08:00:00.000Z",
          "intervalDays": 3
        }
      ]
    }
  ]
}
```

**隐私规则**：

- 不导出成员邮箱、手机号等个人身份信息。
- 成员信息只包含 displayName 和 role。
- 不导出 Convex 内部 ID（_id），使用人类可读的名称关联。
- 不导出推送订阅信息。
- 不导出图片文件本身（只导出结构化数据）。

### 3.4 技术实现路径

**后端（Convex action）**：

1. 用户触发导出 → 前端调用 `exportFamilyData` action。
2. Action 内部查询当前家庭的所有数据（plants + plantTasks + taskCompletionLogs + familyMembers + users）。
3. 根据请求的格式（csv/json）组装数据。
4. 将生成的文件内容写入 Convex `_storage`。
5. 返回 storage URL 给前端。
6. 前端拿到 URL 后触发下载。

**为什么用 action 而非 query**：

- 导出可能涉及大量数据聚合（数百条 logs），action 没有 query 的执行时间限制。
- Action 可以写入 `_storage`，query 不能。

**文件生命周期**：

- 导出文件在 `_storage` 中保留 24 小时后自动清理（通过 scheduled function）。
- 每次导出生成新文件，不缓存旧文件。

## 4. 数据模型变更

### 4.1 无 schema 变更

本功能不需要修改任何现有表结构。导出是纯读取操作 + 临时文件生成。

### 4.2 新增辅助表（可选）

如果需要追踪导出历史（用于限流或审计），可新增：

```
exportLogs: defineTable({
  familyId: v.id("families"),
  userId: v.id("users"),
  format: v.union(v.literal("csv"), v.literal("json")),
  fileStorageId: v.id("_storage"),
  exportedAt: v.number(),
})
```

MVP 阶段可不建此表，直接导出不记录。后续如需限流（如每天最多导出 3 次）再引入。

## 5. 新增接口

| 接口 | 类型 | 权限 | 行为 |
|------|------|------|------|
| `exportFamilyData` | action | 家庭成员 | 聚合当前家庭全部数据，按指定格式生成文件，存入 _storage，返回下载 URL |
| `cleanupExpiredExports` | scheduled mutation | 系统 | 每小时运行，删除超过 24 小时的导出文件 |

### exportFamilyData 参数

```
args: {
  format: v.union(v.literal("csv"), v.literal("json"))
}
returns: {
  downloadUrl: string,
  fileName: string,
  recordCount: number,  // 导出的记录条数（CSV 为 log 行数，JSON 为植物数）
}
```

## 6. 前端交互设计

### 6.1 设置页入口

- 在"应用与通知"分组中，新增"导出数据"行。
- 左侧图标：Download（Lucide），颜色 `--color-ink`。
- 右侧 chevron 指示可点击。

### 6.2 格式选择 ActionSheet

- 标题："选择导出格式"。
- 两个选项卡片，各含标题 + 一行描述：
  - "养护日志（CSV）" / "适合用 Excel 打开分析养护规律"
  - "完整备份（JSON）" / "包含所有植物、任务和记录的完整数据"
- 底部"取消"按钮。

### 6.3 导出进度

- 选择格式后，ActionSheet 关闭，设置页的"导出数据"行变为 loading 态（spinner + "正在生成..."）。
- 生成完成后恢复正常态，同时触发下载。
- 如果生成失败，toast 提示"导出失败，请稍后重试"。

### 6.4 下载触发

- PWA 环境下使用 `<a download>` + Blob URL 触发浏览器下载。
- 如果浏览器不支持自动下载（部分 iOS Safari 限制），fallback 为打开新标签页展示文件内容。

## 7. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 家庭无任何数据（新建家庭） | CSV 只有表头行；JSON 的 plants 为空数组。仍允许导出（不报错） |
| 数据量极大（>1000 条 logs） | Action 内分批查询，避免单次查询超限；文件大小无硬性限制 |
| 导出过程中网络断开 | Action 已在服务端完成，文件已存入 _storage；前端重试下载即可 |
| 并发导出（用户连续点击） | 前端 loading 态期间禁用按钮，防止重复触发 |
| 成员退出家庭后其历史记录 | 仍导出（completedBy 显示其当时的 displayName） |
| 归档植物 | 包含在导出中，标记 `isArchived: true` |
| 图片数据 | 不导出图片文件，JSON 中不含 imageStorageId |
| 导出文件过期（24h 后） | 用户需重新触发导出；不提供历史导出文件列表 |

## 8. 验收标准

### 8.1 CSV 导出

- [ ] 设置页可触发 CSV 格式导出
- [ ] 下载的 CSV 文件可被 Excel 正确打开（UTF-8 BOM，中文不乱码）
- [ ] CSV 包含全部完成日志，列定义与规格一致
- [ ] 按完成时间降序排列
- [ ] 时间格式为用户本地时区

### 8.2 JSON 导出

- [ ] 设置页可触发 JSON 格式导出
- [ ] JSON 结构与规格定义一致，包含 family/members/plants/tasks/logs
- [ ] 不含敏感信息（邮箱、手机号、内部 ID、推送订阅）
- [ ] 归档植物包含在导出中

### 8.3 安全与隐私

- [ ] 只能导出当前用户所属家庭的数据
- [ ] 非家庭成员调用 action 返回权限错误
- [ ] 导出文件 24 小时后自动清理

### 8.4 体验

- [ ] 导出过程有 loading 反馈
- [ ] 导出成功后自动触发下载
- [ ] 导出失败有友好错误提示
- [ ] 连续点击不会重复触发

## 9. 后续候选（本次不做）

- 导出时间范围筛选（只导出最近 3 个月）
- 导出特定植物的数据（而非全家庭）
- 定期自动导出（每月自动备份到邮箱）
- 数据导入能力（从 JSON 恢复数据）
- 导出包含图片（打包为 ZIP）
- PDF 格式的养护报告（带图表可视化）

## 10. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 两种格式（CSV + JSON）而非更多 | CSV 满足分析需求（Excel 友好），JSON 满足备份需求（结构完整）；PDF/ZIP 等复杂格式留给后续 |
| D2 | 用 Convex action 而非 query | 大数据量聚合可能超 query 时间限制；action 可写 _storage 生成文件 |
| D3 | 文件 24 小时过期 | 避免 _storage 无限膨胀；导出是即时操作，不需要长期保留 |
| D4 | 不导出图片 | 图片文件体积大，导出复杂度高（需打包 ZIP）；本次聚焦结构化数据，图片导出留给后续 |
| D5 | 不导出内部 ID | 用人类可读的名称关联数据，降低理解门槛；内部 ID 对用户无意义 |
| D6 | 所有家庭成员均可导出 | 数据属于家庭共有，不限制为 owner 专属；隐私通过"不含个人身份信息"保障 |
| D7 | 不记录导出历史（MVP） | 简化实现；如需限流后续再加 exportLogs 表 |
