# 植物详情页产品设计细化（迭代 v0.2）

**版本**：v0.2（迭代设计）
**日期**：2026-06-11
**关联文档**：基于 [2026-06-05-mvp-requirements.md](./2026-06-05-mvp-requirements.md) 第 5.2 节「植物管理」、第 5.3 节「养护任务管理」、第 5.5 节「完成任务与自动顺延」的细化与扩展
**文档定位**：本文档不重复 MVP 全量需求，只针对**植物详情页（`/plants/:plantId`）**这一个高频核心页面，把页面角色重新定位、信息层级重构、任务展示与完成交互、正反馈闭环与 UI 美化原则讲清楚，作为下一轮迭代的设计依据。

---

## 0. 设计锚点（先统一认知）

用 JTBD 视角，用户来到植物详情页时雇佣它完成的真实任务**不是**「阅读一株植物的资料卡」，而是：

> **"这盆植物现在需要我做什么？我能不能一步完成，然后安心离开？"**

由此推导出植物详情页的三个核心价值：

1. **养护面板** —— 一眼看到这盆植物当前的待办状态（逾期/今天到期/一切正常）。
2. **一步完成** —— 到期任务直接在详情页完成，不需要跳转到待办页。
3. **计划总览** —— 查看这盆植物的所有养护计划和下次时间，心里有数。

植物详情页是**高频操作页**（用户从待办页点击植物缩略图、从列表页点击卡片都会进入），其设计原则是**任务状态前置、操作路径最短、完成有正反馈**。

## 1. 页面角色定位

### 1.1 与待办页的关系

| 维度 | 待办页 `/todo` | 植物详情页 `/plants/:id` |
|------|---------------|--------------------------|
| 核心任务 | 全家所有植物今天该做什么 → 批量执行 | 这一盆植物的养护全貌 → 精准操作 |
| 使用频率 | 高频（每天多次） | 高频（从待办/列表进入） |
| 信息粒度 | 跨植物的任务聚合 | 单株植物的任务 + 资料 |
| 主动作 | 完成任务 / 推迟 | 完成任务 / 查看计划 / 编辑 |
| 设计倾向 | 执行效率（批量） | 执行效率（精准）+ 信息完整 |

### 1.2 用户来到此页面的典型场景

1. **从待办页点击植物头像** → 想看这盆植物的完整养护状态，可能顺手完成其他到期任务
2. **从植物列表点击卡片** → 想查看详情、编辑信息、或管理养护计划
3. **从推送通知进入** → 收到"XX 需要浇水"通知，进来一步完成
4. **日常巡视** → 浇完水后打开确认一下，顺便看看还有没有别的要做

## 2. 页面信息架构（重构）

### 2.1 当前问题

现有详情页的信息层级为：**大封面图（4:3）→ 植物资料（名称/位置/状态/描述/备注）→ 任务列表 → 底部操作栏**。

问题：
- 封面图占据首屏 50%+ 空间，用户需要滚动才能看到任务
- 任务列表不区分紧急程度，所有任务平铺
- 任务卡片信息密度低（每个任务占 4-5 行文字 + 2 个按钮）
- 完成后卡片直接消失，没有告知"下次什么时候"
- 底部操作栏 3 个按钮中"返回列表"和"编辑植物"是低频操作却占据高价值位置

### 2.2 重构后的信息层级（已拍板）✅

从上到下，按用户注意力衰减排列：

| 层级 | 区域 | 内容 | 设计意图 |
|------|------|------|----------|
| L0 | 顶部导航栏 | ← 返回 · 植物名称 · ⋯ 溢出菜单 | 轻量导航，不占内容空间 |
| L1 | 紧凑 Hero | 缩略图 + 名称 + 位置 + 状态摘要 | 确认"我在看哪盆植物"，1 秒识别 |
| L2 | 需要处理区 | 逾期/今天到期的任务（高亮卡片） | **核心操作区**，一步完成 |
| L3 | 养护计划区 | 所有任务的下次时间总览 | 心里有数，知道接下来的安排 |
| L4 | 植物档案区 | 描述/备注/归档操作（可折叠） | 低频信息，需要时展开 |

## 3. 各区域详细设计

### 3.1 顶部导航栏

**决策：用轻量顶栏替代底部"返回列表"按钮。** ✅

结构：`[← 返回] [植物名称居中] [⋯]`

- 左侧：返回按钮，点击回到上一页（植物列表或待办页）
- 中间：植物名称（单行截断，最大宽度 60%）
- 右侧：溢出菜单按钮（⋯），展开后包含：编辑植物、归档/恢复、删除

溢出菜单内的操作项：
- 「编辑植物资料」→ 跳转编辑页
- 「归档这盆植物」/「恢复到看板」→ 带二次确认（复用现有 ArchivePlantAction 逻辑）
- 「删除植物」→ 危险操作，红色文字 + 二次确认

### 3.2 紧凑 Hero 区

**决策：封面图从 4:3 大图改为 48×48 圆角方形缩略图，横向布局。** ✅

布局：水平排列 `[缩略图] [文字信息]`

```
┌──────────────────────────────────────┐
│  ┌────┐  绿萝                         │
│  │ 🪴 │  客厅 · 正在养护中             │
│  └────┘                               │
└──────────────────────────────────────┘
```

- 缩略图：48×48px，border-radius: 12px，1px border，点击可全屏预览大图
- 植物名称：16px/700，单行
- 副信息：12px/400 muted，格式为「位置 · 状态摘要」
- 状态摘要规则：
  - 有逾期任务 → 显示「⚠️ N 项养护已逾期」（warning 色）
  - 今天有到期 → 显示「今天有 N 项待处理」（leaf 色）
  - 全部正常 → 显示「正在养护中」（muted 色）
  - 已归档 → 显示「已归档于 X月X日」（muted 色）

**全屏预览交互：** 点击缩略图后，封面图以全屏 overlay 展示（带关闭按钮），不跳转页面。如果没有封面图则点击无响应。

### 3.3 需要处理区（核心操作区）

**决策：将到期/逾期任务从普通列表中提取出来，形成独立的高亮操作区。** ✅

#### 3.3.1 数据筛选规则

从该植物的所有任务中筛选：
- `nextDueAt < 今天结束时间`（即逾期 + 今天到期）的任务
- 排除「今日已完成」的任务（`lastCompletedAt` 在今天范围内）

如果筛选结果为空，则整个"需要处理"区不渲染（不显示空状态）。

#### 3.3.2 区域标题

格式：`⚠️ 需要处理（N）`
- N = 筛选出的任务数量
- 标题样式：12px/700 uppercase，颜色 `--color-warning`（有逾期时）或 `--color-leaf-light`（仅今天到期时）

#### 3.3.3 任务行设计（紧凑模式）

每个任务渲染为一个紧凑的水平行：

```
┌──────────────────────────────────────────┐
│  💧  浇水     已逾期 2 天          [✓]   │
│  🌱  施肥     今天到期             [✓]   │
└──────────────────────────────────────────┘
```

行内元素（从左到右）：
- 任务类型 emoji（16px，来自 TaskTypeBadge 的 taskTypeEmoji）
- 任务标签（14px/600，如"浇水"、"施肥"、自定义标签）
- 到期状态文案（12px/400）：
  - 逾期：`已逾期 N 天`（warning 色）
  - 今天：`今天到期`（leaf 色）
- 圆形完成按钮（36×36px，右对齐）

**左侧色条：** 每行左侧有 3px 宽的任务类型色条（复用 DueTaskCard 的 typeStrip 设计），视觉上区分不同任务类型。

#### 3.3.4 完成交互流程

1. 用户点击圆形 ✓ 按钮
2. 按钮状态：idle → pending（显示"…"）→ done（绿色 ✓）
3. 触发轻震动（`triggerHaptic()`）+ emoji 微动效（从按钮上浮）
4. 任务行执行折叠退场动画（height 收缩 + opacity 淡出，300ms）
5. 底部弹出 UndoToast：「💧 浇水已完成，下次 6月12日」
6. 5 秒内可撤销

**UndoToast 文案规则：**
- 格式：`{emoji} {任务标签}已完成，下次 {下次到期日期}`
- 日期格式：`M月D日`（使用 zh-CN 的 month: "numeric", day: "numeric"）

#### 3.3.5 全部完成后的状态

当"需要处理"区的所有任务都被完成后：
- 整个区域折叠消失（带 300ms 高度收缩动画）
- Hero 区的状态摘要实时更新为「正在养护中」或「今天有 N 项待处理」（如果还有未完成的）

### 3.4 养护计划区

**决策：展示该植物的所有养护任务（含已完成的），作为"计划总览"。** ✅

#### 3.4.1 区域标题

格式：`📋 养护计划（N）`
- N = 该植物的总任务数
- 右侧：「+ 添加」文字按钮（点击跳转创建任务页）

#### 3.4.2 任务行设计（信息模式）

与"需要处理"区的紧凑行不同，这里的任务行侧重**信息展示**而非操作：

```
┌──────────────────────────────────────────┐
│  💧  浇水    每 5 天    下次 6月12日      │
│  🌱  施肥    每 20 天   下次 6月25日      │
│  💨  喷雾    每 3 天    下次 6月9日       │
│  ✂️  修剪    每 30 天   下次 7月5日       │
└──────────────────────────────────────────┘
```

行内元素（从左到右）：
- 任务类型 emoji（16px）
- 任务标签（14px/600）
- 间隔信息（12px/400 muted）：`每 N 天`
- 下次时间（12px/400）：
  - 逾期 → `已逾期 N 天`（warning 色）
  - 今天 → `今天`（leaf 色）
  - 未来 → `M月D日`（muted 色）

**点击行为：** 点击任务行进入该任务的编辑页面。

#### 3.4.3 空状态

如果该植物没有任何养护任务：

```
┌──────────────────────────────────────────┐
│  📋 养护计划                              │
│                                          │
│  还没有设置养护计划                        │
│  添加浇水、施肥等提醒，系统会自动计算       │
│  下次养护时间并提醒全家。                   │
│                                          │
│         [ + 添加养护计划 ]                 │
└──────────────────────────────────────────┘
```

### 3.5 植物档案区（可折叠）

**决策：植物描述、备注等低频信息折叠展示，默认收起。** ✅

#### 3.5.1 折叠标题

格式：`📝 植物档案 [▾]`（点击展开/收起）

折叠状态持久化：使用 localStorage key `plant-detail:archive-collapsed:{plantId}`，跨会话记住用户选择。默认收起。

#### 3.5.2 展开后内容

```
┌──────────────────────────────────────────┐
│  📝 植物档案                        [▴]   │
│                                          │
│  描述                                     │
│  这是一盆从花市买回来的绿萝，长势很好...    │
│                                          │
│  养护备注                                 │
│  喜欢散射光，不能暴晒。冬天减少浇水频率。  │
│                                          │
│  添加时间：2026年5月20日                   │
│  最近更新：2026年6月8日                    │
└──────────────────────────────────────────┘
```

- 描述：14px/1.6，muted 色
- 养护备注：14px/1.6，muted 色，带「养护备注」小标题（12px/600 leaf-light）
- 时间信息：12px/400 muted

如果描述和备注都为空，展开后显示：「暂无描述和备注，可以在编辑页面补充。」

## 4. 底部操作栏重构

### 4.1 当前问题

现有底部栏包含 3 个按钮（返回列表 / 编辑植物 / 添加任务），其中"返回列表"已被顶部导航栏替代，"编辑植物"已移入溢出菜单。

### 4.2 新方案（已拍板）✅

**底部操作栏精简为单个悬浮"添加养护计划"按钮。**

- 位置：页面右下角悬浮，距底部 `calc(56px + env(safe-area-inset-bottom) + 16px)`（避开底部 Tab 栏）
- 样式：圆角胶囊形（pill），背景 `--color-gold`，文字 `--color-ink`
- 文案：`+ 添加养护`
- 点击：跳转到创建任务页 `/plants/:id/tasks/new`

**显示条件：** 始终显示（即使已有任务）。当植物已归档时隐藏。

## 5. 交互与动效规范

### 5.1 任务完成动效

| 阶段 | 时长 | 效果 | reduced-motion 降级 |
|------|------|------|---------------------|
| 按钮反馈 | 即时 | 按钮变绿 + emoji 上浮 | 按钮变绿，无 emoji |
| 震动反馈 | 即时 | `triggerHaptic()` | 无震动 |
| 行折叠退场 | 300ms | height → 0 + opacity → 0 | 即时消失 |
| UndoToast 入场 | 250ms | 从底部滑入 | 即时显示 |
| UndoToast 自动消失 | 5000ms 后 | 向下滑出 | 即时消失 |

### 5.2 区域折叠/展开动效

- 展开：height 从 0 增长到 auto（使用 CSS `grid-template-rows: 0fr → 1fr` 技巧），300ms ease-out
- 收起：反向，300ms ease-in
- reduced-motion：即时切换，无动画

### 5.3 全屏图片预览动效

- 打开：缩略图位置 scale up 到全屏 + 背景遮罩淡入，360ms cubic-bezier(0.22, 1, 0.36, 1)
- 关闭：反向缩回 + 遮罩淡出，200ms
- reduced-motion：直接显示/隐藏，无缩放

### 5.4 页面进入动效

- 从列表/待办页进入详情页时，内容区域整体 fade-in + 轻微上移（translateY(8px) → 0），200ms
- reduced-motion：无动画

## 6. 状态与边界情况

### 6.1 加载态

- 顶部导航栏立即渲染（植物名称用骨架屏占位）
- 内容区域显示：居中的加载文案「正在加载植物信息…」

### 6.2 植物不存在

- 顶部导航栏显示「← 返回」
- 内容区域显示 EmptyState：「当前家庭中找不到这盆植物」+ 返回按钮

### 6.3 已归档植物

- Hero 区状态摘要显示「已归档于 X月X日」
- "需要处理"区不渲染（归档植物不产生待办）
- "养护计划"区正常展示（但任务行不显示完成按钮）
- 底部悬浮按钮隐藏
- 溢出菜单中「归档」变为「恢复到看板」

### 6.4 无任务的新植物

- "需要处理"区不渲染
- "养护计划"区显示空状态引导

### 6.5 所有任务都未到期

- "需要处理"区不渲染
- "养护计划"区正常展示所有任务的下次时间
- Hero 区状态摘要显示「正在养护中」

## 7. 数据依赖

### 7.1 查询接口

复用现有 `api.plants.getPlantDetail` 查询，返回结构：

```typescript
{
  plant: {
    id: string;
    name: string;
    imageUrl: string | null;
    imageStorageId: string | null;
    location: string | null;
    description: string | null;
    note: string | null;
    isArchived: boolean;
    archivedAt: number | null;
    createdAt: number;
    updatedAt: number;
    familyId: string;
  };
  tasks: Array<{
    id: string;
    taskType: "watering" | "fertilizing" | "misting" | "repotting" | "pruning" | "custom";
    customLabel: string | null;
    intervalDays: number;
    lastCompletedAt: number | null;
    nextDueAt: number;
  }>;
}
```

### 7.2 前端分桶逻辑

前端根据 `tasks` 数组进行本地分桶（复用 `scheduling.ts` 的逻辑）：

```typescript
// 需要处理：逾期 + 今天到期，且今日未完成
const actionableTasks = tasks.filter(task => {
  const isOverdueOrToday = task.nextDueAt < startOfTomorrow;
  const isCompletedToday = computeCompletedToday(task.lastCompletedAt, startOfToday, startOfTomorrow);
  return isOverdueOrToday && !isCompletedToday;
});

// 养护计划：全部任务（按 nextDueAt 升序）
const allTasks = [...tasks].sort((a, b) => a.nextDueAt - b.nextDueAt);
```

### 7.3 Mutation 复用

- 完成任务：`api.tasks.completePlantTask`（已有）
- 撤销完成：`api.tasks.undoCompletePlantTask`（已有）
- 归档/恢复：`api.plants.setPlantArchivedState`（已有）

无需新增后端接口。

## 8. 与其他页面的联动

### 8.1 从待办页进入

待办页的 DueTaskCard 点击植物头像时，跳转到 `/plants/:plantId`。用户在详情页完成任务后，返回待办页时数据已通过 Convex 实时同步更新。

### 8.2 从植物列表进入

植物列表卡片点击后进入详情页。详情页的任何状态变更（完成任务、归档）都会通过 Convex 实时同步反映到列表页。

### 8.3 编辑/创建任务后返回

从详情页跳转到编辑/创建任务页，完成后返回详情页。由于使用 Convex 实时查询，任务列表会自动更新，无需手动刷新。
