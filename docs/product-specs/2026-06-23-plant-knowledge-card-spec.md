**版本**：v1.1
**日期**：2026-06-24
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（§7.2 植物管理）、[迭代储备大纲](./2026-06-17-iteration-backlog-outline.md)（P2-1）、[养护任务灵活性](./2026-06-22-care-task-flexibility-spec.md)（排期建议联动）
**文档定位**：为用户提供植物品种级别的养护知识引导，在创建植物时自动推荐养护参数，在详情页展示品种养护小贴士，降低新手用户的养护决策门槛。

## 0. 本次迭代要解决的核心问题

MVP 中用户创建植物时，需要自行决定浇水频率、施肥间隔等参数，但大多数用户（尤其是新手）并不清楚具体数值该填多少。这导致两个体验缺口：

- **"刚买了一盆龟背竹，不知道几天浇一次水，随便填了个 3 天，结果浇多了烂根"** —— 缺乏养护知识引导：用户在设置任务间隔时没有参考依据，只能凭感觉填写，容易导致养护不当。
- **"我家有 10 盆不同的植物，每次都要去百度查养护方法，再回来设置"** —— 信息获取路径长：用户需要在应用外查找养护知识，再手动转化为应用内的任务配置，流程割裂。

本次通过引入**植物品种知识库**，在创建植物和查看详情时提供品种级别的养护建议，让用户"不用查就知道怎么养"。

---

## 1. 产品目标

让用户在创建植物时能够快速获得该品种的推荐养护参数（浇水频率、光照需求、施肥建议等），一键应用到任务配置中。同时在植物详情页提供常驻的养护知识卡片，作为日常养护的参考指南。

**核心度量指标**（本期观测、不作为硬性 KPI）：

| 指标 | 定义 | 预期基线 |
|------|------|----------|
| 品种匹配率 | 创建植物时输入名称命中品种库的比率 | ≥ 60% |
| 推荐应用率 | 展示推荐卡片后点击"应用推荐"的比率 | ≥ 40% |
| 知识卡片展开率 | 详情页用户主动展开知识卡片的比率 | ≥ 25% |
| 任务参数合理性 | 应用推荐后 30 天内未手动修改间隔的比率 | 观测，不设目标 |

## 2. 用户场景

### 场景 1：新手创建植物时获得推荐

用户刚入坑养植物，买了一盆绿萝。在创建植物页面输入"绿萝"后，系统自动匹配到品种库中的绿萝条目，弹出推荐卡片："绿萝 — 浇水建议每 3~5 天一次，喜散射光，忌强光直射"。用户点击"应用推荐"，浇水任务自动填入间隔 4 天，简介和备注自动预填品种描述，省去了查资料的步骤。

### 场景 2：详情页查看养护贴士

用户养了一盆琴叶榕半年了，最近叶子发黄。她打开植物详情页，看到知识卡片区域写着"琴叶榕 — 冬季减少浇水，保持盆土微干；叶片发黄可能是浇水过多或光照不足"。她意识到可能是冬天没有减少浇水频率，于是去编辑任务间隔。

### 场景 3：品种未匹配时的降级体验

用户养了一盆比较小众的"鹿角蕨"，品种库中没有收录。创建时系统不展示推荐卡片，用户正常手动填写参数。详情页的知识卡片区域显示"暂无该品种的养护指南，欢迎自行记录养护心得到备注中"。

### 场景 4：推荐参数与个人经验不同

用户是资深植物爱好者，知道自己家的环境比较干燥，绿萝需要每 2 天浇一次。系统推荐 3~5 天，用户选择忽略推荐，手动填写 2 天。推荐是建议而非强制。

### 场景 5：编辑页后续关联品种

用户在 MVP 时期创建了一盆"小虎"，当时没有品种库。现在进入编辑页，输入框内改为"虎皮兰"，系统匹配到品种，用户点击候选项完成关联。输入框补全为"虎皮兰"，上方出现品种关联状态条。

### 场景 6：输入部分文字后选择品种

用户输入"蝴蝶"，下拉出现"蝴蝶兰"候选项。用户点击后，输入框文字自动补全为"蝴蝶兰"（而非保持"蝴蝶"），同时输入框视觉状态切换为"已关联"态，让用户明确感知到关联成功。

## 3. 功能设计

### 3.1 品种知识库

**数据结构**（前端静态 JSON）：

```typescript
interface PlantSpecies {
  id: string;                    // 唯一标识，如 "monstera-deliciosa"
  names: string[];               // 品种名称（含别名），如 ["龟背竹", "蓬莱蕉"]
  category: string;              // 分类，如 "观叶植物"
  difficulty: "easy" | "medium" | "hard";  // 养护难度
  briefDescription: string;      // 一句话品种简介，如 "热带观叶植物，叶片有独特孔洞"
  care: {
    watering: {
      intervalRange: [number, number];  // 建议浇水间隔范围（天），如 [3, 5]
      tip: string;                       // 浇水贴士
    };
    light: {
      level: "low" | "medium" | "bright" | "direct";  // 光照需求
      tip: string;
    };
    fertilizing: {
      intervalRange: [number, number];  // 建议施肥间隔范围（天），如 [14, 30]
      tip: string;
      seasonalNote?: string;            // 季节性说明
    };
    humidity?: {
      level: "low" | "medium" | "high";
      tip: string;
    };
    temperature?: {
      range: [number, number];          // 适宜温度范围（℃）
      tip: string;
    };
  };
  commonIssues: string[];        // 常见问题提示，如 ["叶尖发黄可能是空气太干"]
  seasonalNotes: {
    springSummer: string;        // 春夏养护要点
    autumnWinter: string;        // 秋冬养护要点
  };
}
```

**初期规模**：预置 80 种常见室内绿植数据，覆盖中国家庭最常养的品种。

**数据来源**：人工整理 + 园艺资料校验，以 JSON 文件形式存放在前端代码中（`app/src/data/plant-species.json`）。

### 3.2 品种匹配逻辑

**匹配时机**：用户在创建/编辑植物页面的"植物名称"输入框中输入文字时触发。

**触发条件**：输入字符数 ≥ 2 时开始匹配。

**防抖策略**：用户停止输入 200ms 后执行匹配计算（debounce 200ms），避免每次按键都触发遍历。

**匹配规则**：

- 对用户输入的植物名称，在品种库的 `names` 数组中进行模糊匹配（包含匹配）。
- 匹配优先级：完全匹配 > 前缀匹配 > 包含匹配。同优先级内按品种库原始顺序排列。
- 匹配结果最多展示 3 个候选品种（如果有多个匹配）。
- 匹配不区分大小写，支持中文和拉丁学名。

**性能约束**：品种库 80 条数据，前端内存遍历，无需 Web Worker。单次匹配耗时应 < 5ms。如未来品种库扩展至 500+ 条，需评估是否引入前缀索引。

**匹配结果展示**：

- 输入框下方出现匹配候选列表（类似搜索建议）。
- 用户点击某个候选品种后，输入框补全为品种标准名 + 视觉状态切换 + 触发推荐卡片展示。
- 用户也可以忽略候选列表，继续手动填写。

### 3.3 创建植物时的推荐卡片

**触发条件**：用户在创建植物页面选择了一个匹配的品种。

**卡片内容**：

- 品种名称 + 养护难度标签（"好养" / "适中" / "进阶"）。
- 浇水建议：推荐间隔范围 + 一句话贴士。
- 光照需求：图标 + 一句话描述。
- 施肥建议：推荐间隔范围 + 季节性说明。

**操作按钮**：

- "应用推荐"（主按钮，`--color-leaf` 背景色）：将推荐参数自动填入表单多个字段。
- "仅供参考，不应用 →"（文字链接，`--color-muted`）：关闭卡片，不做任何填充。

**"应用推荐"完整行为**（v1.1 细化）：

| 字段 | 预填内容 | 预填规则 |
|------|----------|----------|
| 浇水间隔 | `Math.ceil((min + max) / 2)` | 覆盖写入（任务维度，待任务模块对接） |
| 施肥间隔 | `Math.ceil((min + max) / 2)` | 覆盖写入（任务维度，待任务模块对接） |
| 简介 | `"{category}，{briefDescription}。{care.light.tip}"` | 仅在简介为空时填入；若已有内容则不覆盖 |
| 养护备注 | `"常见问题：{commonIssues[0]}。当季要点：{seasonalNotes.当季}"` | 仅在备注为空时填入；若已有内容则不覆盖 |

**预填冲突处理原则**：推荐永远不覆盖用户已输入的内容。如果字段已有值，该字段跳过预填，Toast 提示中补充说明"部分字段已有内容，未被覆盖"。

**Toast 反馈文案**：
- 全部字段预填成功：`"已应用 {品种名} 的推荐养护参数 🌿"`
- 部分字段跳过：`"已应用推荐（简介/备注已有内容，未覆盖）🌿"`

**卡片退出动画**：点击"应用推荐"后卡片以 scale(0.95) + opacity(0) 向上收缩消失，动画 250ms ease-in；点击"仅供参考"后卡片以 opacity(0) 淡出，动画 200ms ease-in。

### 3.4 详情页知识卡片

**位置**：植物详情页，位于任务列表区域下方，作为独立的信息区块。

**展示条件**：

- 植物的 `speciesId` 字段有值（即创建时匹配了品种）。
- 如果未匹配品种，展示兜底文案。

**卡片内容**：

- 标题："养护指南 · {品种名}"。
- 分区展示：浇水 / 光照 / 施肥 / 常见问题。
- 每个分区一个图标 + 一段简短文字（1~2 行）。
- 底部展示当前季节的养护要点（根据月份自动切换春夏/秋冬文案）。

**季节判定规则**：3~8 月为春夏，展示 `seasonalNotes.springSummer`；9~2 月为秋冬，展示 `seasonalNotes.autumnWinter`。

**交互**：

- 卡片默认折叠，只展示标题行 + 右侧 chevron-down 图标。
- 点击标题行展开完整内容，chevron 旋转 180°。
- 展开态可以再次点击标题行折叠。
- 展开/折叠状态不持久化（每次进入详情页默认折叠）。
- 展开/折叠动画：`grid-template-rows: 0fr → 1fr`，配合 opacity 过渡，时长 250ms ease-out。

**兜底态**（未关联品种）：

- 展示"暂无该品种的养护指南"灰色文案（`--color-muted`）。
- 下方补充引导文字："编辑植物时输入品种名可关联养护指南"。
- 不可展开，无 chevron 图标。

### 3.5 品种关联存储

**plants 表新增字段**：

```
speciesId: v.optional(v.string())  // 关联的品种库 ID，如 "monstera-deliciosa"
```

**写入时机**：用户在创建植物时选择了匹配的品种，将品种 ID 存入。

**不强制关联**：用户可以不选择任何品种（手动填写植物名），此时 `speciesId` 为空。

**取消关联语义**：updatePlant 中 `speciesId: null` 表示清除关联，`speciesId: undefined`（不传）表示不修改。

## 4. 数据模型变更

### 4.1 plants 表新增字段

```
speciesId: v.optional(v.string())  // 品种库 ID，用于关联前端静态知识库
```

**向后兼容**：`v.optional` 保证老数据无需迁移。老植物的 `speciesId` 为 undefined，详情页展示兜底文案。

### 4.2 无新增表

品种知识库为前端静态 JSON，不存入 Convex。理由：

- 数据量小（80 条，约 50KB），前端直接打包无性能问题。
- 数据更新频率极低（季度级别），不需要动态更新能力。
- 避免增加后端复杂度和查询开销。

### 4.3 索引评估

无需新增索引。`speciesId` 仅用于前端展示时的本地查找，不参与后端查询条件。

## 5. 接口变更

### 5.1 createPlant 改造

新增可选参数：

```
speciesId: v.optional(v.string())
```

创建植物时，如果用户选择了品种，将 `speciesId` 一并写入。

### 5.2 updatePlant 改造

新增可选参数：

```
speciesId: v.optional(v.union(v.string(), v.null()))
```

语义：传 `string` 设置新关联，传 `null` 清除关联，不传该字段则不修改。

### 5.3 getPlantForEdit / getPlantDetail 返回值

返回字段中包含 `speciesId: string | null`（未关联时为 null）。

### 5.4 无新增后端接口

品种匹配和知识展示均为纯前端逻辑，不需要新的 Convex function。

## 6. 前端交互设计

### 6.1 品种匹配 — 候选列表

**触发与消失**：

- 用户输入 ≥ 2 字符 + debounce 200ms 后展示候选列表。
- 输入 < 2 字符时立即隐藏列表。
- 用户点击列表外区域（blur）或按 Escape 键隐藏列表。
- 用户选择候选项后隐藏列表。
- 已选择品种后（已关联态），不再展示候选列表，除非用户修改了输入框文本。

**列表视觉规格**：

- 容器：position absolute，z-index 50，紧贴输入框底部，宽度与输入框等宽。
- 背景 `--color-surface`，圆角 `--radius-card`，阴影 `0 4px 16px rgba(0,0,0,0.08)`。
- 每条候选项高度 48px，内边距 12px 16px。
- 候选项内容：🌿 品种名（bold）+ 难度 pill + 分类（`--color-muted`）。
- 高亮态（hover / keyboard focus）：背景 `--color-mist`。
- 入场动画：`opacity 0→1 + translateY(-4px→0)`，150ms ease-out。

**键盘导航**：

- `ArrowDown`：高亮下一项（到底循环回第一项）。
- `ArrowUp`：高亮上一项（到顶循环回最后一项）。
- `Enter`：选择当前高亮项（等同点击）。
- `Escape`：关闭列表，焦点保持在输入框。
- 默认无高亮项，首次按 ArrowDown 高亮第一项。

**无障碍（a11y）**：

- 输入框 `role="combobox"`，`aria-expanded` 反映列表展开状态。
- 列表容器 `role="listbox"`，每个候选项 `role="option"`。
- 高亮项通过 `aria-activedescendant` 关联。
- 列表状态变化通过 `aria-live="polite"` 播报。

### 6.1.1 品种选择确认态

用户点击候选项后，系统进入"品种已关联"确认态，包含以下行为：

**输入框文本补全**：输入框文本立即替换为品种的标准完整名称（`names[0]`）。例如用户输入"蝴蝶"匹配到"蝴蝶兰"，点击后输入框显示"蝴蝶兰"。这确保了输入框内容与实际关联的品种名一致。

**输入框视觉状态变化**：选择品种后输入框进入"已关联"态，具体样式：

- 输入框左侧出现品种标识图标（🌿 emoji 或叶子 SVG），占据 24px 宽度。
- 输入框右侧出现圆形清除按钮（× 图标，`--color-muted`），点击可取消关联。
- 输入框底色变为 `--color-mist`（区别于普通白色输入态）。
- 输入框下边框色变为 `--color-leaf`（原为 `--color-line`）。
- 文字颜色保持 `--color-ink`，字重提升为 600。

**取消关联**（点击 × 按钮）：

- 输入框回到普通输入态（恢复白色底色、默认字重、移除图标）。
- 输入框文本保留当前值不清空（用户可能想在此基础上继续编辑）。
- `speciesId` 清除为 null。
- 如果推荐卡片仍在展示则一并移除。
- 清除后如文本仍 ≥ 2 字符，重新触发匹配。

**用户修改文本时的行为**：

- 如果用户在已关联态下修改输入框内的文字（增删任何字符），自动解除关联。
- 解除时：回到普通输入态样式、清除 speciesId、移除推荐卡片。
- 修改后的新文本如 ≥ 2 字符，经 debounce 200ms 后重新触发匹配。
- 设计理由：品种名称是系统结构化字段，植物名称是用户自由命名——两者不强绑定。用户可以关联"蝴蝶兰"但给自己的植物取名"蝶蝶"。

**植物名称与品种名称的关系说明**：

- "植物名称"字段是用户给自己这盆植物的自定义称呼（如"客厅绿萝"、"蝶蝶"）。
- "品种关联"是系统层面的结构化绑定，决定了是否展示养护指南。
- 选择品种时自动补全名称是便利行为，但用户随后可以自由修改名称。修改名称会解除关联（因为名称变了意味着可能不再是这个品种），用户需重新选择才能重新关联。

### 6.2 创建植物页 — 推荐卡片

- 选择品种后，在名称输入框下方（候选列表位置）展示推荐卡片。
- 卡片背景 `--color-mist`，边框 1px `--color-line`，圆角 `--radius-card`。
- 卡片内容紧凑：品种名 + 难度 pill + "应用推荐"按钮（header 行）；下方三列网格展示浇水/光照/施肥。
- 底部右对齐："仅供参考，不应用 →"文字链接。
- 入场动画：`scale(0.96) → scale(1) + opacity 0→1`，300ms ease-out。

**点击"应用推荐"后的完整行为**：

1. 按钮进入 disabled 态（防重复点击）。
2. 卡片以退出动画消失（250ms）。
3. 自动预填表单字段（参见 §3.3 预填规则表）。
4. Toast 弹出确认文案（3 秒自动消失）。
5. 焦点自动移至表单下一个未填写字段。

**点击"仅供参考"后的行为**：

1. 卡片以淡出动画消失（200ms）。
2. 不做任何字段填充。
3. `speciesId` 关联保持（品种已关联，只是用户不想自动填充参数）。
4. 焦点保持在当前位置。

### 6.3 详情页 — 知识卡片区块

- 位于任务列表下方，与任务区域间距 `--space-lg`（16px）。
- 整个区块有左侧 3px `--color-leaf` 边框条（与表单卡片风格一致）。

**折叠态**：

- 单行展示："🌱 养护指南 · {品种名}"+ 右侧 chevron-down 图标。
- 整行可点击（扩大点击热区至 48px 高度）。
- 文字颜色 `--color-ink`，字号 15px，字重 600。

**展开态**：

- 卡片内部分 4 个区块，纵向排列，每块间距 `--space-md`（12px）。
- 每个区块结构：emoji 图标（24px）+ 标签文字（12px，`--color-muted`）+ 内容文字（14px，`--color-ink`）。
- 四个区块依次为：浇水、光照、施肥、当季要点。
- 当季要点区块额外有高亮态：左边框 2px `--color-leaf` + 背景 `rgba(--color-leaf, 0.04)`。

**展开/折叠动画**：

- 使用 `grid-template-rows: 0fr → 1fr` 实现平滑高度过渡。
- 配合内部元素 opacity 过渡（0 → 1）。
- 时长 250ms，缓动 ease-out。
- chevron 图标同步旋转 180°，时长 250ms。

**未关联品种时**：

- 展示灰色文案："暂无该品种的养护指南"（`--color-muted`，14px）。
- 下方补充引导："编辑植物时输入品种名即可关联"（12px，`--color-muted`）。
- 不可展开，不展示 chevron 图标。
- 整体 opacity 0.7，传达"不可交互"的视觉信号。

### 6.4 编辑植物页 — 品种关联

**页面初始状态**：

- 如果植物已关联品种（`speciesId` 有值），页面加载后：
  - 输入框处于"已关联"态（参见 §6.1.1 视觉规格）。
  - 输入框上方或内嵌展示品种关联状态条（见下方详细描述）。
  - 不展示推荐卡片（仅创建流程展示，编辑页不重复展示）。
- 如果植物未关联品种，输入框为普通态，正常支持品种匹配。

**品种关联状态条**（编辑页专属组件）：

- 位置：紧贴表单卡片顶部，在名称输入框上方。
- 尺寸：高度 36px，宽度 fit-content，圆角 `--radius-pill`。
- 样式：背景 `--color-mist`，内边距 6px 12px。
- 内容：🌿 品种标准名 + " · " + 难度中文标签 + 可点击 × 按钮。
- × 按钮：16px 圆形热区，颜色 `--color-muted`，hover 时 `--color-ink`。
- 入场动画：`speciesTagEnter`（scale 0.9→1 + opacity 0→1），200ms。

**点击 × 取消关联**：

- 状态条以淡出动画消失（150ms）。
- 输入框回到普通态。
- `speciesId` 标记为待清除（保存时写入 null）。
- Toast 提示："已取消品种关联"。
- 用户可以重新输入品种名触发匹配来关联新品种。

**编辑页输入框的匹配行为**：

- 与创建页一致：输入 ≥ 2 字符 + debounce 200ms 后展示候选列表。
- 选择候选项后输入框补全 + 进入已关联态 + 更新 speciesId。
- 编辑页选择品种后**不展示推荐卡片**（避免在编辑已有植物时覆盖用户现有数据）。

**保存行为**：

- 如果 speciesId 未被修改（用户没有操作品种关联），不传 speciesId 字段给 updatePlant。
- 如果用户取消了关联，传 `speciesId: null`。
- 如果用户选择了新品种，传 `speciesId: "new-species-id"`。

### 6.5 候选列表的通用交互规格

本节定义创建页和编辑页共享的候选列表行为。

**加载态**（仅在品种库未就绪时）：

- 展示 shimmer 骨架屏（1 行占位），200ms 后若仍未就绪则展示。
- 实际场景中品种库为前端 JSON，几乎不会出现加载态。

**空结果态**：

- 不展示任何下拉内容（静默消失）。
- 不展示"无结果"文案——避免干扰用户输入流。

**列表条目点击反馈**：

- 点击时条目背景闪烁 `--color-leaf` 50ms（active 态），然后列表消失。
- 提供 haptic feedback（如设备支持 Vibration API，振动 10ms）。

**性能要求**：

- 列表渲染 ≤ 3 条 DOM，无需虚拟滚动。
- 匹配计算在 UI 线程执行，单次 < 5ms（80 条数据）。
- debounce 200ms 保证流畅输入体验。

## 8. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 用户输入的植物名不在品种库中 | 不展示推荐卡片，正常手动创建；详情页展示兜底文案 |
| 品种库 JSON 加载失败 | 静默降级，不展示匹配和推荐功能，不影响核心创建流程 |
| 用户输入拼音或错别字 | 当前版本不支持拼音/模糊纠错匹配，仅支持中文名精确/包含匹配 |
| 品种有多个别名 | names 数组包含所有别名，任一匹配即可 |
| 推荐间隔范围为 [3, 3]（单值） | 直接填入 3，不做中间值计算 |
| 用户应用推荐后又手动修改 | 以用户最终提交的值为准，推荐值不具有约束力 |
| 老植物无 speciesId | 详情页展示兜底文案；用户可通过编辑页面后续关联品种 |
| 品种库更新后 speciesId 对应条目被删除 | 前端查找不到时展示兜底文案，不报错；编辑页状态条不展示 |
| 用户快速连续点击多个候选项 | 以最后一次点击为准，中间状态不残留 |
| 输入框超长文本（> 50 字符） | 不触发匹配（超出合理品种名长度），静默忽略 |
| 用户在已关联态下粘贴新文本 | 等同于"修改文本"，解除关联，用新文本重新匹配 |
| 编辑页植物已关联品种但品种库中找不到 | 不展示状态条，输入框为普通态，等同于未关联 |
| 网络断开时操作品种关联 | 品种匹配为纯前端离线可用；保存时如网络断开走统一离线错误处理 |
| 用户在推荐卡片动画过程中点击其他按钮 | 动画期间卡片内按钮 disabled，防止重复触发 |
| 多个品种名完全相同（理论上不应出现） | 品种 ID 唯一，展示时去重 |

## 11. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 品种库用前端静态 JSON 而非后端存储 | 数据量小（80 条 ≈ 50KB）、更新频率极低（季度级）、避免增加后端复杂度和查询开销；后续如需动态更新再迁移到后端 |
| D2 | 匹配触发阈值为 2 个字符 | 1 个字符匹配结果太多（如"绿"匹配到绿萝、绿宝石等），2 个字符能有效收窄范围 |
| D3 | 推荐是建议而非强制 | 尊重用户的个人经验和环境差异；强制应用会让资深用户反感 |
| D4 | 详情页卡片默认折叠 | 避免信息过载，知识卡片是辅助信息而非核心操作区；用户需要时主动展开 |
| D5 | speciesId 存入 plants 表而非仅前端本地 | 支持跨设备一致体验；为后续"按品种筛选"等功能预留数据基础 |
| D6 | 初期 80 种而非更多 | 平衡数据质量和覆盖率；80 种覆盖 90%+ 的家庭常见绿植；后续根据未匹配高频词补充 |
| D7 | 选择品种后输入框文本补全为标准名 | 即时反馈让用户确认选择生效；输入框内容与实际关联品种一致，消除认知割裂 |
| D8 | 输入框进入"已关联"视觉态 | 普通文本输入 vs 结构化品种关联是两种不同语义，视觉上必须有区分；用户一眼可辨"这不只是文字，而是关联了知识库" |
| D9 | 简介/备注预填绑定到"应用推荐"按钮而非选择品种时自动触发 | 区分"轻量关联"（选品种=只绑定 ID）和"深度应用"（应用推荐=填充多字段）两种行为深度；选择品种后用户可能只想看看推荐卡片的内容做参考 |
| D10 | 预填不覆盖已有内容 | 用户已输入的内容优先级最高；覆盖写入会造成数据丢失和用户不信任感 |
| D11 | 用户修改名称后自动解除品种关联 | 名称是自由字段，品种是结构化字段，两者不强绑定；修改名称暗示用户意图发生变化，解除后可重新匹配 |
| D12 | 编辑页选择品种不展示推荐卡片 | 编辑页的植物已有完整配置（间隔、简介等），展示推荐卡片可能误导用户覆盖现有设置；编辑页的核心诉求是"换一个关联品种"而非"重新配置参数" |
| D13 | debounce 200ms | 平衡响应速度和性能；100ms 太灵敏（中文输入法连续输入），300ms 太迟钝（用户感知卡顿） |
| D14 | 候选列表无结果时静默消失而非展示"无结果" | 减少视觉噪音；用户自由输入不应被"你输入的名字找不到"这类消息干扰 |
