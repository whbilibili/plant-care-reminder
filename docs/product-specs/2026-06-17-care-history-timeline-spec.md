# 养护历史时间线

**版本**：v1.0
**日期**：2026-06-17
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（后续候选·养护历史时间线）
**文档定位**：为家庭植物养护行为提供可回溯的历史视图，让"完成任务"从一次性动作变成有积累感的养护档案。

---

## 0. 本次迭代要解决的核心问题

MVP 上线以来，用户每次完成任务（浇水、施肥、喷雾等）系统都会写入一条 `taskCompletionLog`，但这些数据**前端完全没有消费端**——用户看不到任何养护历史。这导致两个体验缺口：

- **"这盆绿萝到底上次谁浇的水？我不确定需不需要再浇"** —— 家庭协作的核心痛点：完成了但不知道谁做的、做了多久。
- **"养了三个月了，没有任何成就感，感觉只是在还债"** —— 缺乏正反馈：养护变成了只有待办没有回顾的苦差事。

本次通过引入**养护历史时间线**，让完成记录可见、可回溯、有成就感，同时增强家庭协作的透明度。

---

## 1. 产品目标

让用户能够回答三个问题：这盆植物最近被谁照顾了？它的养护频率是否健康？我们家一起养了多久？

具体而言：在植物详情页增加"养护记录"区域，按时间倒序展示该植物的完成日志；在待办页增加家庭级的"最近动态"入口，展示全家庭的养护活动流；提供简单的养护统计摘要，增强成就感。

---

## 2. 用户场景

### 场景 1：单株植物的养护回顾

用户打开某盆植物的详情页，在养护计划区下方看到"养护记录"折叠区。展开后看到按时间倒序的完成记录列表，每条显示：谁完成的、什么任务、什么时间。用户一目了然"上次浇水是 3 天前妈妈做的"。

### 场景 2：家庭养护动态

用户在待办页完成今日任务后，底部出现"最近动态"区域，显示家庭成员最近的养护活动（"妈妈 2 小时前给发财树浇了水"）。这提供了家庭协作的"在场感"——知道其他人也在参与。

### 场景 3：养护里程碑

用户连续 30 天照顾了某盆植物（不间断完成任务），系统在时间线头部展示一个温和的里程碑提示（"已连续照顾 30 天"）。这不是弹窗打扰，而是时间线里的一条特殊记录。

---

## 3. 功能设计

### 3.1 植物详情页 —— 养护记录区

**位置**：在 L3 养护计划区（PlanSection）与 L4 植物档案区（PlantArchiveSection）之间，新增 L3.5"养护记录"折叠区。

**展示规则**：

- 默认折叠（与植物档案区一致），标题格式为"养护记录（N）"，N 为该植物的总完成次数。
- 展开后按时间倒序展示完成日志，每条记录为一个紧凑行。
- 单条记录格式：`[任务类型图标] [成员头像/首字母] 成员名 完成了 任务名 · 相对时间`。例如：`水滴图标 [头像] 妈妈 完成了 浇水 · 3 天前`
- 时间格式：24 小时内用相对时间（"刚刚"/"2 小时前"），超过则用日期（"6月15日"），跨年显示完整日期。
- 初始加载最近 20 条，底部有"加载更多"按钮，每次追加 20 条。

**数据来源**：新增 Convex query `listPlantCompletionLogs`，按 plantId 过滤，按 `completedAt` 降序排列，分页返回。需 join `users` 表获取完成人昵称和头像。

### 3.2 待办页 —— 家庭最近动态

**位置**：待办页底部，在今日任务和即将到期任务之后，新增"最近动态"区域。

**展示规则**：

- 标题"家庭动态"，展示家庭全员最近的 5 条养护完成记录。
- 单条格式与植物详情一致，但多一个植物名维度：`[任务类型图标] [头像] 成员名 给 植物名 完成了 任务名 · 相对时间`。
- 不展开为独立页面，就是待办页底部的一个轻量区域。
- 如果家庭还没有任何完成记录，展示友好的空状态文案（"还没有养护记录，完成一次任务就能看到啦"）。

**数据来源**：新增 Convex query `listFamilyRecentActivity`，按 `by_familyId_and_completedAt` 索引降序取最近 5 条，join `users` 和 `plants` 表获取关联信息。

### 3.3 养护统计摘要

**位置**：植物详情页养护记录区的折叠标题旁，展示简要统计。

**统计项**：

- 总完成次数：该植物所有任务的累计完成次数。
- 最近照顾：最近一次完成的相对时间和完成人。
- 连续养护天数（可选增强，MVP 可不做）：连续有完成记录的天数，断一天归零。

**数据来源**：`listPlantCompletionLogs` 的首条记录提供"最近照顾"信息，总数由 query 额外返回 `totalCount`。

## 4. 对现有逻辑的影响

### 4.1 数据模型

现有 `taskCompletionLogs` 表结构**无需变更**，已有字段完全覆盖时间线展示需求。现有字段包括 taskId、plantId、familyId、completedBy、completedAt、taskType、intervalDays。后续增强（完成备注、完成照片）可在 v1.1 通过 `v.optional` 增量添加字段，不影响本次迭代。

### 4.2 新增接口

| 接口 | 类型 | 权限 | 行为 |
|------|------|------|------|
| `listPlantCompletionLogs` | query | 家庭成员 | 按 plantId 过滤，completedAt 降序，cursor-based 分页，join users 表返回 completedByName 和 completedByAvatar |
| `listFamilyRecentActivity` | query | 家庭成员 | 按 familyId + completedAt 降序取 limit 条，join users + plants 表返回关联信息 |

### 4.3 现有接口改造

- `completePlantTask`：**无需改动**，已正常写入 `taskCompletionLogs`。
- `undoCompletePlantTask`：**无需改动**，撤销时删除 log，时间线自动移除（Convex 响应式）。
- `deletePlant`：**无需改动**，级联删除 logs 已实现。

### 4.4 索引评估

- `by_plantId_and_completedAt`：**建议新增**索引 `["plantId", "completedAt"]`，支持按植物维度的时间线高效查询。当前表已有 `by_taskId` 和 `by_familyId_and_completedAt` 索引，缺少按 plantId 维度的索引。
- `by_familyId_and_completedAt`：已存在，支持家庭动态查询，无需新增。

## 5. 交互设计要点

### 5.1 养护记录区视觉

- 折叠触发器样式与 PlantArchiveSection 一致（箭头 + 标题 + 计数）。
- 记录行高度紧凑（48px），成员头像复用 `MemberAvatar` 组件（24x24），任务类型图标复用 `getTaskTypeIcon`。
- 时间文案用 `--color-muted`，成员名用 `--color-ink`，任务名用 `--color-leaf-light`。
- 全部走 design token，禁硬编码 hex 色值，图标走 Icon 壳。

### 5.2 加载更多

- "加载更多"按钮为 `variant="ghost"` 文字型，居中。
- 加载中显示 loading 态（复用既有 spinner 或旋转图标）。
- 所有记录加载完后隐藏按钮。

### 5.3 空状态

- 植物详情：折叠标题显示"养护记录（0）"，展开后居中显示"还没有养护记录"。
- 待办页动态：如果家庭无任何完成记录，整个区域不渲染（不占空间）。

---

## 6. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 植物无任何完成记录 | 养护记录区可折叠查看，展开显示空状态文案 |
| 完成人已退出家庭 | 显示其昵称（来自 log 时刻的 users 表），头像用兜底首字母；不显示"已离开"标签（避免社交尴尬） |
| 撤销完成 | Convex 响应式自动从时间线移除，无需手动刷新 |
| 大量历史记录（>100 条） | cursor-based 分页，每次 20 条，避免一次性加载 |
| 跨时区用户 | `completedAt` 存储 UTC 时间戳，前端根据本地时区渲染相对时间/日期 |

## 7. 验收标准

### 7.1 植物详情页养护记录

- [ ] 详情页在养护计划区下方展示"养护记录（N）"折叠区
- [ ] 展开后按时间倒序展示完成日志，每条含任务类型图标、成员头像、成员名、任务名、相对时间

- [ ] 撤销完成后对应记录实时消失

### 7.2 待办页家庭动态

- [ ] 待办页底部展示"家庭动态"区域，最近 5 条全家庭养护记录
- [ ] 每条含植物名维度（谁给什么植物做了什么）
- [ ] 家庭无记录时该区域不渲染

- [ ] 新增 `by_plantId_and_completedAt` 索引，植物维度查询走索引

## 9. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 养护记录区放在 L3 和 L4 之间 | 养护计划（当前任务）和植物档案（静态资料）之间，时间线是"已发生的养护行为"，语义上承上启下 |
| D2 | 默认折叠 | 与植物档案区一致的交互模式，不增加详情页默认信息负载 |
| D3 | 不新增独立页面 | 时间线嵌入详情页+待办页，不增加导航层级，保持"三步内完成"的设计原则 |
| D4 | 不改 taskCompletionLogs schema | 现有字段足够展示基本时间线，备注/照片留给下个迭代，避免一次改动过大 |
| D5 | cursor-based 分页而非 offset | Convex 推荐的分页模式，对实时订阅友好 |
| D6 | 待办页动态只展示 5 条 | 待办页核心是任务执行，动态只是轻量补充，不喧宾夺主 |
