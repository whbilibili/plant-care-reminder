# 植物导航与可发现性迭代（迭代 v0.3）

**版本**：v0.3（已拍板，落地依据）
**日期**：2026-06-15
**关联文档**：基于 [2026-06-11-plant-detail-iteration-spec.md](./2026-06-11-plant-detail-iteration-spec.md)（详情页 v0.2，已落地）的导航层补强
**文档定位**：本文档**不重做详情页内容布局**（v0.2 的五层信息架构保留），只解决一组问题——**用户从列表/待办进入后，搞不清"点了会去哪、操作藏在哪、编辑指的是哪个编辑"，删除确认体验差，以及表单页是死胡同（无返回出口）**。这是一次围绕「心智模型 + 可发现性 + 退出路径」的导航专项迭代。

> 本版已与产品负责人逐条确认，§5「已拍板决策」为最终结论，§3/§4/§9/§10 为落地设计。

## 0. 这次到底在解决什么（先对齐问题，再谈方案）

用户原话：

> "植物列表页的卡片，用了几次，还是没有太明白，引导不是很好。我不知道点击卡片之后会进入详情页还是任务列表页。在任务列表页如何进入编辑页。在编辑页又无法删除和归档植物。"

把这段话拆开，背后是**四个真实的认知断点**，而不是四个功能缺失：

| # | 用户的困惑 | 代码现状（事实核查） | 问题性质 |
|---|-----------|---------------------|----------|
| C1 | "点卡片进详情页还是任务列表页？" | 根本没有"任务列表页"。点卡片 = 进详情页 `/plants/:id`，详情页里内嵌"养护计划区"作为任务清单 | **概念污染**：用户把详情页里的"养护计划区"误当成了一个独立页面 |
| C2 | "卡片上既有铅笔又有箭头，点哪个去哪？" | 铅笔 = 直达编辑页；箭头/卡片体 = 进详情页。两个入口、两条路径 | **入口歧义**：一张卡片给了两个目的地，违反"一卡一意图" |
| C3 | "在'任务列表页'怎么进编辑页？" | 编辑入口在详情页右上角 ⋯ 溢出菜单里，用户没发现 | **可发现性为零**：核心管理操作藏在最不显眼的控件里 |
| C4 | "在编辑页无法删除和归档？" | 删除/归档不在编辑页，在详情页 ⋯ 菜单里；且"删除"是假的——`deletePlant` 未实现，点删除实际执行的是归档 | **位置错配 + 功能欺骗**：用户预期"编辑页能管理这条记录"，实际操作在别处，且删除名实不符 |

**核心判断：v0.2 详情页的内容设计是好的，问题全部出在"导航语义"和"操作可发现性"这一层。** 所以本次不推翻 v0.2，而是在它之上补三件事——收敛入口、显性化管理操作、统一"编辑"语义；并顺手修掉"假删除"这个诚信坑。

两个关键代码事实（影响方案，必须知道）：

1. **`OverflowMenu.tsx:108-115` 的"删除"是假的。** `deletePlant` mutation 未实现，`handleDelete()` 在二次确认后实际调用 `handleArchiveToggle()`——即**点"删除"执行的是"归档"**。这是一个名实不符的坑，用户看到"删除后数据无法恢复"的红字警告，点下去数据其实还在。

## 2. 设计原则（本次迭代的判断依据）

1. **一卡一意图（One Card, One Destination）**：一张卡片只应有一个主目的地。点卡片的任何位置，都去同一个地方——详情页。
2. **详情页是单株植物的唯一管理中枢**：看状态、做任务、改资料、归档/删除，全部在详情页完成。列表页只负责"选哪盆"，不承担管理动作。
3. **高频操作显性、低频操作收纳、危险操作设防**：完成任务（高频）最显眼；编辑资料（中频）一步可达；删除/归档（低频危险）收纳但可发现，且有明确二次确认。
4. **语义不撞车**：明确区分"编辑植物资料"与"编辑养护任务"，文案和入口位置都要让人一眼分清。
5. **名实相符**：UI 上叫"删除"就必须真删除；做不到真删除，就不要叫删除。

## 3. 方案（核心）

### 3.1 列表页卡片：撤掉铅笔，回归"一卡一意图" ✅ 主推

**决策：删除植物卡片上的编辑铅笔图标，整张卡片只有一个动作——进入详情页。**

理由：
- 直接消灭 C2（入口歧义）。卡片不再有"两条路"，用户不需要思考"点哪去哪"。
- 编辑本就是低频操作，没必要在每张卡片上常驻一个入口。把它交给详情页，符合"详情页是管理中枢"的原则。
- 顺带解决反馈 1（铅笔 UI 不和谐）——最好的美化是删掉它，违和感和歧义一起消失。

改造后卡片：

```
┌──────────────────────────────────────────┐
│  ┌────┐  发财树                       ›  │   ← 整卡可点，唯一去向=详情页
│  │图片│  客厅茶几                          │
│  └────┘  💧 浇水  明天到期                 │
└──────────────────────────────────────────┘
```

- 移除铅笔按钮及其 `onEdit` 链路（`PlantCard.tsx` 的 `editIconStyle`、`editPressed` 状态、`onEdit` prop）。
- 保留右侧 `›` 作为"可进入"的视觉暗示（affordance），但它不再是独立按钮，只是装饰性指示符，点击行为与点卡片体一致。
- `PlantListPage.tsx` 移除 `onEdit={() => navigate(.../edit)}` 传参。

> 备选方案（不推荐）：保留铅笔但美化成带圆形 hit-area 的规范按钮。否决理由——治标不治本，C2 入口歧义依然存在，用户照样要在"卡片体 vs 铅笔"之间做选择。

### 3.2 详情页：管理操作"分层落地"，并移除 ⋯ 菜单 ✅ 已拍板

**核心决策（C 方案）：不同操作按"频率 × 风险"分层放置，而非全部塞在一处。同时彻底移除 ⋯ 溢出菜单。**

判断依据：编辑、归档、删除三者的使用频率和危险程度天差地别，硬塞同一处必然别扭。按"高频安全→显性、低频危险→收纳设防"分层：

| 操作 | 频率 | 风险 | 落点 | 样式 |
|------|------|------|------|------|
| 编辑植物资料 | 中频 | 安全 | **Hero 区右侧常驻编辑图标按钮** | lucide `Pencil`，一眼可见、一步可达 |
| 归档植物 | 低频 | 中（可恢复） | **档案区底部"管理"分区** | 次级按钮 |
| 删除植物 | 低频 | 高（不可逆） | **档案区底部"管理"分区** | danger 红色文字 + 独立确认面板（见 §4）|

**布局示意：**

```
┌──────────────────────────────────────────┐
│  ┌────┐  发财树                    [✎]   │   ← Hero 右侧常驻编辑入口（中频安全）
│  │ 图 │  客厅 · 正在养护中                 │
│  └────┘                                  │
└──────────────────────────────────────────┘
            … L2 需要处理 / L3 养护计划 …
┌──────────────────────────────────────────┐
│  📝 植物档案                        [▴]   │   ← 默认折叠（沿用 v0.2 D5）
│  描述 / 备注 / 添加时间…                    │
│  ──────────────────────────────────────  │
│  管理这盆植物                              │
│  [📥 归档植物]        低频、可恢复          │
│  [🗑 删除植物]        危险、不可逆 →独立确认 │
└──────────────────────────────────────────┘
```

**为什么编辑放 Hero、归档/删除放档案区？**
- 编辑是你最可能用的管理动作，放 Hero 右侧常驻，可发现性最高、路径最短——彻底解决 C3。
- 归档/删除是低频且危险的"生命周期操作"，放在档案区（这盆植物的元信息归属地）语义最顺；默认折叠还能天然避免危险操作误触，符合"危险操作收纳设防"。

**⋯ 溢出菜单：移除。** ✅ 已拍板
原本 ⋯ 菜单存在的唯一理由就是装编辑/归档/删除这三个操作。现在三者已在页面上显性化，菜单要么空、要么和页面重复——重复入口只会增加认知负担（"这俩是不是不一样？"）。因此 `DetailNavBar` 右侧的 `EllipsisVertical` 按钮及整个 `OverflowMenu` 组件直接删除，顶栏只保留 `[← 返回]` + `[植物名称]`，更聚焦干净。

### 3.3 统一"编辑"语义，消除撞车 ✅

明确两个 edit 的文案和入口：

| 操作 | 含义 | 文案 | 入口位置 |
|------|------|------|----------|
| 编辑植物资料 | 改名称/位置/图片/描述/备注 | **「编辑植物资料」**（不要只写"编辑"） | §3.2 的管理区块 + ⋯ 菜单 |
| 编辑养护任务 | 改某条浇水/施肥的间隔等 | 养护计划任务行点击进入，行尾加一个 `✎` 暗示「可编辑」 | L3 养护计划区任务行 |

补充：L3 养护计划区任务行目前**点击整行直接跳编辑页，但没有任何"可点"的视觉暗示**（用户根本不知道行可以点）。本次在每行尾部补一个浅色 `✎` 或 `›`，明确"这行能点、点了去编辑这条计划"。

### 3.4 引导气泡：本次不做 ✅ 已拍板

首次进入的一次性引导气泡（原讨论项）**本次不做**。理由：3.1（一卡一意图）+ 3.2（管理操作显性化）落地后，导航心智已大幅清晰，无需额外引导补丁。保留为后续候选，若数据仍显示用户找不到任务编辑入口再加。

## 4. 真删除 + 删除二次确认重设计 ⚠️ 本次重点

### 4.1 实现真删除（C4 的根）✅ 已拍板

**问题**：`OverflowMenu.tsx:108-115` 的"删除植物"是假的——`deletePlant` mutation 未实现，点"确认删除"实际调用 `handleArchiveToggle()` 执行归档，但 UI 写着"删除后所有数据将无法恢复"。名实不符，损害信任。

**决策：实现真正的 `deletePlant` mutation。** 归档与删除作为两个独立、并存的操作：
- **归档** = 从看板暂时收起，养护任务和历史保留，可随时恢复。
- **删除** = 永久移除，级联清理该植物的 `plantTasks` 与 `taskCompletionLogs`，不可恢复。

实现红线（遵循 ARCHITECTURE.md 数据层约束）：
- mutation 必须校验当前用户属于该植物的 `familyId`，禁止跨家庭删除。
- 删除为级联操作：先删 `taskCompletionLogs` → 再删 `plantTasks` → 最后删 `plants` 本体，保证不留孤儿数据。
- 删除成功后 `navigate("/plants", true)` 返回列表（replace，避免回退到已删除的详情页）。

### 4.2 删除二次确认重设计 ✅ 已拍板（反馈 6）

**现状问题（见 2026-06-15 截图）**：点"删除植物"后，菜单项**原地变红成"确认删除"+ 下方红字警告**。三个硬伤：

1. **把破坏性确认塞进随时会误关的轻量溢出菜单里**——菜单点外部就关，承载"永久删除"过于脆弱。
2. **"原地变红字"是最弱的确认形态**——不阻断、不聚焦。两次点击落在**同一位置**，手指没移动就能连点完成删除，等于没有二次确认（删除误触经典陷阱）。
3. **归档与删除视觉权重相近**，只隔一条线，该归档时易误删。

而且讽刺的是——项目里**归档操作已经用了规范的 `ConfirmSheet`，更危险的删除反而用了最简陋的原地变红**，风险与确认强度完全倒挂。

**决策：删除改用独立 danger `ConfirmSheet`（底部弹出确认面板），与归档体验对齐且权重更高。**

设计要点（破坏性操作 UI 原则）：
- **从底部滑出独立面板 + 半透明遮罩压暗背景**，物理上打断操作流，强制聚焦。
- **标题点明对象与后果**：`确认删除「{植物名}」吗？`
- **正文说清代价**：`这盆植物的资料、养护计划和完成记录都会被永久删除，无法恢复。`
- **两个按钮拉开距离、错开位置**：`取消`（中性，默认焦点）+ `确认删除`（`variant="danger"` 红色实心）。确认按钮**不与触发位置重叠**，杜绝连点误触。
- 复用现有 `ConfirmSheet` 组件（归档已在用），新增/启用 danger 实心变体。
- 删除进行中按钮显示 loading，禁止重复提交。

> 不引入"输入植物名才能删除"这类重确认——对家庭小应用过重。"独立 danger sheet + 取消默认聚焦 + 按钮错位"已足够设防。

## 6. 表单页返回出口缺失（反馈 7）✅ 已拍板

### 6.1 问题（已核实代码）

`EditPlantPage.tsx` **没有任何返回/取消按钮**——直接渲染 `PlantForm`，只有一个"更新植物"提交按钮，提交成功才跳转。用户进编辑页后若不想改，**没有任何退出出口**，只能点底部 Tab 强行跳走。这是典型的**死胡同页面（dead-end）**，违反"任何页面都必须有明确退出路径"。

**根因是系统性的**：项目有两套页头——详情类用带返回的 `DetailNavBar`，表单类用 `PageHeader`，而 `PageHeader` **结构上就没有返回按钮的位置**（只有 eyebrow/title/description/actions）。因此所有走 `PageHeader` 的表单页（编辑植物、创建植物、创建任务、编辑任务）**大概率都缺返回入口**——这是组件层缺陷，不是单页问题。

### 6.2 决策：统一表单页返回栏 ✅

**复用 `DetailNavBar` 的设计语言，为表单页提供统一的顶部返回栏**（新建通用 `FormNavBar`，或给 `PageHeader` 增加可选返回按钮槽）。
- 左上角：`← 取消` / `← 返回`（lucide `ChevronLeft`，与 `DetailNavBar` 一致）。
- 接入范围：编辑植物、创建植物、创建任务、编辑任务等所有表单页，全 app 页头返回语义一致。
- 点击返回 = `navigate` 回上一页（编辑植物→详情页，创建植物→列表页）。

### 6.3 未保存改动保护 ✅

表单有未保存改动时点返回，弹轻量 `ConfirmSheet`：`放弃未保存的修改？` —— `继续编辑` / `放弃修改`。无改动则直接返回，不打扰。防止误触返回丢失编辑内容。

- 脏检查：表单当前值与 `initialValue` 不一致即为"有改动"。
- 复用现有 `ConfirmSheet`。

## 6A. 删除任务的二次确认（反馈 8）🟡 待确认

### 6A.1 问题（已核实代码）

`DeleteTaskAction.tsx` 的删除确认，和反馈 6 的删除植物确认是**同一个坏模式**——点"删除任务"后，在按钮**正下方原地展开一个红框**（`dialogStyle`：mist 底 + error 描边），框内标题、正文全是 error 红色，"取消 / 确认删除"两个按钮都是 `variant="ghost"` 弱权重。问题：

1. **视觉过载**：整块（eyebrow + 标题 + 正文 + 弹框）全用 error 红，红到失去层次，反而稀释了"确认删除"按钮该有的警示重量。
2. **不统一**：项目里已经有规范的底部 `ConfirmSheet`（支持 `danger-solid` 变体 + 遮罩 + focus trap + 动画），归档已在用；唯独两个删除走了简陋的原地变红，体验割裂。
3. **连点误触风险**：确认框就在触发按钮正下方，"删除任务→确认删除"两次点击落点相近。
4. **缺物理打断**：原地展开不压暗背景、不强制聚焦，破坏性操作分量不足。

> 这与反馈 6 是**同一问题的两个实例**，应一并按同一标准修复。

### 6A.2 决策：统一用 danger ConfirmSheet 🟡

删除任务复用现有 `ConfirmSheet`（`variant="danger-solid"`），与删除植物（反馈 6）**同款交互、同款组件**：
- 从底部滑出独立面板 + 遮罩压暗背景，物理打断。
- 标题：`确认删除「{任务名}」吗？`；正文：`删除后，这条养护提醒会从这盆植物上彻底消失，无法恢复。`
- `取消`（默认聚焦）+ `确认删除`（danger-solid 红实心），按钮与触发位置错开。
- 删除中 loading、禁重复提交。
- 上方"删除任务"区块的红色降级：仅 eyebrow/按钮保留警示色，标题正文回归常规文本色，避免整块刷红。
- `DeleteTaskAction` 内联的 `dialogStyle` 系列样式整体废弃。

> 收益：删除植物 + 删除任务两处破坏性确认彻底统一到一套组件、一套视觉，全 app 危险操作语言一致。

## 6B. 无图植物卡片的兜底展示（反馈 9）🟡 待确认

### 6B.1 真实根因：CSS 定位 bug（已二次核实，刷新仍复现 → 排除动画竞态）

> 第一版误判为"动画竞态"，但用户反馈**刷新后依然不展示**，且只发生在**无图卡片**上 → 排除竞态，定位到确定性的 CSS bug。

`PlantImage.tsx` 的**无图分支**直接返回一个 `position: absolute; inset: 0` 的占位 div：

```tsx
// 无图分支（第 17–25 行）
if (!imageUrl || status === "error") {
  return (
    <div style={placeholderStyle}>   {/* placeholderStyle = position:absolute; inset:0 */}
      <span>🌿</span>
    </div>
  );
}
```

而它在 `PlantCard` 里的父容器 `imageWrapStyle`（80×80 的图槽）**没有 `position: relative`**：

```ts
const imageWrapStyle = { width:"80px", height:"80px", overflow:"hidden", background:"var(--color-mist)" };
// ❌ 缺 position: relative
```

**后果**：`position: absolute` 的占位 div 找不到 positioned 父级约束，脱离 80×80 图槽——图槽塌成 0 内容，占位 🌿 相对更上层（或 viewport）的定位上下文铺开，于是整张卡片视觉错位、文字被覆盖或挤没，只剩一个孤零零的 🌿。

**对比印证**：PlantImage **有图分支**用的是 `wrapStyle`（`position: relative`），占位被正确约束在图槽内 → 有图卡片一切正常。问题**只在无图分支**，与"刷新仍复现 + 只发生在无图卡"完全吻合。

**数据无辜**：后端 `listPlantsWithNextDue` 正常返回 name/location/description，是渲染层把它毁了。

### 6B.2 决策：修 CSS 定位 + 规范默认头像 🟡

1. **修定位（根因）**：给 `imageWrapStyle` 加 `position: relative`（图槽自己成为定位上下文）；或让无图分支复用与有图分支同一套 `wrapStyle` 包裹。优先前者——一行修复，且对有图/无图都更稳。
2. **规范默认头像**：见 §6B.3 设计方案。
3. **验证目标**：新增一盆"只填文字、不传图"的植物，列表项必须**完整展示文字 + 规范默认头像**，反复刷新均稳定。

### 6B.3 默认头像设计方案（基于 frontend-design 技能评估）

设计目标：无图时不应是"占位符的将就感"，而应是**一个有设计感、成体系的默认头像**，与 Organic Biophilic 风格统一，且全 app 任何用到植物缩略图的地方（列表卡、详情 Hero、归档项）都能复用。

**评估三个候选方向：**

| 方向 | 做法 | 评估 |
|------|------|------|
| A 单色 emoji（现状） | 🌿 emoji | ❌ 跨系统渲染不一，无"头像"感，与反馈2 emoji 规范化诉求冲突 |
| B 单线 lucide icon | `Sprout` icon + mist 底 | 🟡 规范、轻量，但"扁平描边图标"略显单薄，缺少头像的体量感 |
| **C 渐变底 + 线性 icon（推荐）** | botanical 柔和渐变底 + 居中 lucide `Sprout`（leaf 色）+ 细微纹理 | ✅ 有体量、有设计感、成体系，最像"真头像"的兜底 |

**推荐方案 C 细节（production 级）：**
- **底**：从 `--color-mist` → 略深的 leaf-tint 的**柔和对角线性渐变**（135deg），制造微妙的"有机生长"光感，而非死板纯色。
- **图标**：居中 lucide `Sprout`（嫩芽，比 `Leaf` 更有"新生植物"语义），经统一 `Icon` 壳渲染，`colorVar="--color-leaf"`，`strokeWidth` 略细（1.75）显精致。
- **可识别性增强（可选）**：用植物名首字派生一个**稳定的色相偏移**（同名永远同色），让不同植物的默认头像有细微色差、便于区分——类似 GitHub/Notion 的默认头像逻辑。纯 CSS/计算，零额外资源。
- **尺寸自适应**：同一组件吃 `size` prop，列表卡（80）、详情 Hero（大图）、归档项（小图）共用，圆角随容器。
- **token 纪律**：全部走 CSS 变量，**禁字面 hex**；图标走 `icon-system-v0.4`。
- **降级**：`prefers-reduced-motion` / 低端机不加任何动效，渐变是静态的，零性能负担。

> 一句话：默认头像 = 柔和 botanical 渐变底 + 居中嫩芽线性 icon（+ 可选按名派生色相），抽成可复用的 `PlantAvatar` 组件，替换现有 emoji 占位。

## 8. 验收标准

1. 列表卡片只有一个目的地，点任意位置都进详情页（C2 消除）
2. 详情页 Hero 区有常驻可见的"编辑资料"入口；档案区有常驻可见的归档/删除入口，无需任何菜单即可发现（C3 消除）
3. ⋯ 溢出菜单及 `OverflowMenu` 组件已彻底移除
4. "编辑植物资料"与"编辑养护任务"文案/入口清晰可分（语义不撞车）
5. "删除"为真删除，名实相符；删除后数据真正消失且不影响他人（familyId 隔离）（C4 消除）
6. 删除走独立 danger `ConfirmSheet`，取消默认聚焦、确认按钮与触发位置错开，无原地变红
7. 养护计划区任务行有"可点击编辑"的视觉暗示
8. 所有表单页（编辑/创建植物、编辑/创建任务）均有明确返回出口；有未保存改动时返回会二次确认
9. 所有改动遵守 `icon-system-v0.4`（lucide + Icon 壳，禁字面 hex）与 botanical token
