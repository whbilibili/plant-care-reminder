# 多图上传与养护图集

**版本**：v1.0
**日期**：2026-06-22
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（§7.2 植物管理）、[养护历史时间线](./2026-06-17-care-history-timeline-spec.md)（v1.1 联动）
**文档定位**：让植物从"一张封面照"进化为"生长日记"，支持多图上传、时间轴浏览和任务完成时附加照片，增强养护的记录感和成就感。

---

## 0. 本次迭代要解决的核心问题

MVP 每盆植物只能上传一张照片作为封面，用户无法记录植物的生长变化。这导致两个体验缺口：

- **"养了半年，完全看不出长了多少"** —— 缺乏生长对比：用户想回顾植物从小苗到茂盛的过程，但只有一张静态封面。
- **"每次浇完水想拍一张记录，但没地方放"** —— 缺乏养护随手记：完成任务时的植物状态是最有价值的记录时刻，但系统不支持附加照片。

本次通过引入**植物图集**和**任务完成附图**，让养护过程可视化、有积累感。

---

## 1. 产品目标

让用户能够为每盆植物积累一本"生长相册"：随时上传照片记录生长变化，完成任务时顺手拍一张作为养护日志的视觉补充。最终让用户在回顾时能直观看到"这盆植物在我手里是怎么一点点长大的"。

---

## 2. 用户场景

### 场景 1：主动上传生长记录

用户发现阳台的绿萝长出了新叶子，打开植物详情页，在图集区域点击"添加照片"，拍照或从相册选取，照片按时间排列在图集中。一个月后回看，能清晰看到绿萝从 3 片叶子长到了 10 片。

### 场景 2：完成任务时附加照片

用户给栀子花浇完水，点击"完成"时弹出可选的"拍张照片记录一下？"入口。用户拍了一张花苞照片，这张照片同时出现在植物图集和养护历史时间线中，形成"什么时候做了什么 + 当时长什么样"的完整记录。

### 场景 3：浏览生长对比

用户打开植物详情页的图集区域，左右滑动浏览按时间排列的照片。最早的照片是刚买回来的小苗，最新的是枝繁叶茂的状态。用户获得强烈的成就感。

---

## 3. 功能设计

### 3.1 植物详情页 —— 图集区

**位置**：在 Hero 区（植物封面大图）下方、L2 养护计划区上方，新增"图集"横向滚动区域。

**展示规则**：

- 图集区以横向滚动卡片形式展示，每张缩略图 80×80 圆角方形，间距 8px。
- 按上传时间倒序排列（最新在最左）。
- 首位固定一个"+"添加按钮，点击触发图片选择（拍照/相册）。
- 默认展示最近 10 张缩略图，末尾有"查看全部（N）"入口，点击进入全屏图集浏览模式。
- 图集为空时，展示单个"+"按钮 + 引导文案"记录植物的成长变化"。

**全屏浏览模式**：

- 全屏 swiper，左右滑动切换，底部显示拍摄日期。
- 支持双指缩放查看细节。
- 右上角"删除"按钮（二次确认）。
- 顶部返回按钮关闭全屏模式。

### 3.2 任务完成附图

**触发时机**：用户点击"完成"按钮后，任务完成成功的正反馈动画结束时，底部浮现一个轻量入口"📷 记录一下？"，持续 3 秒后自动消失。

**交互流程**：

- 用户点击"记录一下？"后，弹出系统图片选择器（拍照/相册）。
- 选择照片后，照片自动关联到该植物的图集，并在 `taskCompletionLogs` 中记录 `imageStorageId`。
- 用户不点击或等待消失，不影响任务完成流程（完成是即时的，附图是可选的后置动作）。

### 3.3 植物封面与图集的关系

**规则**：

- 植物封面（Hero 大图）仍由 `plants.imageStorageId` 单独控制，不自动跟随图集最新图。
- 用户可在全屏浏览模式中长按某张图片选择"设为封面"。
- 图集中的照片独立于封面存在，删除封面不影响图集，删除图集照片不影响封面。

### 3.4 存储与性能

**上限**：每盆植物最多 20 张图集照片（不含封面）。达到上限后"+"按钮变为不可用态，提示"图集已满，请删除旧照片后再添加"。

**缩略图策略**：

- 上传时前端压缩生成缩略图（短边 200px，JPEG quality 0.7），与原图分别存储。
- 图集横向滚动区加载缩略图，全屏浏览加载原图。
- 缩略图 storageId 与原图 storageId 成对存储。

**加载策略**：

- 图集区域使用懒加载，进入视口后才请求缩略图 URL。
- 全屏浏览模式预加载当前图的前后各 1 张原图。

## 4. 数据模型变更

### 4.1 plants 表扩展

新增字段：

```
gallery: v.optional(v.array(v.object({
  imageStorageId: v.id("_storage"),
  thumbnailStorageId: v.id("_storage"),
  uploadedBy: v.id("users"),
  uploadedAt: v.number(),  // UTC timestamp
})))
```

约束：

- `gallery` 数组最大长度 20。
- `v.optional` 保证向后兼容，老数据无需迁移。
- 数组元素按 `uploadedAt` 降序排列（写入时维护顺序）。

### 4.2 taskCompletionLogs 表扩展

新增可选字段：

```
imageStorageId: v.optional(v.id("_storage"))
```

用于记录任务完成时附加的照片。该照片同时写入对应植物的 `gallery` 数组。

### 4.3 索引评估

- 无需新增索引。图集数据内嵌在 `plants` 文档中，随植物详情一起加载。
- `taskCompletionLogs` 的现有索引足够支持时间线展示附图。

## 5. 新增接口

| 接口 | 类型 | 权限 | 行为 |
|------|------|------|------|
| `addPlantGalleryImage` | mutation | 家庭成员 | 上传图片到 `_storage`，将 {imageStorageId, thumbnailStorageId, uploadedBy, uploadedAt} 追加到 `plants.gallery` 数组头部；校验数组长度 ≤ 20 |
| `removePlantGalleryImage` | mutation | 家庭成员 | 从 `plants.gallery` 中移除指定 imageStorageId 的元素，并删除 `_storage` 中的原图和缩略图 |
| `setPlantCoverFromGallery` | mutation | 家庭成员 | 将指定图集照片的 imageStorageId 设为 `plants.imageStorageId`（替换封面） |
| `generateUploadUrl` | mutation | 家庭成员 | 复用现有的 Convex storage upload URL 生成逻辑 |

### 现有接口改造

- `completePlantTask`：新增可选参数 `imageStorageId?: Id<"_storage">`。若提供，写入 `taskCompletionLogs.imageStorageId`，同时追加到对应植物的 `gallery`。

## 6. 交互设计要点

### 6.1 图集区视觉

- 缩略图卡片：80×80，border-radius: 8px，overflow: hidden。
- "+"按钮：同尺寸虚线边框卡片，居中 Plus 图标（Lucide），颜色 `--color-muted`。
- 横向滚动：隐藏滚动条，支持触摸滑动。
- 图集区标题不单独显示（空间紧凑），通过"+"按钮和照片本身传达语义。

### 6.2 全屏浏览

- 背景 `--color-ink` 全黑，图片居中自适应。
- 底部日期文案用 `--color-paper` 白色，格式"2026年6月15日"。
- 删除确认使用统一 ConfirmSheet 组件（danger 样式）。

### 6.3 完成附图入口

- 浮现动画：从底部 slide-up，opacity 0→1，持续 3 秒后 fade-out。
- 样式：pill 形状，背景 `--color-surface`，文案 + 相机图标。
- 不阻塞任何操作，用户可忽略。

---

## 7. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 图集已满（20 张）再上传 | "+"按钮 disabled + toast 提示"图集已满" |
| 上传失败（网络/文件过大） | toast 提示失败原因，不影响已有图集 |
| 图片文件过大 | 前端压缩后上传；原图超过 10MB 提示"图片过大，请选择较小的照片" |
| 删除正在作为封面的图集照片 | 允许删除，封面不受影响（封面是独立的 storageId 引用） |
| 植物归档后 | 图集只读，不允许添加/删除，全屏浏览仍可用 |
| 完成附图时上传失败 | 任务完成不受影响（已成功），toast 提示"照片保存失败" |

## 10. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 图集内嵌 plants 文档而非独立表 | 每盆植物最多 20 张，数据量可控；内嵌避免额外 join，随植物详情一次性加载 |
| D2 | 封面与图集独立 | 封面是"代表照片"语义，图集是"时间记录"语义，两者职责不同；避免自动切换封面带来的困惑 |
| D3 | 完成附图为可选后置动作 | 任务完成是核心路径，不能被拍照阻塞；附图是锦上添花，3 秒消失保证不打扰 |
| D4 | 上限 20 张 | 平衡存储成本和用户需求；20 张足够记录一年的月度变化；超出可删旧增新 |
| D5 | 前端压缩生成缩略图 | 避免后端图片处理依赖（Convex 无原生图片处理）；前端 canvas 压缩成熟可靠 |
| D6 | 归档植物图集只读 | 与归档语义一致（不再活跃养护），但保留浏览能力供回忆 |
