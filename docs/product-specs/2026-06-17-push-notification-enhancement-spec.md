# 推送通知体验增强

**版本**：v1.0
**日期**：2026-06-17
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（§5.4 到期提醒）、[caveats.md](../caveats.md)（iPhone PWA Web Push 脆弱性）
**文档定位**：从"能推就推"升级为"智能推、推得准、点得到"，解决 MVP 推送通知的三个体验短板。

---

## 0. 本次迭代要解决的核心问题

MVP 的推送通知系统功能可用但体验粗糙，存在三个短板：

> - **"通知点进去只到首页，我还得自己翻到那盆植物"** —— 推送内容有植物名和任务类型，但点击后跳转的是 `/todo` 待办页全局视图，用户需要自己定位到具体植物。路径断裂。
> - **"早上 6 点被通知吵醒，就为了告诉我该浇水"** —— cron 每 30 分钟扫描一次，任务刚到期就推，不管用户是否在睡觉。没有时间偏好。
> - **"一天收了 8 条通知，每盆植物一条"** —— 多盆植物同一时段到期时，逐条推送，通知轰炸。没有聚合策略。

本次通过三项改造——通知深链跳转、推送时间偏好、到期任务聚合推送——将推送从"能用"升级为"好用"。

---

## 1. 产品目标

让推送通知成为用户回到 app 的高效入口而非烦扰源。用户收到通知时知道要做什么，点击后直达目标；在自己想要的时间段收到提醒，不在休息时被打扰；多个任务到期时收到一条聚合通知而非消息轰炸。

---

## 2. 用户场景

### 场景 1：通知深链——点击直达

用户收到推送"「发财树」浇水任务已到期"，点击通知后 app 直接打开发财树的详情页，顶部"需要处理"区高亮浇水任务，用户直接点完成。整个路径从"收到通知"到"完成任务"控制在 2 步内。

### 场景 2：推送时间偏好——不被打扰

用户在设置页的通知区域设置"每天 8:00 提醒"。系统不再在任务到期的瞬间推送，而是在每天早上 8 点统一推送当天及已逾期的所有待办。用户在早餐时间查看今天要照顾哪些植物，从容安排。

### 场景 3：聚合推送——一条就够

用户家里 5 盆植物今天到期。系统不发 5 条通知，而是发 1 条："你有 5 个养护任务待完成——发财树（浇水）、绿萝（施肥）等"。点击后进入待办页查看全部。

---

## 3. 功能设计

### 3.1 通知深链跳转

**核心改动**：推送 payload 的 `url` 字段从固定的 `/todo` 改为动态的植物详情页路径。

**推送 payload 变更**：

| 字段 | 当前值 | 改造后 |
|------|--------|--------|
| `url` | `"/todo"` | 单任务：`"/plants/{plantId}"`；聚合推送：`"/todo"` |

**前端 Service Worker 改造**：

- `notificationclick` 事件处理中，从 `event.notification.data.url` 取路径，调用 `clients.openWindow(url)` 或 `client.navigate(url)` 定位到对应页面。
- 如果 app 已打开（在前台），使用 `postMessage` 通知主线程执行路由跳转，避免开新窗口。

**交互细节**：

- 单任务通知点击 → 植物详情页，Action Zone（需要处理区）中对应任务高亮。
- 聚合通知点击 → 待办页，无特殊高亮（用户已知要处理多个任务）。

### 3.2 推送时间偏好

**数据模型**：在 `familyMembers` 表（或 `users` 表）增加通知偏好字段。

建议放在 `users` 表（个人偏好，不随家庭变化）：

```
notificationPreferences: v.optional(v.object({
  preferredHour: v.number(),     // 0-23，偏好推送时段的小时数（用户本地时区）
  preferredMinute: v.number(),   // 0-59
  timezone: v.string(),          // IANA 时区标识符，如 "Asia/Shanghai"
  quietHoursEnabled: v.boolean(),
  quietHoursStart: v.optional(v.number()),  // 0-23
  quietHoursEnd: v.optional(v.number()),    // 0-23
}))
```

**最小可行方案（建议 v1.0 只做这些）**：

- 只实现 `preferredHour` + `timezone`，不做完整的静默时段。
- 用户在设置页通知区域选择"每天几点提醒我"（默认 8:00）。
- 保留一个"立即推送"选项（当前默认行为），给不想等的用户。

**后端逻辑改造**：

- `processDueTaskNotifications` action 改为：先查询用户的 `preferredHour` + `timezone`，将用户本地偏好时间转换为 UTC，只在匹配窗口内发送。
- cron 间隔维持 30 分钟不变（保证 preferredHour 精度在 ±30 分钟内，对日常提醒足够）。
- 如果用户未设置偏好（`notificationPreferences` 为 undefined），沿用当前行为（到期即推）。

### 3.3 到期任务聚合推送

**聚合规则**：

- 同一用户在同一推送窗口内如果有 ≥2 个任务到期，合并为一条聚合通知。
- 聚合通知 title：`你有 N 个养护任务待完成`。
- 聚合通知 body：列出前 3 个任务（"发财树（浇水）、绿萝（施肥）、龟背竹（喷雾）"），超过 3 个追加"等 N 个任务"。
- 聚合通知 url：`"/todo"`（跳待办页全局视图）。
- 聚合通知 tag：`"daily-summary"`（覆盖同 tag 的旧通知，避免堆积）。

**单任务仍单独推送的情况**：

- 同一窗口内只有 1 个任务到期，保持当前行为，单条推送 + 深链到植物详情。

## 4. 对现有逻辑的影响

### 4.1 数据模型变更

| 变更 | 表 | 字段 | 类型 |
|------|---|------|------|
| 新增 | `users` | `notificationPreferences` | `v.optional(v.object({...}))` |

不需要数据迁移——新字段为 `v.optional`，现有用户默认无偏好，沿用当前行为。

### 4.2 现有接口改造

| 接口 | 当前逻辑 | 改造后 |
|------|---------|--------|
| `processDueTaskNotifications` | 直接遍历到期任务逐条推送 | 先按用户分组，检查偏好时间窗口，聚合后推送 |
| `sendPushToEndpoints`（内部调用） | 每个任务独立 payload | 聚合任务时生成合并 payload |

### 4.3 新增接口

| 接口 | 类型 | 权限 | 行为 |
|------|------|------|------|
| `updateNotificationPreferences` | mutation | 登录用户 | 更新 `users.notificationPreferences` |
| `getNotificationPreferences` | query | 登录用户 | 读取当前用户的通知偏好 |

### 4.4 前端改动范围

| 文件 | 改动 |
|------|------|
| `sw.js` / Service Worker | `notificationclick` 处理深链 url |
| `NotificationPromptCard.tsx` | 在通知开启状态下，增加"提醒时间"设置入口 |
| 新增 `NotificationPreferencesSheet.tsx` | 时间选择器（小时选择），保存偏好 |
| `FamilySettingsPage.tsx` | 通知区域展示当前偏好状态（"每天 8:00 提醒"） |

## 5. 交互设计要点

### 5.1 通知偏好设置

- 入口：设置页通知区域，通知已开启状态下展示"提醒时间"行。
- 点击后弹出 Sheet，展示一个小时选择器（滚轮或列表选择 0-23 点）。
- 默认值：8:00（早上 8 点）。
- 用户可选"任务到期时立即提醒"（即关闭 preferredHour，沿用当前行为）。
- 时区自动检测（`Intl.DateTimeFormat().resolvedOptions().timeZone`），不需要用户手动选择。

### 5.2 通知文案

**单任务**：

```
标题: 发财树 · 浇水
正文: 「发财树」的浇水任务已到期，记得去处理哦！
```

**聚合（2-3 个任务）**：

```
标题: 你有 3 个养护任务待完成
正文: 发财树（浇水）、绿萝（施肥）、龟背竹（喷雾）
```

**聚合（>3 个任务）**：

```
标题: 你有 5 个养护任务待完成
正文: 发财树（浇水）、绿萝（施肥）、龟背竹（喷雾）等 5 个任务
```

## 6. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 用户设置偏好时间为凌晨 3 点 | 允许（有人喜欢夜间打理植物），不做强制限制 |
| 用户时区变更（旅行/出差） | 前端每次打开 app 检测时区变化，提示用户是否更新 |
| 植物被删除后点击该植物通知 | 深链到详情页发现植物不存在，显示"该植物已被删除"兜底页，引导回待办页 |
| 推送偏好保存失败 | 前端展示 toast 提示重试，不影响已有推送行为 |
| 用户未设置偏好 | 维持当前行为（到期即推），不强制迁移 |

## 7. 验收标准

### 7.1 深链跳转

- [ ] 单任务通知点击后跳转到对应植物详情页
- [ ] 聚合通知点击后跳转到待办页
- [ ] app 已在前台时点击通知，在当前窗口内跳转而非开新窗口
- [ ] 植物已删除时点击通知，展示兜底页

### 7.2 推送时间偏好

- [ ] 设置页通知区域可设置"每天几点提醒"
- [ ] 设置保存后 `users.notificationPreferences` 正确写入
- [ ] 推送仅在用户偏好时间窗口内触发
- [ ] 未设置偏好的用户不受影响，维持到期即推

### 7.3 聚合推送

- [ ] 同一窗口 ≥2 个任务到期时，发 1 条聚合通知
- [ ] 聚合通知 body 列出前 3 个任务名+类型
- [ ] 单任务仍单独推送并带深链

---

## 8. 后续候选（本次不做）

- 静默时段（quietHours）：指定时间段内不推送，到窗口结束后补发
- 任务完成通知推送给其他家庭成员（"妈妈刚给发财树浇了水"）
- 通知历史页面（查看历史推送记录）
- 按植物/任务类型设置通知开关（精细化订阅管理）
- 逾期升级通知（逾期 3 天加重提醒）

## 9. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 偏好放 users 表而非 familyMembers | 通知偏好是个人习惯，不随家庭变化；一个人换家庭不需要重新设置 |
| D2 | 只做 preferredHour，不做完整静默时段 | 最小可行，覆盖 90% 场景（用户只关心"几点提醒我"），静默时段推 v1.1 |
| D3 | 聚合阈值 ≥2 | 2 个任务就聚合，避免"一下子来两条"的轻度骚扰感；1 个任务保持单独推送带深链
