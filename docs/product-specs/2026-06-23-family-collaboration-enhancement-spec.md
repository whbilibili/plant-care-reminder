# 家庭协作增强

**版本**：v1.0
**日期**：2026-06-23
**关联文档**：承接 [迭代储备大纲](./2026-06-17-iteration-backlog-outline.md)（P2-3）、[角色权限管理](./2026-06-17-role-permission-management-spec.md)（权限矩阵依赖）、[推送通知体验增强](./2026-06-17-push-notification-enhancement-spec.md)（通知能力依赖）
**文档定位**：将家庭协作从"所有人看到所有任务"升级为"任务可指派、通知精准推送、完成可互相知晓"，让多人协作养护植物时职责清晰、信息不遗漏。

---

## 0. 本次迭代要解决的核心问题

MVP 的家庭协作模式是"全员共享一切"——所有成员看到所有任务、收到所有推送。当家庭成员超过 2 人时，这种模式带来三个体验缺口：

- **"待办页全是任务，但不知道哪些是我该做的"** —— 职责不清：所有任务对所有人可见且无标记，成员之间容易互相依赖（"我以为你浇了"）或重复操作。
- **"每天收到 8 条推送，其中 6 条不是我负责的植物"** —— 通知噪音：全员推送导致信息过载，用户开始忽略推送甚至关闭通知权限。
- **"老公浇了水我不知道，我又浇了一遍"** —— 完成不可见：成员完成任务后其他人无法及时感知，导致重复操作。

本次通过引入**任务指派**、**精准推送**和**完成通知**三项能力，让家庭协作从"共享"升级为"协作"。

---

## 1. 产品目标

让家庭成员能够明确分工：每个养护任务可以指定一位负责人，只有负责人收到该任务的到期推送；任务完成后，关注该植物的其他成员收到完成通知。最终实现"谁负责谁收到提醒，做完了大家都知道"的协作体验。

---

## 2. 用户场景

### 场景 1：分配浇水任务

三口之家，妈妈负责客厅的植物，爸爸负责阳台的植物，孩子负责自己房间的多肉。妈妈在创建/编辑任务时，将客厅绿萝的浇水任务指派给自己，阳台月季的浇水任务指派给爸爸。此后，只有爸爸收到月季浇水的推送提醒。

### 场景 2：精准推送减少噪音

爸爸之前每天收到 8 条推送（全家所有植物的任务），启用任务指派后，他只收到自己负责的 3 盆阳台植物的推送。推送数量从 8 条降到 3 条，每条都是他需要行动的。

### 场景 3：完成通知

爸爸完成了月季的浇水任务，妈妈收到一条轻量通知："爸爸刚刚完成了月季的浇水 ✓"。妈妈知道月季已经浇过了，不需要再操心。

### 场景 4：未指派任务的兼容

家里有一盆公共区域的吊兰，没有指定负责人。这种情况下所有成员都能看到任务，所有成员都收到推送（与 MVP 行为一致）。任务指派是可选的，不强制。

### 场景 5：负责人请假/忘记

妈妈出差一周，她负责的客厅植物任务逾期了。系统在逾期 24 小时后，向家庭中的其他 admin/owner 成员发送一条补位提醒："妈妈负责的「绿萝 · 浇水」已逾期 1 天，需要有人帮忙处理吗？"

## 3. 功能设计

### 3.1 任务指派

**指派入口**：任务创建/编辑页面，新增"负责人"选择器。

**选择器设计**：

- 位置：排期模式选择器下方。
- 默认值："所有人"（不指派，全员负责，与 MVP 行为一致）。
- 选项列表：当前家庭的所有活跃成员 + "所有人"选项。
- 每个选项展示：成员头像（小圆形）+ displayName + 角色标签。
- 单选（一个任务只能指派给一个人或"所有人"）。

**指派规则**：

- 任何家庭成员（owner/admin/member）都可以被指派任务。
- 任何家庭成员都可以指派任务给任何人（包括自己）。
- 指派不影响任务的可见性——所有成员仍然可以在待办页看到所有任务。
- 指派只影响推送通知的接收人。

**指派变更**：

- owner 和 admin 可以修改任何任务的负责人。
- member 只能修改自己创建的任务的负责人。
- 修改负责人不触发通知（静默变更）。

### 3.2 待办页指派标识

**展示规则**：

- 每个任务卡片上，如果已指派负责人，在任务类型标签旁展示负责人头像（16px 小圆形）。
- 如果负责人是当前用户自己，展示"我的"标签（`--color-leaf` 色）。
- 如果未指派（"所有人"），不展示额外标识（与 MVP 一致）。

**筛选能力**（可选增强）：

- 待办页顶部增加筛选标签："全部" / "我的" / "其他人的"。
- 默认展示"全部"，用户可切换只看自己负责的任务。
- 筛选状态不持久化（每次进入待办页默认"全部"）。

### 3.3 精准推送

**推送规则变更**：

| 任务指派状态 | 推送接收人 |
|------------|-----------|
| 已指派给某成员 | 仅该成员收到到期推送 |
| 未指派（"所有人"） | 全部家庭成员收到推送（MVP 行为） |

**与推送时间偏好的联动**：

- 精准推送仍遵守用户的推送时间偏好（P0-2）。
- 如果被指派人设置了"每天 8:00 推送"，则该任务的推送在 8:00 发出。

**逾期补位提醒**：

- 已指派任务逾期超过 24 小时后，系统向家庭中的 owner 和 admin 发送一条补位提醒。
- 补位提醒内容："「{植物名} · {任务类型}」已逾期 {N} 天（负责人：{成员名}），需要有人帮忙处理吗？"
- 补位提醒只发一次（逾期 24h 时），不重复发送。
- 如果负责人本身就是 owner/admin，补位提醒发给其他 owner/admin。

### 3.4 完成通知

**触发条件**：任何成员完成一个已指派的任务时。

**通知接收人**：

- 如果任务已指派：通知除完成者之外的所有家庭成员。
- 如果任务未指派：不发送完成通知（避免全员推送时再加一层通知造成噪音）。

**通知内容**：

- 标题："{成员名}完成了养护任务"
- 正文："{植物名} · {任务类型} ✓"
- 点击跳转：植物详情页（复用 P0-2 的深链跳转能力）。

**通知频率控制**：

- 完成通知走应用内通知（不走 Push），避免过度打扰。
- 在待办页顶部以轻量 toast 形式展示（如果用户正在使用应用）。
- 如果用户不在应用内，不发送 Push（完成通知是"锦上添花"，不是"必须知道"）。

### 3.5 应用内通知中心（轻量版）

**为什么需要**：完成通知不走 Push，需要一个应用内的消息承载位置。

**设计**：

- 待办页顶部增加一个通知铃铛图标（右上角）。
- 有未读通知时铃铛上显示红点。
- 点击铃铛展开通知列表（下拉面板或跳转通知页）。
- 通知列表展示最近 7 天的完成通知，按时间倒序。
- 每条通知：成员头像 + "{成员名}完成了{植物名}的{任务类型}" + 时间。
- 通知可标记已读（点击即已读）。

**数据存储**：

- 通知数据存入新的 `notifications` 表（见数据模型变更）。
- 保留 30 天，超过 30 天自动清理。

## 4. 数据模型变更

### 4.1 plantTasks 表新增字段

```
// 任务负责人：指派给某个家庭成员。null/undefined 表示"所有人"（全员负责）。
assignedTo: v.optional(v.id("users"))
```

**向后兼容**：`v.optional` 保证老数据无需迁移。老任务的 `assignedTo` 为 undefined，按"所有人"处理。

### 4.2 新增 notifications 表

```
notifications: defineTable({
  familyId: v.id("families"),
  recipientId: v.id("users"),          // 通知接收人
  type: v.union(
    v.literal("task_completed"),        // 任务完成通知
    v.literal("overdue_backup")         // 逾期补位提醒
  ),
  // 通知内容（结构化，前端根据 type 渲染）
  payload: v.object({
    actorId: v.optional(v.id("users")),       // 触发者（完成任务的人）
    actorName: v.optional(v.string()),        // 触发者名称（冗余，避免 join）
    plantId: v.optional(v.id("plants")),      // 相关植物
    plantName: v.optional(v.string()),        // 植物名称（冗余）
    taskType: v.optional(v.string()),         // 任务类型
    assigneeName: v.optional(v.string()),     // 负责人名称（逾期补位时用）
    overdueDays: v.optional(v.number()),      // 逾期天数（逾期补位时用）
  }),
  isRead: v.boolean(),
  createdAt: v.number(),
})
  .index("by_recipientId_and_createdAt", ["recipientId", "createdAt"])
  .index("by_recipientId_and_isRead", ["recipientId", "isRead"])
  .index("by_familyId_and_createdAt", ["familyId", "createdAt"])
```

### 4.3 索引评估

- `plantTasks` 表：无需新增索引。`assignedTo` 的查询场景是"某用户负责的任务"，可通过现有 `by_familyId_and_nextDueAt` 索引查询后在应用层过滤。如果后续性能不足，再考虑新增 `by_assignedTo_and_nextDueAt` 索引。
- `notifications` 表：`by_recipientId_and_createdAt` 支持"我的通知列表"查询；`by_recipientId_and_isRead` 支持"未读数"查询。

## 5. 接口变更

### 5.1 createPlantTask / updatePlantTask 改造

新增可选参数：

```
assignedTo: v.optional(v.id("users"))
```

校验规则：

- `assignedTo` 如果提供，必须是当前家庭的活跃成员。
- 后端校验该 userId 存在于 `familyMembers` 表中且对应家庭。

### 5.2 completeTask 改造

完成任务时，如果任务已指派（`assignedTo` 有值），触发完成通知生成：

- 遍历家庭成员（排除完成者自己），为每人创建一条 `task_completed` 类型的 notification。
- 通知生成为异步操作（scheduled function），不阻塞完成任务的主流程。

### 5.3 推送逻辑改造（cron.ts）

到期任务扫描时的推送接收人判断：

- 如果任务 `assignedTo` 有值：只向该用户发送推送。
- 如果任务 `assignedTo` 为空：向全部家庭成员发送推送（现有逻辑）。

新增逾期补位检查：

- 扫描已指派且逾期超过 24 小时的任务。
- 检查是否已发送过补位提醒（通过 `notifications` 表中是否存在对应记录）。
- 如未发送，向家庭 owner/admin 生成 `overdue_backup` 通知 + Push 推送。

### 5.4 新增接口

| 接口 | 类型 | 权限 | 行为 |
|------|------|------|------|
| `notifications.list` | query | 家庭成员 | 查询当前用户的通知列表（分页，最近 30 天） |
| `notifications.unreadCount` | query | 家庭成员 | 查询当前用户的未读通知数 |
| `notifications.markAsRead` | mutation | 家庭成员 | 标记指定通知为已读 |
| `notifications.markAllAsRead` | mutation | 家庭成员 | 标记所有通知为已读 |
| `notifications.cleanup` | scheduled mutation | 系统 | 每天运行，删除超过 30 天的通知 |

### 5.5 notifications.list 参数

```
args: {
  cursor: v.optional(v.string()),  // 分页游标
  limit: v.optional(v.number()),   // 每页条数，默认 20
}
returns: {
  notifications: Array<{
    _id: Id<"notifications">,
    type: string,
    payload: object,
    isRead: boolean,
    createdAt: number,
  }>,
  nextCursor: string | null,
}
```

## 6. 前端交互设计

### 6.1 任务创建/编辑 — 负责人选择器

- 位置：排期模式选择器下方，独立一行。
- 标签："负责人"。
- 默认值：展示"所有人"（带多人图标）。
- 点击展开成员列表（ActionSheet 或内联下拉）。
- 成员列表每行：头像（24px）+ displayName + 角色标签（owner/admin/member）。
- 选中后收起列表，选择器展示被选中成员的头像 + 名称。
- "所有人"选项始终在列表第一位。

### 6.2 待办页 — 指派标识

- 已指派任务：任务卡片右上角展示负责人头像（16px 圆形）。
- 自己负责的任务：头像旁加"我"标签（微型，`--color-leaf` 背景白色文字）。
- 未指派任务：不展示额外标识。

### 6.3 待办页 — 筛选标签

- 位置：日期标题下方、任务列表上方（如果有季节横幅则在横幅下方）。
- 三个标签："全部"（默认）/ "我的" / "其他人的"。
- 使用 SegmentedControl 或 Chip 组样式。
- 切换时任务列表平滑过渡（fade 动画）。
- "我的"筛选逻辑：`assignedTo === currentUserId` 或 `assignedTo === undefined`（未指派的任务对所有人可见）。

### 6.4 通知铃铛

- 位置：待办页右上角（与页面标题同行）。
- 图标：Bell（Lucide），颜色 `--color-ink`。
- 未读红点：有未读通知时，铃铛右上角展示红色圆点（6px）。
- 点击跳转到通知列表页。

### 6.5 通知列表页

- 页面标题："通知"。
- 顶部"全部标为已读"文字按钮（右上角）。
- 列表每条通知：
  - 左侧：触发者头像（32px）。
  - 中间：通知文案（"{成员名}完成了{植物名}的{任务类型}"）+ 时间（相对时间，如"2小时前"）。
  - 未读态：左侧有绿色圆点标记。
  - 点击通知：标记已读 + 跳转到对应植物详情页。
- 空状态："暂无通知"+ 插图。
- 下拉加载更多（分页）。

### 6.6 完成任务时的即时反馈

- 如果其他成员正在使用应用，且有人完成了任务：
  - 待办页顶部滑入一条轻量 toast："{成员名}刚完成了{植物名}的{任务类型} ✓"。
  - Toast 3 秒后自动消失。
  - 这依赖 Convex 的实时订阅能力（notifications 表的 query 自动更新）。

## 7. 权限与安全

### 7.1 指派权限

| 操作 | owner | admin | member |
|------|-------|-------|--------|
| 指派任务给任何人 | ✅ | ✅ | ❌ |
| 指派自己创建的任务 | ✅ | ✅ | ✅ |
| 修改他人创建的任务的负责人 | ✅ | ✅ | ❌ |
| 查看所有任务（含他人负责的） | ✅ | ✅ | ✅ |

### 7.2 通知隔离

- 通知数据以 `familyId` 隔离，用户只能查询自己所属家庭的通知。
- `recipientId` 确保用户只能看到发给自己的通知。
- 标记已读操作校验 `recipientId === currentUserId`。

### 7.3 成员退出处理

- 成员退出家庭后，其被指派的任务自动变为"所有人"（`assignedTo` 清空）。
- 退出成员的历史通知保留（但该成员无法再访问）。
- 退出操作触发一次性检查：遍历该成员被指派的所有任务，清空 `assignedTo`。

## 8. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 家庭只有 1 个成员 | 负责人选择器只有"所有人"和自己两个选项；功能可用但价值有限 |
| 被指派人关闭了推送权限 | 推送发送失败静默处理（与现有逻辑一致）；不自动转发给其他人 |
| 被指派人被移除出家庭 | 其负责的任务 assignedTo 清空，恢复全员推送 |
| 同时完成同一任务（并发） | Convex OCC 机制保证只有一次完成生效；完成通知只发一次 |
| 通知量过大（大家庭频繁完成） | 30 天自动清理 + 分页加载；前端只展示最近 50 条 |
| 负责人自己完成任务 | 不给自己发完成通知（排除完成者） |
| 未指派任务被完成 | 不发送完成通知（避免全员推送场景下的通知噪音） |
| 逾期补位提醒的负责人是唯一的 admin/owner | 不发送补位提醒（没有其他人可以补位） |

## 9. 验收标准

### 9.1 任务指派

- [ ] 任务创建/编辑页展示负责人选择器
- [ ] 可选择"所有人"或具体家庭成员
- [ ] 选择的负责人正确存入 plantTasks.assignedTo
- [ ] owner/admin 可修改任何任务的负责人
- [ ] member 只能修改自己创建的任务的负责人

### 9.2 精准推送

- [ ] 已指派任务到期时，只有负责人收到推送
- [ ] 未指派任务到期时，全部成员收到推送（MVP 行为不变）
- [ ] 推送仍遵守用户的时间偏好设置

### 9.3 完成通知

- [ ] 已指派任务完成后，其他成员收到应用内通知
- [ ] 未指派任务完成后，不发送完成通知
- [ ] 通知列表正确展示，支持标记已读
- [ ] 未读数红点正确显示和清除

### 9.4 逾期补位

- [ ] 已指派任务逾期 24h 后，owner/admin 收到补位提醒
- [ ] 补位提醒只发送一次，不重复
- [ ] 负责人本身是 owner/admin 时，提醒发给其他管理者

### 9.5 成员退出兼容

- [ ] 成员退出后其负责的任务自动变为"所有人"
- [ ] 退出后推送恢复全员模式

### 9.6 待办页展示

- [ ] 已指派任务展示负责人头像标识
- [ ] 筛选标签可切换"全部/我的/其他人的"
- [ ] 通知铃铛展示未读红点

## 10. 后续候选（本次不做）

- 多人指派（一个任务指派给多人，轮流负责）
- 任务轮换（本周妈妈浇水，下周爸爸浇水，自动轮换）
- 指派建议（根据历史完成记录推荐负责人）
- 协作统计（每位成员的完成任务数、响应速度排行）
- 任务评论/留言（成员间就某个任务沟通）
- 完成通知走 Push（当前只走应用内通知）
- 任务交接（成员请假时一键将任务临时转给他人）
- @提醒（在完成记录中 @某人查看）

## 11. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 指派是可选的，默认"所有人" | 向后兼容 MVP 行为；小家庭（2人）可能不需要指派；不强制用户改变习惯 |
| D2 | 指派不影响可见性，只影响推送 | 所有成员仍需了解全局养护状态（如查看某植物是否被浇过）；隐藏任务会导致信息不对称 |
| D3 | 完成通知走应用内而非 Push | 完成通知是"知晓"而非"行动"，紧迫度低；走 Push 会增加通知噪音，与精准推送的目标矛盾 |
| D4 | 逾期补位只提醒 owner/admin | member 没有管理职责，不应承担"催促"角色；owner/admin 有责任确保家庭养护正常运转 |
| D5 | 单人指派而非多人 | 简化实现和心智模型；"一个任务一个负责人"职责最清晰；多人指派容易回到"互相依赖"的问题 |
| D6 | 通知保留 30 天 | 平衡存储成本和查阅需求；30 天足够回溯近期协作情况；超过 30 天的通知价值极低 |
| D7 | member 只能指派自己创建的任务 | 与角色权限矩阵一致（member 无管理权限）；避免 member 随意改变他人的任务分配 |
| D8 | 成员退出时清空 assignedTo 而非删除任务 | 任务本身属于家庭而非个人；退出只影响指派关系，不影响养护计划的连续性 |
