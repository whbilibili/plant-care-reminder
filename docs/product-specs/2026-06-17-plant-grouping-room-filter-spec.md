# 植物分组与房间筛选

**版本**：v1.0
**日期**：2026-06-17
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（后续候选·按位置/房间分组管理植物）、[植物导航与发现性 PRD](./2026-06-15-plant-navigation-discoverability-spec.md)
**文档定位**：利用已有 `location` 字段构建植物分组视图，让多植物用户能按房间/位置快速定位和批量管理。

---

## 0. 本次迭代要解决的核心问题

MVP 已在 `plants` 表提供了 `location` 字段（可选自由文本，最长 30 字符），前端植物创建/编辑时可填写，`listPlantsWithNextDue` 查询也已返回该字段。但前端**没有任何基于 location 的分组或筛选 UI**——植物列表是一个平铺的长列表，当植物数量超过 10 盆时，用户很难快速找到特定房间的植物。

> - **"我家阳台有 6 盆、客厅有 4 盆、卧室有 3 盆，每次找植物都要翻好久"** —— 多植物用户的定位效率问题。
> - **"我想看看阳台的植物今天有哪些需要浇水"** —— 按场景聚焦的养护需求，当前无法满足。

本次通过在植物列表页引入"按房间/位置分组"视图和筛选能力，让用户快速定位目标植物。

---

## 1. 产品目标

让拥有多盆植物的家庭能够按物理位置组织和查找植物。用户在植物列表页可以切换"全部"和"按房间"两种视图；按房间视图中，植物自动按 location 字段分组展示；筛选能力同时延伸到待办页——用户可以只看某个房间的待办任务。

---

## 2. 用户场景

### 场景 1：按房间浏览植物

用户打开植物列表页，点击"按房间"视图切换。列表变为分组视图：阳台（6 盆）、客厅（4 盆）、卧室（3 盆）、未分组（2 盆）。每个分组可折叠/展开。用户直接定位到"阳台"分组查看。

### 场景 2：按房间筛选待办

用户在待办页顶部看到房间筛选标签栏（"全部"/"阳台"/"客厅"/...）。选择"阳台"后，待办列表只展示阳台植物的任务。周末集中打理阳台时，不被其他房间的任务干扰。

### 场景 3：快速设置/修改位置

用户在植物列表的分组视图中发现某盆植物在"未分组"里。长按该植物卡片（或点击编辑图标），快速弹出位置选择器。选择器展示家庭已有的位置列表 + "新建位置"入口，一键归类。

## 3. 功能设计

### 3.1 植物列表页 —— 视图切换

**入口**：植物列表页顶部，现有搜索栏下方增加视图切换器，提供两种模式。

- **"全部"视图**（默认）：现有的平铺列表，排序不变。
- **"按房间"视图**：按 `location` 字段值分组展示植物。

**视图切换器样式**：使用 SegmentedControl（两个选项），与整体设计语言一致。

### 3.2 分组视图设计

**分组规则**：

- 将家庭内所有植物按 `location` 字段值分组。
- `location` 为空或 undefined 的植物归入"未分组"（排在最后）。
- 分组排序：按分组内植物数量降序（多的在前），"未分组"永远在最后。
- 组内排序：与全部视图一致（按名称字母序或创建时间）。

**分组展示**：

- 每个分组有一个分组头，显示：位置名称 + 植物数量（如"阳台（6）"）。
- 分组头可折叠/展开，默认全部展开。
- 分组头右侧有折叠/展开箭头。
- 分组下的植物卡片与全部视图复用同一个 `PlantCard` 组件。

**空状态**：

- 家庭没有任何植物：与当前全部视图的空状态一致。
- 所有植物都没有设置 location：只有一个"未分组"分组。在分组头上方展示引导文案"给植物设置位置，按房间管理更方便"。

### 3.3 待办页 —— 房间筛选

**入口**：待办页顶部，日期选择器下方增加房间筛选标签栏（horizontal scrollable chips）。

**筛选标签**：

- 第一个永远是"全部"（默认选中）。
- 后续为家庭内所有被使用的 location 值，按使用频率排序。
- 不展示"未分组"选项（没设 location 的植物任务在"全部"中可见，不单独成为筛选项）。
- 只展示当前有到期/即将到期任务的 location（避免空筛选项），这个可以由前端从已返回的任务数据中提取，无需后端新接口。

**筛选行为**：

- 选择某个 location 后，待办列表只展示该 location 植物关联的任务。
- 筛选为前端过滤（数据已全量返回），无需后端新接口。
- 切换筛选不影响完成/延后等操作。

### 3.4 位置选择器增强

**当前状态**：植物编辑页的 location 字段是一个自由文本输入框（最长 30 字符）。

**增强方案**：

- 输入框增加自动补全（autocomplete）能力：用户输入时，下拉展示家庭内已有的 location 值列表。
- 这减少同义不同名的问题（"阳台" vs "阳台上" vs "南阳台"）。
- 仍允许自由输入新值（不强制从已有列表中选）。
- 已有 location 值列表由前端从 `listPlantsWithNextDue` 返回数据中提取去重，无需新增后端接口。

## 4. 对现有逻辑的影响

### 4.1 数据模型

**无需变更**。`plants` 表已有 `location: v.optional(v.string())` 字段，`listPlantsWithNextDue` 已返回该字段。

### 4.2 新增接口

**无需新增后端接口**。所有分组和筛选逻辑均可在前端基于已有 query 返回数据实现：

- 分组视图：对 `listPlantsWithNextDue` 返回结果按 `location` 字段 `groupBy` 即可。
- 待办筛选：对任务列表按关联植物的 `location` 前端过滤。
- 已有位置列表：从植物列表数据中 `Set` 去重。

### 4.3 现有接口改造

无。

### 4.4 前端改动范围

| 文件/组件 | 改动 |
|-----------|------|
| `PlantListPage.tsx` | 增加视图切换器 + 分组视图渲染逻辑 |
| 新增 `PlantGroupView.tsx` | 分组视图组件（分组头 + 折叠/展开 + 组内列表） |
| `TodoPage.tsx` | 增加房间筛选标签栏 + 筛选逻辑 |
| 新增 `RoomFilterChips.tsx` | 房间筛选标签组件 |
| `PlantFormPage.tsx`（编辑页） | location 输入框增加 autocomplete |
| 新增 `LocationAutocomplete.tsx` | 位置自动补全输入组件 |

## 5. 交互设计要点

### 5.1 视图切换器

- 使用 SegmentedControl，两个选项"全部"/"按房间"，紧凑样式。
- 位于搜索栏下方，与搜索栏等宽。
- 切换有轻微的过渡动画（200ms fade），不要生硬跳变。
- 用户的视图选择持久化到 localStorage，下次打开保持上次选择。

### 5.2 分组头

- 高度 40px，背景用 `--color-surface-secondary`（与卡片区分）。
- 字体用 `--font-body-sm`，粗体，颜色 `--color-ink`。
- 数量标记用 `--color-muted`。
- 折叠/展开箭头用 `ChevronDown`/`ChevronUp`，尺寸 16px。

### 5.3 筛选标签栏

- 横向滚动，不换行，间距 8px。
- 选中标签：`--color-leaf` 背景 + 白色文字。
- 未选中标签：`--color-surface-secondary` 背景 + `--color-ink` 文字。
- 圆角 16px（pill 形状）。
- 全走 design token。

## 6. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 所有植物 location 为空 | 分组视图只有"未分组"一组 + 引导文案 |
| location 名称很长（接近 30 字符） | 分组头和筛选标签内文本截断加省略号 |
| 家庭有 20+ 个不同 location | 分组视图全部展示（可折叠管理），筛选标签栏横向滚动 |
| 用户在分组视图下搜索 | 搜索结果仍按分组展示（搜索在组内过滤） |
| 新建植物时设了新 location | 返回列表后分组视图自动出现新分组（Convex 响应式） |

## 9. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 纯前端分组和筛选，不新增后端接口 | 家庭植物数量在百级以下，前端分组 groupBy 完全够用；避免增加后端复杂度 |
| D2 | 沿用自由文本 location，不引入独立的"位置"实体表 | MVP 阶段保持简单，通过 autocomplete 引导一致性，不需要位置的 CRUD 管理 |
| D3 | 默认"全部"视图 | 新用户和少植物用户不需要分组，避免强制复杂化；分组是多植物用户的效率工具 |
| D4 | 待办页筛选只展示有任务的 location | 空筛选项无意义且占空间 |
| D5 | 视图选择持久化 | 用户习惯一旦形成就不需要每次重新选择 |
