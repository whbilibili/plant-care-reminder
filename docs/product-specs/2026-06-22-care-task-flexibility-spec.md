# 养护任务灵活性

**版本**：v1.0
**日期**：2026-06-22
**关联文档**：承接 [MVP 需求文档](./2026-06-05-mvp-requirements.md)（§7.3 任务管理）、[迭代储备大纲](./2026-06-17-iteration-backlog-outline.md)（P1-2）
**文档定位**：将养护任务的排期能力从"固定间隔天数"扩展为三种模式（固定间隔 / 每周几 / 季节性调整），同时引入固定类型任务唯一性约束，让任务配置既灵活又不混乱。

---

## 0. 本次迭代要解决的核心问题

MVP 的养护任务只有"每 N 天提醒一次"一种排期模式，且同一植物可以创建多个相同类型的任务。这导致三个体验缺口：

- **"我想每周一三五浇水，但只能设 2 天一次，周末就会多提醒"** —— 固定间隔无法表达"每周特定几天"的需求，用户只能用近似值凑合。
- **"夏天 3 天浇一次，冬天 7 天浇一次，每次换季都要手动改"** —— 没有季节意识，用户需要记住每年换季时去调整间隔，经常忘记。
- **"不小心给绿萝创建了两个浇水任务，待办页出现两条一样的"** —— 缺乏唯一性约束，固定类型任务重复创建造成困惑和重复提醒。

本次通过三项改造——排期模式扩展、季节性调整、固定类型唯一性约束——让任务配置既灵活又清晰。

---

## 1. 产品目标

让用户能够用最自然的方式描述养护节奏：可以是"每 3 天一次"，也可以是"每周一三五"，还可以是"夏天 3 天、冬天 7 天"。同时确保每盆植物的任务列表清晰无重复，一种养护动作对应一个任务配置。

---

## 2. 用户场景

### 场景 1：每周固定几天浇水

用户新买了一盆蕨类植物，养护指南说"每周一三五浇水"。用户创建浇水任务时，选择"每周几"模式，勾选周一、周三、周五。系统在这三天提醒浇水，周末不打扰。

### 场景 2：季节性调整施肥频率

用户的发财树在生长季（春夏）需要每 14 天施肥一次，休眠季（秋冬）只需每 30 天一次。用户创建施肥任务时选择"季节性调整"模式，分别设置春夏间隔 14 天、秋冬间隔 30 天。系统根据当前日期自动切换间隔，用户无需手动调整。

### 场景 3：防止重复创建

用户尝试给绿萝再创建一个"浇水"任务（已有一个），系统提示"这盆植物已有浇水任务，可以去编辑现有任务的排期设置"，并提供跳转到编辑页的入口。避免了重复任务造成的混乱。

### 场景 4：自定义任务不受限制

用户给琴叶榕创建了"擦叶片"和"转盆朝向"两个自定义任务，系统正常允许——自定义任务靠名称区分语义，不受固定类型唯一性约束。

## 3. 功能设计

### 3.1 排期模式扩展

**三种排期模式**：

| 模式 | 标识 | 适用场景 | 到期计算逻辑 |
|------|------|---------|------------|
| 固定间隔 | `interval` | "每 N 天一次"（现有行为） | `nextDueAt = lastCompletedAt + intervalDays × 86400000` |
| 每周几 | `weekly` | "每周一三五" | 完成后计算下一个匹配的星期几（UTC） |
| 季节性调整 | `seasonal` | "夏天 3 天、冬天 7 天" | 根据当前日期所在季节选择对应间隔，再按固定间隔逻辑计算 |

**模式选择器 UI**：

- 任务创建/编辑页面，在"任务类型"选择后、"间隔天数"输入前，新增"排期方式"分段选择器（SegmentedControl）。
- 三个选项："固定间隔"（默认）、"每周几"、"按季节"。
- 切换模式时，下方表单区域动态切换为对应的配置项。

### 3.2 固定间隔模式（现有行为保留）

- 输入项：间隔天数（1~365 正整数）。
- 计算逻辑：与 MVP 完全一致，无变更。
- 这是默认模式，老数据自动归入此模式。

### 3.3 每周几模式

**输入项**：

- 星期选择器：7 个可点击的圆形按钮（一~日），至少选择 1 个，最多全选。
- 选中态用 `--color-leaf` 填充，未选态用 `--color-surface` 描边。

**到期计算逻辑**：

- 完成任务后，从当前时间开始，找到下一个匹配的星期几作为 `nextDueAt`。
- 如果今天是选中的星期几且刚完成，下次到期为下一个匹配日（不是今天）。
- 多个星期几选中时，取最近的下一个匹配日。
- 示例：选中周一三五，周一完成 → 下次到期周三；周五完成 → 下次到期下周一。

**逾期处理**：

- 逾期后完成，仍按"从完成时间找下一个匹配日"计算，不累积逾期。

### 3.4 季节性调整模式

**输入项**：

- 春夏间隔天数（3月~8月）：正整数 1~365。
- 秋冬间隔天数（9月~2月）：正整数 1~365。
- 两个输入框并排，标签分别为"春夏（3~8月）"和"秋冬（9~2月）"。

**到期计算逻辑**：

- 完成任务时，根据完成日期所在月份判断当前季节。
- 3月~8月使用春夏间隔，9月~2月使用秋冬间隔。
- `nextDueAt = completedAt + 当前季节间隔 × 86400000`。
- 如果 nextDueAt 跨越了季节边界（如 8月28日 + 7天 = 9月4日），仍使用计算时的季节间隔（不在中途切换）。

**季节切换时机**：

- 系统不主动调整已计算好的 `nextDueAt`。季节切换的效果体现在"下一次完成后的重新计算"中。
- 示例：8月底完成，用春夏间隔算出 nextDueAt 在 9月初。9月初到期完成后，此时已进入秋冬，用秋冬间隔计算下一次。

### 3.5 固定类型任务唯一性约束

**规则**：

- 同一植物下，每种固定类型（watering / fertilizing / misting / repotting / pruning）最多只能存在**一个启用的任务**。
- 自定义类型（custom）不受此限制，可创建多个（靠 customLabel 区分）。
- 已停用（enabled=false）的任务不计入唯一性检查——用户可以停用旧任务后创建新的同类型任务。

**创建时校验**：

- 后端 `createPlantTask` mutation 在插入前检查：该植物是否已有同类型且 enabled=true 的任务。
- 若已存在，返回错误："这盆植物已有「浇水」任务，请编辑现有任务或先停用它。"
- 前端在任务类型选择器中，对已存在的固定类型标注"已有"标签（灰色），点击后弹出提示而非进入配置。

**编辑不受影响**：

- 编辑现有任务时不触发唯一性检查（因为是修改同一个任务）。

## 4. 数据模型变更

### 4.1 plantTasks 表扩展

新增字段：

```
// 排期模式：interval（默认/现有）、weekly、seasonal
scheduleMode: v.optional(v.union(
  v.literal("interval"),
  v.literal("weekly"),
  v.literal("seasonal")
))

// 每周几模式的配置：0=周日, 1=周一, ..., 6=周六
weeklyDays: v.optional(v.array(v.number()))

// 季节性模式的配置
seasonalIntervals: v.optional(v.object({
  springSummer: v.number(),  // 春夏间隔天数（3~8月）
  autumnWinter: v.number(),  // 秋冬间隔天数（9~2月）
}))
```

**向后兼容策略**：

- `scheduleMode` 为 `v.optional`，老数据无此字段时按 `"interval"` 处理。
- `intervalDays` 字段保留，在 `interval` 和 `seasonal` 模式下仍有意义（seasonal 模式下存储当前生效的间隔值，供待办页展示"每 N 天"）。
- `weeklyDays` 仅在 `scheduleMode === "weekly"` 时有值。
- `seasonalIntervals` 仅在 `scheduleMode === "seasonal"` 时有值。

### 4.2 索引评估

- 无需新增索引。唯一性检查通过 `by_plantId` 索引查询该植物的所有任务后在应用层过滤。
- 现有 `by_familyId_and_nextDueAt` 索引仍支持待办页查询（nextDueAt 计算逻辑变了，但存储格式不变）。

## 5. 接口变更

### 5.1 createPlantTask 改造

新增参数：

```
scheduleMode: v.optional(v.union(v.literal("interval"), v.literal("weekly"), v.literal("seasonal")))
weeklyDays: v.optional(v.array(v.number()))
seasonalIntervals: v.optional(v.object({ springSummer: v.number(), autumnWinter: v.number() }))
```

新增校验：

- 唯一性检查：非 custom 类型 + enabled=true + 同 plantId → 拒绝创建。
- `weekly` 模式：`weeklyDays` 必填，数组长度 1~7，元素为 0~6 的整数，无重复。
- `seasonal` 模式：`seasonalIntervals` 必填，两个值均为 1~365 正整数。
- `interval` 模式：行为与现有一致。

### 5.2 updatePlantTask 改造

支持修改排期模式及对应配置。切换模式时重新计算 `nextDueAt`。

### 5.3 completePlantTask 改造

完成任务时的 `nextDueAt` 计算逻辑根据 `scheduleMode` 分支：

- `interval`：现有逻辑不变。
- `weekly`：调用新的 `computeNextWeeklyDueAt(completedAt, weeklyDays)` 函数。
- `seasonal`：调用新的 `computeNextSeasonalDueAt(completedAt, seasonalIntervals)` 函数。

### 5.4 新增共享计算函数

在 `app/src/features/tasks/scheduling.ts` 中新增：

```typescript
computeNextWeeklyDueAt(completedAt: number, weeklyDays: number[]): number
computeNextSeasonalDueAt(completedAt: number, seasonalIntervals: { springSummer: number; autumnWinter: number }): number
getEffectiveIntervalDays(task: PlantTask): number  // 根据模式返回当前生效的间隔天数（供展示用）
```

## 6. 前端交互设计

### 6.1 排期模式选择器

- 使用 SegmentedControl 组件（三段），放在任务类型选择之后。
- 切换模式时，下方配置区域平滑过渡（fade + height 动画）。
- 默认选中"固定间隔"，与现有体验一致。

### 6.2 每周几选择器

- 7 个圆形按钮横排，标注"一二三四五六日"。
- 选中态：`--color-leaf` 背景 + `--color-paper` 文字。
- 未选态：`--color-surface` 背景 + `--color-ink` 文字。
- 至少选 1 个的校验：未选时按钮区域下方显示红色提示"请至少选择一天"。

### 6.3 季节性输入

- 两个数字输入框并排，各占 50% 宽度。
- 标签："春夏（3~8月）"、"秋冬（9~2月）"。
- 输入框内 placeholder："天"。
- 校验与固定间隔一致（1~365 正整数）。

### 6.4 已有类型标注

- 任务类型选择列表中，已存在的固定类型选项右侧显示"已有"灰色标签。
- 点击已有类型时，弹出 toast："这盆植物已有「浇水」任务，可前往编辑"，附"去编辑"按钮跳转。
- 自定义类型永远可选。

### 6.5 待办页展示适配

- 固定间隔：展示"每 N 天"（现有）。
- 每周几：展示"每周一三五"格式。
- 季节性：展示"春夏每 N 天 / 秋冬每 M 天"，当前生效的季节加粗。

## 7. 边界与异常处理

| 场景 | 处理 |
|------|------|
| 老数据无 scheduleMode 字段 | 按 `"interval"` 模式处理，`intervalDays` 为唯一排期依据 |
| weekly 模式选了今天，今天完成 | 下次到期为下一个匹配日（不是今天） |
| weekly 模式只选了一天（如周一） | 等价于"每 7 天"，但语义更清晰 |
| seasonal 跨季节边界 | 不中途切换，下次完成时才用新季节间隔 |
| 用户从 weekly 切回 interval | 清空 weeklyDays，用 intervalDays 重算 nextDueAt |
| 停用的同类型任务存在时创建新任务 | 允许（唯一性只检查 enabled=true） |
| 编辑任务切换模式 | 立即用新模式重算 nextDueAt 并保存 |

## 8. 验收标准

### 8.1 排期模式

- [ ] 任务创建/编辑页展示三种排期模式选择器
- [ ] 固定间隔模式行为与现有完全一致（回归无破坏）
- [ ] 每周几模式：可选择 1~7 天，完成后正确计算下一个匹配日
- [ ] 季节性模式：可分别设置春夏/秋冬间隔，完成后根据当前季节正确计算
- [ ] 切换模式时立即重算 nextDueAt

### 8.2 唯一性约束

- [ ] 同一植物不能创建两个相同固定类型的启用任务
- [ ] 创建时前端标注"已有"，后端拒绝并返回友好错误信息
- [ ] 自定义类型不受限制，可创建多个
- [ ] 停用的任务不影响唯一性检查

### 8.3 向后兼容

- [ ] 老数据（无 scheduleMode）正常按固定间隔工作
- [ ] 现有任务的 nextDueAt 计算不受影响
- [ ] 待办页正确展示三种模式的排期描述

### 8.4 计算正确性

- [ ] weekly 模式单测覆盖：单日选择、多日选择、跨周、逾期后完成
- [ ] seasonal 模式单测覆盖：春夏内完成、秋冬内完成、跨季节边界
- [ ] 推迟逻辑（postpone）在三种模式下均正常工作

## 9. 后续候选（本次不做）

- 更精细的季节划分（春/夏/秋/冬四季独立配置）
- 基于天气 API 的动态间隔建议
- "每月第 N 天"排期模式
- 任务间依赖（"施肥后 3 天再浇水"）
- 批量修改多盆植物的同类型任务间隔

## 10. 设计决策记录

| # | 决策 | 理由 |
|---|------|------|
| D1 | 三种模式而非更多 | 覆盖 90% 用户需求（固定节奏/周规律/季节变化），避免过度复杂；更精细的需求留给后续 |
| D2 | scheduleMode 为 optional | 向后兼容：老数据无此字段时默认 interval，无需数据迁移 |
| D3 | 季节按月份划分（3~8/9~2）而非天文季节 | 简单直观，用户不需要知道春分秋分的精确日期；与中国大部分地区的养护习惯匹配 |
| D4 | 固定类型唯一性约束 | 防止用户误创建重复任务导致待办混乱；真正的灵活性应通过排期模式解决，而非多任务堆叠 |
| D5 | 唯一性只检查 enabled=true | 允许"停用旧的、创建新的"工作流，不过度限制用户自由 |
| D6 | seasonal 不主动调整已计算的 nextDueAt | 避免"半夜系统突然改了我的提醒时间"的困惑；季节切换效果在下次完成时自然生效 |
| D7 | intervalDays 字段保留 | seasonal 模式下存储当前生效间隔供展示；weekly 模式下该字段无意义但保留避免 schema 破坏性变更 |
