## 0. 设计锚点（先统一认知）

用 JTBD（Jobs To Be Done）视角，用户打开待办页时雇佣它完成的真实任务**不是**「看一个任务列表」，而是：

> **"我现在 / 今天，该为家里的植物做点什么，做完后心里踏实（不焦虑漏养）。"**

由此推导出待办页的两个核心价值：

1. **降低决策成本** —— 用户一眼知道「现在该干嘛」，不需要自己做二次筛选。
2. **消除焦虑、给予正反馈** —— 做完有「照顾好了」的踏实感，而不是被无穷任务追着跑。

下面所有设计决策都服务于这两点。待办页是**高频执行页**（一天可能打开多次），其设计原则是**信息密度优先、即时可操作、情感温和**，而非「展示尽可能多的信息」。

## 1. 列表加载逻辑

### 1.1 加载范围：按「状态区段」组织，而非「时间窗」

不采用「未来 N 天全平铺」的口径，因为那是系统视角（间隔天数）而非用户心智。用户脑中只有三种状态：

| 区段 | 口径 | 是否默认展示 | 优先级 |
|------|------|-------------|--------|
| **逾期** | `nextDueAt < 今天 00:00`，且今天未完成 | 是（主区，最靠前） | 最高 |
| **今天到期** | `nextDueAt` 落在今天，且今天未完成 | 是（主区） | 高 |
| **即将到期** | `nextDueAt` 落在未来 1–3 天 | 是（次要分区，可弱化/折叠） | 低 |

> 「即将到期」存在的意义是给用户**预期感（心里有数）**，不是让其今天动手，因此视觉权重必须明显低于前两个区段。

### 1.2 排序规则

- 一级排序：`逾期` → `今天到期` → `即将到期`
- `逾期` 区段内：**逾期天数越多越靠前**（欠得越久越优先还）
- `今天到期` 区段内：可按任务类型聚合（见 §2.2）
- `即将到期` 区段内：按 `nextDueAt` 升序

### 1.3 关键规则：完成后是否还展示？

**问题场景**：浇水间隔 3 天，今天点完成 → `下次提醒 = 今天 + 3 天 = 第 3 天`。若窗口含第 3 天，任务会重新出现在列表，造成「我不是刚浇过吗？是不是没保存？」的困惑，破坏完成正反馈。

**设计原则**：核心不是「抑制任务重复出现」，而是 **「今天完成过的任务，今天不再催」**。

> 注意边界：若间隔 = 1 天，今天完成 → 明天到期，这是**正常且期望**的，不应抑制。所以规则锚定在「今天内不再催」，而非「永久去重」。

**最终决策（2026-06-10 已拍板）：采用方案 A。** ✅

- 待办**主区**只显示「逾期 + 今天到期且未完成」。
- 点击完成后，任务**立即从主区移除**，给予即时正反馈。
- 该任务的下次提醒若落在未来 1–3 天，则在「即将到期」分区以**灰态**出现，并标注「✅ 已完成 · 下次 X 月 X 日」。

> 方案 A 兼顾即时反馈与未来可见性，实现上仅需按 `nextDueAt` 分桶 + 标记「今日是否已完成」。

**备选方案（方案 B，已否决）**：今天完成过的任务，今天内不在任何分区出现，次日才按新 `nextDueAt` 重新参与计算。

## 2. 列表展示优化（搜索 / 分组）

### 2.1 搜索：待办页 MVP **不做**搜索

| 页面 | 场景性质 | 是否需要搜索 |
|------|---------|-------------|
| 植物列表页 | 管理 / 查阅（列表会越来越长） | ✅ 保留（已有） |
| 待办页 | 执行 / 打勾（天然短列表） | ❌ 不做 |

> 原则：**管理态需要搜索，执行态需要分组和排序**。家庭场景每日到期任务通常为个位数，待办页放搜索属过度设计。

### 2.2 分组：一级用紧急度（必做），紧急度内可按任务类型聚合（加分项）

| 分组维度 | 结论 | 说明 |
|---------|------|------|
| 按紧急时间（逾期/今天/即将） | ✅ 默认且唯一必要 | 直接服务核心 Job，与 §1 加载逻辑同源 |
| 按任务类型（浇水/施肥…） | 🔶 次要 / 加分 | 真实价值是**批量动作**：拿喷壶一次浇完 → 一键完成多个浇水任务 |
| 按植物归属 | ❌ 不做 | 植物维度属「植物列表页」，在待办页分组会割裂「今天要干的事」整体视图 |

**任务类型聚合 + 批量完成**（强烈建议）：把「逐个打勾 5 次」变成「浇完水，一键完成 5 个浇水任务」，直接对应家庭真实家务动线，体验提升显著。

## 3. 页面美化与情感设计

### 3.1 Banner：不放传统营销 Banner

高频执行页放静态 Banner，第一次有惊喜、第三次变障碍物（占地方、需滚动才能看到任务）。高频工具页应**信息密度优先、即时可操作**。

### 3.2 替代方案：顶部「状态化情感反馈区」

用一个随状态变化的小卡片 / 问候区代替 Banner，这才是高级感来源：

| 状态 | 文案与设计方向 |
|------|---------------|
| 全部完成 | 「🌿 今天的植物都照顾好啦」+ 轻松文案 + 满足感插画（**正反馈闭环**，对留存关键） |
| 有逾期 | 温和提示「有 2 株植物在等你」，**不用红色警报式施压**（养植是放松的事，不该有 KPI 感） |
| 有今日任务 | 「今天要照顾 3 株植物」 |

### 3.3 具体 UI 提升方向（结合已有 botanical 主题）

现有 design token 已是植物风（`--color-leaf` 绿 + `--color-paper` 米白），方向正确。可提升：

- **任务卡片左侧用任务类型色条/图标**（浇水蓝、施肥棕、喷雾青…），扫一眼知道要干嘛，降低决策成本。
- **完成动作微交互**：打勾时水滴/叶子小动画 + iOS Haptic 触觉反馈，把「打勾」变成小确幸（高频工具留存的隐藏武器）。
- **逾期用「天数 + 柔和提示色」**，避免大红感叹号。
- **空状态精心设计**：无任务时不白屏，放温暖插画 + 引导（「去添加第一株植物」或「今天无需养护，去看看你的植物」）。

> 📐 **UI 设计规格**：本节为产品侧的设计意图与方向；具体的布局结构、配色应用、组件状态、交互与动效原则，已由 UI 设计师角色细化到独立文档 [2026-06-10-todo-page-ui-spec.md](./2026-06-10-todo-page-ui-spec.md)，两份配套使用。

## 4. MVP 增项建议（来自讨论补充的 4 点）

### 4.1 「推迟」已确认纳入本迭代（原文档列在第 11 节后续版本）✅

**痛点**：现实中常「今天没空，明天再浇」。当前只有「完成」，用户为让任务消失可能**虚假打勾**，污染养护历史真实性，且逾期列表会因「没空做又不想假打勾」而堆积，制造焦虑。

**决策（2026-06-10 已拍板）**：把「推迟」从后续版本**提前到本迭代**，因为它与待办页核心体验强相关。

- 行为：`nextDueAt` 后移指定天数，**不修改 `lastCompletedAt`**（区别于完成）。
- 不写入 `TaskCompletionLog`（推迟不是完成）。

> 推迟的具体逻辑（推迟基准时间如何取、推迟上限、是否记录推迟次数、推迟后是否仍算逾期等）待 §8 专项细化。

### 4.2 显示「谁完成的」

家庭协作场景，「老婆刚浇过了」可避免重复劳动并带来轻协作激励。待办页 / 详情页显示「✅ 张三 · 今天完成」。成本低、价值不小（数据已有 `TaskCompletionLog.完成人`）。

### 4.3 完成后「撤销」

移动端手误打勾常见。完成后给 3–5 秒「撤销」浮条，优于让用户翻历史改时间。

- 实现：撤销 = 回滚 `lastCompletedAt` 与 `nextDueAt`，并删除刚写入的 `TaskCompletionLog`。

### 4.4 首次进入引导（空状态 onboarding）

新家庭空间为空，待办页第一眼是空状态，此处 onboarding 决定留存。应直接引导「添加第一株植物」，而非让用户面对空列表自行琢磨。

## 5. 完成 / 推迟 / 撤销 状态机（待细化）

下表为后续讨论的起点，字段以现有数据对象（`PlantTask` / `TaskCompletionLog`）为准：

| 动作 | `lastCompletedAt` | `nextDueAt` | `TaskCompletionLog` | 待办主区可见性 |
|------|-------------------|-------------|---------------------|----------------|
| 完成 | ← 今天 | ← 今天 + 间隔 | 新增一条（含完成人、时间） | 今天内移出主区 |
| 推迟 N 天 | 不变 | ← 当前 nextDueAt + N | 不写入 | 推后，按新时间重新参与分桶 |
| 撤销完成 | 回滚到上一次 | 回滚到上一次 | 删除最近一条 | 恢复到主区 |

> 撤销的回滚需要保留「上一次」的值；实现细节（是否依赖 `TaskCompletionLog` 倒推、还是临时缓存上一状态）留待技术方案阶段确定。

## 8. 推迟逻辑专项细化

### 8.1 推迟基准时间（已拍板）✅

**决策：采用 `今天 + N`，并用透明文案消除歧义。**

- 推迟服务的 Job 是「让这个任务现在别烦我，明天/几天后再说」，所以新下次提醒 = **今天 + N**。
- 不采用 `nextDueAt + N`：对逾期任务而言那样推了可能仍是逾期（推了等于没推），与用户预期不符。
- **推迟会“洗掉”原有逾期天数的问题，不在算法层补偿历史亏欠，而是在交互层用文案明确告知用户**，让用户知情且自主决策。

### 8.2 推迟告知文案（已拍板）✅

点击「推迟」时，根据任务是否逾期展示不同文案，让用户在确认前看到“这次推迟意味着什么”：

| 场景 | 文案示例 |
|------|---------|
| 任务已逾期 | 「该任务已逾期 **3 天**，推迟后将从**今天**起顺延，下次提醒 **6 月 11 日**」 |
| 任务今日到期（未逾期） | 「推迟后下次提醒将变为 **6 月 11 日**」 |

> 文案原则：语气温和、不说教（不用「你又拖延了」这类施压表达），只客观告知事实与结果。逾期天数与下次日期均为动态计算。

### 8.3 推迟粒度（已拍板）✅

**决策：本迭代只做「推迟 1 天」单按钮，可选天数排后续版本。**

- 家庭养护场景下「今天没空明天补」是绝对主流，1 天可覆盖绝大多数需求。
- 单按钮交互最轻（待办卡片上一个「推迟」按钮点一下即可），符合 MVP「常用操作 3 步内」原则。
- 可选天数会引出二级选择器，增加交互层级，放验证后的下一版（可考虑「推迟 1 天」主按钮 + 长按/次级入口选「推迟到指定日期」）。

### 8.4 推迟上限与温和提示（已拍板）✅

**决策：不做硬性次数限制 + 记录连续推迟次数 + 达阈值给温和提示。**

设计理由：

- 这是家庭自用的养护工具，不是 KPI 考核系统。硬性拦截（弹「不能再推」）会逼用户**假打勾**，反而回到最初要解决的问题。
- 但一株植物被连续推迟多次，是个有价值的信号（要么顾不上、要么养护间隔设的不合理），不该被悄悄忽略。

具体规则：

- **不拦截**，可无限推迟。
- 记录「连续推迟次数」，**完成一次后清零**。
- 连续推迟达阈值（初步定 **3 次**，可调）时，在卡片上以温和方式提示，把“管教”转为“贴心建议”，并顺势引导用户去调整养护周期。
  - 文案示例：「这株植物最近总被推迟，要不要调整一下养护周期？」

> 实现拆分建议：本迭代至少先落地「记录连续推迟次数」（数据层）；「达阈值提示」（交互层）若工作量偏重可略后，但两者同期交付体验更完整。

### 8.5 单次推迟是否标记（已拍板）✅

**决策：单次推迟不标记，只在连续推迟达阈值时露出温和建议。**

- 推迟 1 天后任务次日会作为正常「今日任务」重现，若标「已推迟」反而让用户觉得“这是个有问题的任务”，与「养护是放松的事」基调不符。
- 信息露出原则：**偶尔推一次是常态，悄无声息；总在推才值得提醒**。因此唯一需要露出的是 §8.4 那个“连续推迟达阈值”的信号。

### 8.6 推迟记录方式（已拍板）✅

**决策：轻方案——只在 `PlantTask` 加一个 `consecutivePostponeCount` 字段，不建独立推迟记录表。**

- 该字段记录「连续推迟次数」，**推迟时 +1，完成一次后清零**，足以支撑 §8.4 的达阈值提示。
- 推迟**不写 `TaskCompletionLog`**（推迟不是完成）；养护历史回溯以 `TaskCompletionLog` 为主线，推迟只是中间过程，无需留完整流水。
- 独立推迟流水表在家庭 2–6 人场景价值不大，属过度设计；未来若真有深度分析需求，再升级为独立表也不迟。

### 8.7 多人协作露出（已拍板）✅

**决策：完成露出、推迟隐去——推迟结果实时同步，但不显示「谁推迟的」。**

- **推迟结果必须实时同步**：一人推迟后，`nextDueAt` 后移，其他家庭成员看到的待办列表随之更新（任务从主区移出 / 重新分桶）。这是 MVP 既有的实时一致性要求，Convex 原生支持，无需额外设计。
- **不显示推迟人**（与 §4.2「显示完成人」相反处理）：
  - 「谁推迟的」带隐含负面色彩（“他又没做”），容易在家庭间制造摩擦，与「养护是放松的事」基调冲突。
  - 对协作无决策价值——别人是否做、由谁做，看「下次提醒时间」即可，不需要知道是谁推的。
  - 成本不划算：显示推迟人需额外记录 `lastPostponedBy`，而它只会带来负向情绪。
- 对照原则：**正向动作（完成）值得署名鼓励，中性/负向动作（推迟）只同步结果、不归因到人。**

> §8 推迟逻辑至此全部拍板。

## 9. 撤销与其他问题

### 9.1 撤销实现路径（已拍板）✅

**决策：撤销 = 短时浮条 + 缓存上一状态还原，不做长期可撤销。**

- 撤销定位为**手误补救**，只覆盖刚打完勾的那几秒（§4.3 定的 3–5 秒撤销浮条）。
- 实现：完成时把动作前的 `lastCompletedAt` / `nextDueAt` 临时缓存一份（会话内），撤销时直接还原并删除刚写入的 `TaskCompletionLog`。
- **不做「翻历史撤销」**：浮条消失（刷新/切人/超时）后不再提供撤销，如需修正走“编辑养护记录”路径（另一功能）。
- 该定位下无需依赖日志倒推，实现最轻；未来若真要「历史记录可撤销」再升级为日志倒推方案。

### 9.2 「即将到期」窗口（已拍板）✅

**决策：即将到期窗口固定 3 天，本迭代不开放配置，作为可调参数留在代码里。**

- 该窗口的 Job 是给用户**预期感（心里有数）**，不是催他今天动手；3 天能覆盖最高频的浇水（间隔多为 3–7 天）预览，又不会把一周后的任务堆进来造成压迫。
- 窗口过长会让即将到期区变长，稀释主区「今天该干嘛」的焦点，违背「执行页信息密度优先」原则。
- **不开放配置**：给合理默认值比给选项更重要（少即是多）；可配置会引出「在哪配 / 是否等同于通知提前量」等连锁复杂度。
- 实现上把 3 天做成一个**常量 / 配置项放在代码里**，后续改默认值或开放用户配置都无需重构。

### 9.3 批量完成交互形态（已拍板）✅

**决策：采用分区级一键全部完成，不做多选批量。**

- 该功能来自 §2.2，价值锚点是「拿起喷壶一次浇完一类」的真实家务动线；分区级一键正好命中。它在 §6 为**加分项 / 可排后**，交互越轻越易落地。
- **不做多选批量**：多选态（进入选择→逐个勾→确认）交互重，偏管理态，与待办页「即时可操作、打一下就完事」的轻量基调相背；「挑着完成」需求频率低，单个打勾已可满足。
- 具体形态：类型分组标题栏右侧放一个「✅ 全部完成」轻按钮，点击后给**带撤销的批量完成反馈**（复用 §9.1 撤销浮条，文案如「已完成 5 个浇水任务」）。
- 某株今天实际未做的例外情况，走单个操作即可，无需为低频需求引入多选复杂度。
