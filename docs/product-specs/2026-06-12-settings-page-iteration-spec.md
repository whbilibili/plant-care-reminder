# 设置页产品设计细化（迭代 v0.2）

**版本**：v0.2（迭代设计）
**日期**：2026-06-12
**关联文档**：基于 [2026-06-05-mvp-requirements.md](./2026-06-05-mvp-requirements.md) 第 5.1 节「账号与家庭空间」、第 5.4 节「到期提醒」，以及 [2026-06-09-ui-ux-visual-spec.md](../design-docs/2026-06-09-ui-ux-visual-spec.md) 第 6.9 节「设置页」的细化与扩展
**文档定位**：本文档不重复 MVP 全量需求，只针对**设置页（`/settings`）**这一个低频辅助页，把它从「家庭信息展示页」升级为真正的「设置中心」。聚焦信息架构重组、缺失能力补齐、权限语义落地、邀请码可控性、文案与视觉层级，作为下一轮迭代的设计依据。

---

## 0. 设计锚点（先统一认知）

用 JTBD 视角，用户来到设置页时雇佣它完成的真实任务**不是**「看一眼家里几个人」，而是：

> **"我想管好这个家庭空间——把家人拉进来、改我自己的信息、确保提醒能正常推到我手机上，必要时还能清理掉不该在的人或退出。"**

由此推导出设置页的三个核心价值：

1. **协作入口** —— 分享邀请码、管理成员，把"家庭共享"这件事真正落地。
2. **自我掌控** —— 改自己的称呼、管自己的通知、需要时干净退出。
3. **可信兜底** —— 邀请码可重置、推送收不到有排障路径、破坏性操作有确认。

设置页是**低频辅助页**（不像待办页每天多次打开），其设计原则是**分区清晰、能力闭环、操作安全、不喧宾夺主**。

## 1. 页面角色定位

### 1.1 与其他主页面的关系

| 维度 | 待办页 `/todo` | 植物列表页 `/plants` | 设置页 `/settings` |
|------|---------------|---------------------|-------------------|
| 核心任务 | 今天该做什么 | 家里有什么 | 怎么管这个家庭和我自己 |
| 使用频率 | 高频（每天多次） | 中频（每周数次） | 低频（按需，偶尔） |
| 信息粒度 | 任务级 | 植物级 | 账号 / 家庭 / 设备级 |
| 主动作 | 完成任务 | 进入详情 / 添加 | 复制邀请码 / 改设置 |
| 设计倾向 | 执行效率 | 识别效率 | 分区清晰、操作安全 |

### 1.2 用户来到此页面的典型场景

1. **拉家人进来** → 复制邀请码发给家人（最高频动作）
2. **开启提醒** → 第一次用，或换了设备，要把推送打开
3. **改我自己的称呼** → onboarding 时随便填了，现在想改正式点
4. **清理成员** → 家人换号了 / 误加入了别人，管理员要移除
5. **排障** → "为什么我收不到提醒？"——需要一个能自查的入口
6. **退出登录** → 借用别人手机登录过，用完要退出

## 2. 核心问题诊断（当前实现）

当前设置页是规范且干净的，但更像一个**只读的家庭信息展示页**，而非真正的"设置"。逐条诊断：

| # | 问题 | 影响 | 严重度 |
|---|------|------|--------|
| 1 | **设置页里没有"设置"**：用户改不了自己的称呼，onboarding 设完就再无入口回去改 | 自我掌控缺失，最基础的预期落空 | P0 |
| 2 | **成员管理只读**：区分了 admin/member 角色、展示了"管理员"徽章，但管理员不能移除成员、不能转让，角色字段形同虚设 | 误加入 / 换号场景无解，权限语义空转 | P0 |
| 3 | **邀请码不可控**：一经生成永久不变、永久有效，任何拿到码的人可永久加入，无撤销机制 | 安全兜底缺失 | P1 |
| 4 | **信息层级与命名冗余**：成员卡 eyebrow 与 h2 都写"家庭成员"；家庭摘要用 h1(24px)把视觉重心压在低价值信息上，高频的复制邀请码反而权重不足 | 视觉噪声、重心错位 | P1 |
| 5 | **缺应用层信息与自助入口**：无版本号、无关于、无帮助/反馈；iOS PWA 推送本就脆弱，收不到时用户只能干瞪眼 | 排障无门，信任流失 | P2 |
| 6 | **退出登录无二次确认**：PWA 里重新登录成本不低，当前点一下直接退 | 破坏性操作误触 | P2 |

## 3. 信息架构重组

### 3.1 决策：从扁平五卡重组为三个语义分组 ✅

当前是无分组的五块卡片堆叠（家庭摘要 / 邀请码 / 通知 / 成员 / 退出）。重组为符合"设置"心智的三组，每组内是卡片或设置项：

```
设置（页面标题 h1）
│
├── 个人
│     ├── 我的称呼            ← 新增：可编辑
│     ├── 我的账号（邮箱，只读）  ← 新增：信息展示
│     └── 退出登录            ← 从底部独立区迁入本组末尾
│
├── 家庭
│     ├── 家庭摘要（名称 + 成员数）
│     ├── 邀请码分享 + 重置邀请码  ← 重置为新增
│     └── 家庭成员管理（含移除）   ← 移除为新增
│
└── 通知与应用
      ├── 设备通知开关（沿用现有 NotificationPromptCard）
      ├── 收不到提醒？（排障指引）  ← 新增
      └── 版本号 / 关于 / 反馈      ← 新增
```

### 3.2 分组理由

- **个人 / 家庭 / 通知与应用** 是用户最朴素的三类心智边界，找东西的预期路径清晰。
- 退出登录从"页面底部独立区"迁到"个人"组末尾，仍满足视觉规范"不隐藏、不埋深、风格与普通项区分"的要求，同时归属更合理。
- 为后续迭代（归档植物入口、深色模式、数据导出等）预留了挂载点，无需再次重排。

### 3.3 分组的视觉表达（非强制层级标题）

分组**不强制**用大标题割裂页面，可用「分组小标题（次级、克制）+ 卡片留白」表达，避免页面变得像后台管理系统。具体视觉在配套 UI Spec 细化。

## 4. 缺失能力补齐

### 4.1 P0｜我的称呼（可编辑）✅

**产品语义**：用户在 onboarding 设置的称呼，是家庭成员之间相互识别的唯一标签（成员列表、未来的"谁完成了任务"都依赖它），必须可回头修改。

| 项 | 规则 |
|----|------|
| 入口 | 「个人」组首项，点击进入编辑（行内编辑或轻量页/底部 sheet，UI Spec 定） |
| 复用 | 复用 onboarding 的 `ProfileBootstrapForm` 校验逻辑，不重造规则 |
| 约束 | 非空、长度上限沿用 onboarding 既有约束 |
| 同步 | 保存后，家庭成员列表中"我"的显示名实时更新 |
| 反馈 | 保存成功有明确反馈，失败有错误态文案 |

### 4.2 P0｜成员移除（管理员）✅

**产品语义**：让"管理员"角色真正具备最小可用的管理能力。MVP 不做复杂权限矩阵，只做"移除"这一个动作。

| 规则 | 说明 |
|------|------|
| 谁能操作 | 仅 `role === "admin"` 的成员，成员行展示「移除」入口 |
| 不能移除谁 | 不能移除自己；不能移除家庭创建者（`family.createdBy`） |
| 二次确认 | 移除是破坏性操作，必须二次确认（含被移除人名字的确认文案） |
| 移除效果 | 删除其 `familyMembers` 记录；该用户回到"创建/加入家庭"状态；其历史完成记录（`TaskCompletionLog`）是否保留见 §7.1 |
| 普通成员视角 | 普通成员看不到任何成员的「移除」入口，列表保持只读 |

**MVP 明确不做**：转让管理员、邀请审批、成员权限分级、成员备注。记入 §10 后续候选。

### 4.3 P1｜重置邀请码（管理员）✅

**产品语义**：给永久有效的邀请码一个"撤销"开关，防止旧码被无限转发。

| 规则 | 说明 |
|------|------|
| 谁能操作 | 仅管理员可见「重置邀请码」入口 |
| 行为 | 生成一个新的唯一邀请码，旧码立即失效 |
| 二次确认 | 重置前确认（提示"旧邀请码将立即失效，已发出去的旧码无法再加入"） |
| 反馈 | 重置成功后页面展示新码，并提示重新分享 |
| 复用 | 复用 `families.ts` 既有的 `generateInviteCode` + 唯一性校验逻辑 |

### 4.4 P2｜"收不到提醒？"排障指引 ✅

**产品语义**：iOS PWA Web Push 链路脆弱（需加主屏、需授权、需 standalone），失败时用户需要一个自查清单，而不是放弃。

入口放在「通知与应用」组通知卡下方，点击展开一个静态指引（无需后端），至少覆盖：

1. 是否已把网页"添加到主屏幕"并从主屏图标打开（而非 Safari 标签）
2. 是否已在系统弹窗中"允许通知"
3. 系统设置里该 App 的通知是否被关闭
4. 仍不行时：待办页始终是兜底，所有到期任务都在那里可见

### 4.5 P2｜版本号 / 关于 / 反馈 ✅

「通知与应用」组末尾展示：应用版本号（取构建版本，只读）、一句话"关于"（家庭内部植物养护提醒工具）。反馈入口 MVP 可先做静态（如一句"问题反馈请联系家庭管理员"或留空挂载位），不强求做工单系统。

## 5. 文案与信息层级清理

### 5.1 决策：全页只保留一个 h1 ✅

- 页面顶部新增页面标题 **「设置」（h1）**。
- 各卡片标题统一降为 **h2**，与待办/植物页的页面标题层级保持一致。
- 家庭摘要卡的家庭名从 h1(24px) 降为卡内信息（可仍较大，但不抢页面标题层级）。

### 5.2 决策：消除重复命名 ✅

- 成员卡当前 eyebrow「家庭成员」+ h2「家庭成员」重复，**去掉其一**：保留一个语义清晰的标题（如 eyebrow「家庭」+ h2「成员（N）」）。
- 邀请码卡 eyebrow「邀请码与分享」+ h2「把这串邀请码发给家人」语义重叠度高，可精简为 eyebrow「邀请码」+ 一句行动文案。

### 5.3 文案原则

- 沿用 MVP 视觉规范的"家庭语言"原则：口语、温度、第二人称，不用系统术语（如不说"鉴权失败"，说"登录状态过期了，请重新登录"）。
- 破坏性操作的确认文案必须**点名后果**：移除成员说清"TA 将离开这个家庭，需要重新邀请才能回来"；重置邀请码说清"旧码立即失效"。

## 6. 交互与状态规范

### 6.1 加载态

- 整页 summary 加载时沿用现有文案风格：「正在同步家庭信息…」，不使用骨架屏（数据量小，加载快）。

### 6.2 破坏性操作确认（统一规范）

| 操作 | 确认形式 | 确认按钮文案 | 取消 |
|------|---------|-------------|------|
| 退出登录 | 轻确认（底部 sheet 或弹窗） | 「退出登录」 | 「取消」 |
| 移除成员 | 二次确认（含名字） | 「移除 XXX」 | 「取消」 |
| 重置邀请码 | 二次确认 | 「重置邀请码」 | 「取消」 |

确认形式的具体视觉（sheet / dialog）由 UI Spec 统一，但三者必须风格一致。

### 6.3 权限态

- 普通成员进入设置页：看不到「移除成员」「重置邀请码」入口，且不应有任何"按钮被禁用置灰"的暗示（直接不渲染，避免引导用户去点不能点的东西）。
- 管理员进入：正常展示全部管理入口。

### 6.4 空 / 异常态

| 场景 | 处理 |
|------|------|
| 家庭只有 1 个成员（自己） | 成员列表正常展示自己；邀请码卡文案可强调"还没有家人加入，把邀请码发出去吧" |
| 邀请码复制失败 | 沿用现有 fallback（自动展开可选中区域 + 长按提示），不回归 |
| 修改称呼 / 移除 / 重置 失败 | 行内错误态文案，不静默失败 |
| 通知 unsupported / needs_install | 沿用现有三态 + needs_install 兜底，不回归 |

## 7. 边界场景与数据影响

### 7.1 移除成员的数据处理

- 删除该成员的 `familyMembers` 记录。
- **历史完成记录（`TaskCompletionLog`）保留**：完成记录属于"植物养护历史"，不应因为人走了就丢失（也为后续养护历史时间线功能保底）。被移除人的名字若已不可解析，展示降级文案（如"已离开的成员"）。
- 被移除用户下次打开应用：回到"创建 / 加入家庭"流程，不报错崩溃。

### 7.2 创建者保护

- `family.createdBy` 对应的成员不可被移除（即使是另一个管理员操作）。这是 MVP 阶段避免"家庭无主"的最简兜底。MVP 不做创建者主动转让，记入后续候选。

### 7.3 单一家庭约束

- 沿用 MVP「一个用户只属于一个家庭」约束。设置页不提供"切换家庭""退出家庭后重新选择"的复杂流程；用户如需换家庭，走"被移除 / 重新加入"路径。

## 9. 验收标准

本次迭代完成后，至少满足以下结果：

1. 设置页按「个人 / 家庭 / 通知与应用」三组分区展示，全页仅一个 h1「设置」。
2. 用户可在设置页修改自己的称呼，保存后成员列表中"我"的显示名实时更新。
3. 管理员可移除非创建者、非自己的成员，移除前有含名字的二次确认；普通成员看不到移除入口。
4. 管理员可重置邀请码，重置后旧码立即失效、页面展示新码，重置前有二次确认。
5. 通知卡下方提供"收不到提醒？"排障指引，点击可展开自查清单。
6. 设置页展示应用版本号与一句话关于信息。
7. 退出登录有二次确认，且仍位于明确可见、不被埋深的位置。
8. 成员卡不再出现 eyebrow 与标题重复命名；邀请码复制失败 fallback 不回归。
9. 移除成员后，相关历史完成记录不丢失；被移除用户重新打开应用回到加入家庭流程，不崩溃。

## 10. 后续版本候选

以下内容明确不在本次迭代范围，记录备查：

- 转让管理员 / 创建者主动移交
- 成员邀请审批（而非凭码即入）
- 成员权限分级与更细的权限矩阵
- 切换家庭 / 一个用户加入多个家庭
- 深色模式、语言切换等应用级偏好
- 数据导出 / 家庭整体删除
- 完整的反馈工单系统
- 归档植物的管理入口收口到设置页
