# 植物列表页产品设计细化（迭代 v0.2）

**版本**：v0.2（迭代设计）
**日期**：2026-06-11
**关联文档**：基于 [2026-06-05-mvp-requirements.md](./2026-06-05-mvp-requirements.md) 第 5.2 节「植物管理」的细化与扩展
**文档定位**：本文档不重复 MVP 全量需求，只针对**植物列表页（`/plants`）**这一个中频核心页面，把展示逻辑、排序规则、归档策略、卡片信息架构、交互模式与动效原则讲清楚，作为下一轮迭代的设计依据。

---

## 0. 设计锚点（先统一认知）

用 JTBD 视角，用户来到植物列表页时雇佣它完成的真实任务**不是**「浏览一个好看的植物相册」，而是：

> **"我家里有哪些植物、哪些最近需要关注、我能快速找到某一盆并进去看详情或调整。"**

由此推导出植物列表页的三个核心价值：

1. **资产总览** —— 一眼扫完家里所有活跃植物的状态全貌。
2. **快速识别** —— 通过图片 + 名称在 1-2 秒内认出目标植物。
3. **状态感知** —— 不进详情就能知道哪盆植物最近有养护压力。

植物列表页是**中频浏览页**（不像待办页每天多次打开），其设计原则是**信息密度适中、识别效率优先、操作路径短**。

## 1. 页面角色定位

### 1.1 与待办页的关系

| 维度 | 待办页 `/todo` | 植物列表页 `/plants` |
|------|---------------|---------------------|
| 核心任务 | 今天该做什么 → 立即执行 | 家里有什么 → 浏览管理 |
| 使用频率 | 高频（每天多次） | 中频（每周数次） |
| 信息粒度 | 任务级（浇水、施肥） | 植物级（整株状态） |
| 主动作 | 完成任务 | 进入详情 / 添加植物 |
| 设计倾向 | 执行效率 | 识别效率 |

### 1.2 用户来到此页面的典型场景

1. **找到某株植物** → 点击进入详情页查看任务或编辑信息
2. **添加新植物** → 家里新买了一盆，需要录入
3. **扫一眼整体状态** → 哪些植物最近养护压力大
4. **管理归档** → 偶尔查看已归档植物、恢复误归档

## 2. 列表展示逻辑

### 2.1 数据范围

| 区域 | 数据口径 | 是否默认展示 |
|------|---------|-------------|
| **活跃植物** | `isArchived === false` | 是（主区） |
| **已归档植物** | `isArchived === true` | 否（折叠入口，点击展开） |

### 2.2 默认排序规则（已拍板）✅

**决策：默认按「最近到期任务时间」升序排列。**

排序逻辑：

- 一级排序：`nextDueTask.nextDueAt` 升序（最紧迫的排最前）
- 无任务的植物沉底（排在所有有任务植物之后）
- 无任务植物之间：按创建时间降序（最新添加的靠前）

**设计理由**：

- 用户的心智模型是「哪盆最近该照顾了」，不是「按拼音找植物」
- 与待办页形成呼应但不重复——待办页是「今天该做什么」，植物列表是「哪些植物最近需要关注」
- 搜索框已覆盖「按名称找特定植物」的需求，排序不必为此服务

**备选方案（已否决）**：

- 按名称字母排序：工程师默认选择，不符合用户心智
- 按创建时间排序：对日常使用无信息价值
- 提供排序切换器：家庭场景植物数量有限（5-15 盆），多排序模式属过度设计

### 2.3 搜索

保留当前搜索框，匹配范围为：植物名称、摆放位置、描述。

搜索为辅助手段，不是主要浏览路径。当植物数量 < 5 时，搜索框可视觉弱化（但不隐藏，保持一致性）。

### 2.4 分组：MVP 不做

| 分组维度 | 结论 | 说明 |
|---------|------|------|
| 按位置分组（客厅/阳台） | ❌ 不做 | 家庭植物通常 5-15 盆，分组增加认知层级但收益有限 |
| 按养护状态分组 | ❌ 不做 | 排序已解决「紧迫感知」问题，分组会让列表碎片化 |
| 按标签/类型分组 | ❌ 不做 | MVP 无标签体系，不引入新概念 |

**后续版本候选**：当植物超过 20 盆时，可考虑按位置分组 + 组内按到期排序。

## 3. 归档植物策略

### 3.1 归档的产品语义

归档 ≠ 删除。归档代表「这盆植物暂时不需要日常养护了」（送人了、休眠期、搬走了），但数据保留，随时可恢复。

### 3.2 归档植物的展示位置

**决策：列表底部折叠入口。** ✅

```
┌─────────────────────────────────┐
│  [活跃植物列表]                   │
│  ...                            │
│  ...                            │
├─────────────────────────────────┤
│  ▶ 已归档 (3)                    │  ← 默认收起
└─────────────────────────────────┘
```

点击展开后：

```
├─────────────────────────────────┤
│  ▼ 已归档 (3)                    │  ← 展开
│  ┌───────────────────────────┐  │
│  │ 薄荷 · 2026-05-20 归档     │  │  ← 灰态卡片
│  │              [恢复]        │  │
│  └───────────────────────────┘  │
│  ...                            │
└─────────────────────────────────┘
```

### 3.3 归档卡片的视觉处理

- 整体降低饱和度（opacity 0.6 或灰度滤镜）
- 不显示任务状态 pill（归档后任务无意义）
- 显示归档时间
- 提供「恢复」操作按钮
- 不提供「编辑」入口（恢复后再编辑）

### 3.4 归档操作的触发位置

归档操作不在列表页触发，而是在植物详情页内操作（当前已实现 `ArchivePlantAction` 组件）。列表页只负责展示归档结果和恢复入口。

## 4. 卡片信息架构

### 4.1 当前问题诊断

从现有实现和截图分析，当前卡片存在以下产品问题：

| 问题 | 影响 | 严重度 |
|------|------|--------|
| 图片占比过大（16:10），单卡几乎占满一屏 | 信息密度低，需大量滚动 | 高 |
| 「查看详情」按钮冗余 | 用户犹豫「点卡片还是点按钮」 | 中 |
| 「编辑」按钮与主动作平级 | 低频操作争夺注意力 | 中 |
| 名称、位置、任务状态层级不够分明 | 快速扫描时信息获取效率低 | 中 |

### 4.2 卡片设计决策

**决策 1：整卡可点击，去掉「查看详情」按钮。** ✅

- 点击卡片任意区域 → 进入植物详情页
- 这是移动端通用心智模型，无需显式按钮引导
- 卡片应有点击态反馈（按压缩放），暗示可交互

**决策 2：「编辑」入口降级为右上角图标按钮。** ✅

- 编辑是低频操作（录入后很少改），不应在列表页与主动作争夺注意力
- 使用铅笔图标（16×16），半透明，悬浮在图片区右上角
- 点击编辑图标 → 直接进入编辑页，不进详情页
- 图标需要足够的点击热区（最小 44×44），但视觉面积小

**决策 3：采用紧凑横向布局（方案 A）。** ✅

```
┌──────────┬──────────────────────────────┐
│          │  发财树                        │  ← 名称 16px 加粗
│  植物图片  │  📍 客厅茶几                   │  ← 位置 12px 灰色
│  1:1 方图 │                               │
│  80×80   │  💧 浇水 · 明天到期             │  ← 任务状态行
│          │                               │
└──────────┴──────────────────────────────┘
```

**设计理由**：

- 一屏可展示 4-5 株植物，信息密度提升约 2.5 倍
- 图片缩小为 1:1 方图（80×80pt），仍然足够识别「这是哪盆」
- 左图右文的横向布局符合中文阅读习惯（从左到右扫描）
- 信息层级清晰：名称（主）→ 位置（辅）→ 任务状态（状态）

**备选方案（已否决）**：

- 方案 B（竖向卡片压缩版）：虽然保留了图片主导感，但一屏仍只能看 2-3 株，对于「资产管理」页面信息密度不够
- 当前方案（大图 + 双按钮）：信息效率最低，已分析问题

### 4.3 卡片信息元素规范

| 元素 | 是否必须 | 视觉规格 | 说明 |
|------|---------|---------|------|
| 植物图片 | 是 | 80×80pt，圆角 12px，`object-fit: cover` | 识别锚点 |
| 植物名称 | 是 | 16px，`--font-heading`，`--color-ink`，font-weight 600 | 主信息 |
| 摆放位置 | 条件展示 | 12px，`--color-muted`，前缀 📍 | 有值才展示 |
| 任务状态行 | 条件展示 | 12px，任务类型 emoji + 类型名 + 到期文案 | 有任务才展示 |
| 无任务提示 | 条件展示 | 12px，`--color-muted`，文案「还没有养护任务」 | 无任务时展示 |
| 编辑图标 | 是 | 16×16 图标，热区 44×44，右上角绝对定位 | 低频辅助操作 |

### 4.4 无图状态

当植物没有上传图片时：

- 图片区域显示统一占位：浅绿底色（`--color-mist`）+ 居中 🌿 emoji（32px）
- 不显示「轻触上传照片」文案（列表页不是上传入口）
- 占位态与有图态尺寸完全一致，不影响布局

### 4.5 任务状态行的文案规则

| 场景 | 展示文案 | 颜色 |
|------|---------|------|
| 有任务且已逾期 | `💧 浇水 · 逾期 2 天` | `--color-warning` |
| 有任务且今天到期 | `💧 浇水 · 今天到期` | `--color-ink` |
| 有任务且明天到期 | `💧 浇水 · 明天到期` | `--color-muted` |
| 有任务且 2+ 天后到期 | `💧 浇水 · 后天到期` 或 `💧 浇水 · 3 天后到期` | `--color-muted` |
| 无任务 | `还没有养护任务` | `--color-muted` |

任务类型 emoji 映射：

| 类型 | emoji |
|------|-------|
| watering | 💧 |
| fertilizing | 🧪 |
| misting | 🌫️ |
| repotting | 🪴 |
| pruning | ✂️ |
| custom | 📋 |

## 5. 交互规范

### 5.1 点击行为

| 点击区域 | 行为 | 说明 |
|---------|------|------|
| 卡片主体（图片 + 文字区） | 进入植物详情页 `/plants/:id` | 主交互路径 |
| 编辑图标 | 进入植物编辑页 `/plants/:id/edit` | 辅助操作，需阻止事件冒泡 |

### 5.2 点击反馈

- 按下时：卡片整体 `scale(0.98)`，过渡时间 120ms
- 松开时：恢复 `scale(1)`，过渡时间 200ms（略慢于按下，有弹性感）
- 编辑图标按下时：图标 opacity 从 0.5 → 1.0

### 5.3 列表加载态

- 首次加载时显示文案：「正在同步家庭中的植物…」
- 不使用骨架屏（植物数量少，加载通常 < 500ms，骨架屏闪烁反而干扰）

### 5.4 空状态

当家庭中没有任何植物时：

- 标题：「你的家庭植物看板还是空的」
- 描述：「先添加第一盆植物，后续才能继续配置养护提醒。」
- 主按钮：「添加植物」

当搜索无结果时：

- 标题：「没有找到匹配的植物」
- 描述：「试试输入植物名称，或者它所在的位置。」

## 6. 动效原则

### 6.1 设计约束

- 所有动效必须尊重 `prefers-reduced-motion`
- 动效目的是「确认感」和「流畅感」，不是「惊喜感」
- 这是每天打开的工具，任何「第一次觉得酷但第十次觉得慢」的动效都是负分

### 6.2 允许的动效

| 动效 | 触发时机 | 规格 | 目的 |
|------|---------|------|------|
| 列表渐入 | 页面首次加载完成 | 卡片依次 fade-in + translateY(8px→0)，stagger 50ms，总时长 ≤ 300ms | 页面「活」的感觉 |
| 按压缩放 | 手指按下卡片 | scale 1→0.98，120ms ease-out | 触觉确认 |
| 图片渐显 | 图片加载完成 | opacity 0→1，200ms | 避免图片突然闪现 |

### 6.3 不做的动效

- 卡片悬浮抬起 / 大阴影变化
- 列表项拖拽排序
- 翻转、弹跳、3D 变换
- 页面切换时的复杂转场

## 7. 边界场景与异常处理

### 7.1 植物数量极端情况

| 场景 | 处理方式 |
|------|---------|
| 0 株植物 | 显示空状态（§5.4） |
| 1-5 株 | 正常列表，搜索框视觉存在但不突出 |
| 6-15 株 | 正常列表，搜索框正常权重 |
| 15+ 株 | 正常列表，后续版本可考虑分组 |

### 7.2 图片加载失败

- 先尝试重新获取 URL（Convex storage URL 有过期机制）
- 若仍失败，回落到占位态（🌿）
- 不允许出现浏览器默认坏图图标

### 7.3 任务数据异常

- 植物有任务但 `nextDueAt` 为空或无效：显示「还没有养护任务」，不崩溃
- 多个任务同时到期：只展示最紧迫的那一条（当前逻辑已如此）

## 8. 与现有实现的差异对照

| 维度 | 当前实现 | 本次迭代目标 |
|------|---------|-------------|
| 排序 | 按名称字母排序 | 按最近到期时间升序 |
| 卡片布局 | 竖向大图（16:10）+ 下方信息 | 横向紧凑（左图 80×80 + 右文） |
| 操作按钮 | 「查看详情」+「编辑」并排 | 整卡可点击进详情，编辑降为图标 |
| 归档植物 | 完全不可见 | 底部折叠入口 |
| 点击反馈 | 无 | 按压缩放 0.98 |
| 列表动效 | 无 | 渐入 stagger |

## 9. 验收标准

本次迭代完成后，至少满足以下结果：

1. 植物列表默认按最近到期任务时间排序，最紧迫的排最前面。
2. 一屏（iPhone 标准视口 390×844）可见至少 4 株植物的完整卡片信息。
3. 点击卡片任意区域可进入植物详情页。
4. 编辑入口为右上角图标，不与主点击区混淆。
5. 归档植物在列表底部有折叠入口，展开后可恢复。
6. 卡片按下时有 scale 缩放反馈。
7. 无图植物显示统一占位态，不出现浏览器默认坏图。
8. 逾期任务在卡片状态行以警告色展示。

## 10. 后续版本候选

以下内容明确不在本次迭代范围，记录备查：

- 按位置分组展示（等植物 > 20 盆再考虑）
- 排序模式切换（按名称 / 按位置 / 按到期）
- 卡片长按菜单（归档、删除等快捷操作）
- 植物标签体系
- 列表视图 / 网格视图切换
- 拖拽自定义排序
