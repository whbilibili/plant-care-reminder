## P1 — 体验增强层（建议 2~3 个迭代后启动）

### P1-1 多图上传与养护图集

**核心诉求**：MVP 每盆植物只能上传一张照片作为封面，用户无法记录植物的生长变化。缺乏"生长日记"感。

**初步方案**：

- 植物详情页增加"图集"区域，支持多图上传（建议上限 20 张/盆植物）。
- 图片按上传时间排列，可左右滑动浏览。
- 完成任务时可选附加照片（与养护历史时间线 v1.1 联动）。
- 技术要点：复用 Convex `_storage` 文件存储，plants 表增加 `imageStorageIds: v.optional(v.array(v.id("_storage")))` 字段。
- 需关注存储容量和加载性能（缩略图 + 原图分离）。

**前置依赖**：无硬性依赖，但建议在养护历史时间线（P0-1）完成后做，两者可联动。

### P1-2 养护任务灵活性

**核心诉求**：MVP 的养护任务只有"固定间隔天数"一种排期模式，缺乏灵活性。用户无法处理"每周一三五浇水"或"冬天减半频率"的需求。

**初步方案**：

- 排期模式扩展：固定间隔（现有）、每周几（weekly with specific days）、季节性调整（夏季 N 天、冬季 M 天）。
- schema 变更：`plantTasks` 表的 `intervalDays` 扩展为更灵活的调度结构，需考虑向后兼容。
- 自定义任务类型：除预设的浇水/施肥/喷雾/转盆外，允许用户自定义任务类型名称。
- 前端：任务创建/编辑弹窗增加排期模式选择器。

**前置依赖**：需谨慎设计 schema 迁移策略，避免影响已有任务的调度计算。

### P1-3 数据导出

**核心诉求**：用户（特别是专业植物爱好者）希望导出自己的养护数据用于分析或备份。

**初步方案**：

- 设置页增加"导出数据"入口。
- 导出格式：CSV（养护日志）+ JSON（完整数据备份）。
- 导出范围：当前家庭的所有植物信息 + 任务配置 + 完成日志。
- 技术要点：Convex action 生成文件 → 存入 `_storage` → 前端下载链接。
- 隐私考量：只导出当前用户所属家庭的数据，不含其他家庭成员的个人信息（邮箱等）。

**前置依赖**：养护历史时间线（P0-1）完成后导出数据更有价值。

## P2 — 长期增值层（视用户反馈决定是否推进）

### P2-1 植物知识卡片

**核心诉求**：用户添加新植物时，不确定浇水频率和养护方式该怎么设置。缺乏植物养护知识的引导。

**初步方案**：

- 预置常见植物品种的养护指南数据库（浇水频率建议、光照需求、施肥建议等）。
- 创建植物时输入品种名后，自动推荐养护参数。
- 植物详情页展示该品种的知识卡片（养护小贴士）。
- 数据来源：初期可预置 50~100 种常见绿植数据（JSON 静态数据），后期可接 API。
- 权衡：维护植物知识库的长期成本 vs 用户价值。

### P2-2 季节性养护建议

**核心诉求**：植物的养护需求随季节变化（夏天多浇、冬天少浇），MVP 没有季节意识。

**初步方案**：

- 在待办页或植物详情页，根据当前季节展示养护提示（"夏季高温，建议增加浇水频率"）。
- 与植物知识卡片联动：如果有品种信息，给出该品种的季节性建议。
- 与排期灵活性联动：支持按季节自动调整浇水间隔（需 P1-2 完成）。
- 轻量实现：前端硬编码季节规则 + 通用提示文案，不需要后端支持。

### P2-3 家庭协作增强

**核心诉求**：MVP 的家庭协作是"所有人看到所有任务"，没有任务分配和提醒转发。

**初步方案**：

- 任务指派：创建/编辑任务时可指定负责人（家庭成员之一）。
- 只有被指派的人收到该任务的推送通知（替代当前的全员推送）。
- 完成通知：成员 A 完成任务后，通知其他关注该植物的成员。
- 与角色权限联动：owner 可指派任务给任何成员。
- 前置依赖：推送通知增强（P0-2）和角色权限管理（已有 PRD v0.4）。

## 附：优先级判断依据

| 级别 | 判断标准 | 典型信号 |
|------|---------|---------|
| P0 | 数据已有但未展示、体验明显粗糙、实现成本低 | "完成了但看不到"/"被通知吵醒"/"找不到植物" |
| P1 | 功能可用但不够灵活、进阶用户有明确诉求 | "只能传一张图"/"排期不够灵活"/"想导出数据" |
| P2 | 锦上添花、需要外部数据或复杂联动 | "不知道该怎么养"/"希望分配任务" |
