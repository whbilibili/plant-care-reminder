# 排期功能重构提案：统一重复任务选择器

> 状态：待迭代（Backlog）
> 创建日期：2026-06-23
> 优先级：P2（体验优化）

## 背景

当前排期功能采用三种独立模式：

| 模式 | 交互 | 痛点 |
|------|------|------|
| 固定间隔 | 输入天数 | 基础能力，够用 |
| 每周固定 | 多选星期几 | 无法表达"每两周" |
| 季节适应 | 填春夏/秋冬天数 | 概念偏专业，普通用户理解成本高 |

核心问题：

- 三种模式是并列平级的 tab 切换，用户需要先理解三种模式的区别再做选择
- 无法表达"每两周一次""每月15号"等常见周期
- "季节适应"概念偏专业，普通用户难以理解何时用

## 目标

将三种排期模式合并为一个**统一的「重复」选择器**，类似市面主流 App（iOS 提醒事项、Todoist、滴答清单）的交互范式，降低认知负担、提升灵活性。

## 设计方向

### 交互方案：单一入口 + 渐进展开

```
┌─────────────────────────────────┐
│  重复频率                    ▼  │
├─────────────────────────────────┤
│  ○ 每天                         │
│  ○ 每 X 天          [  7  ] 天  │
│  ○ 每周    [选择星期几...]       │
│  ○ 每 X 周          [  2  ] 周  │
│  ○ 每月    [选择日期...]         │
│  ○ 季节适应（高级）              │
└─────────────────────────────────┘
```

用户点击"重复频率"后展开选项，选中一项后折叠显示简短描述。

### 数据模型（草案）

```typescript
// 统一的重复规则
type RepeatRule =
  | { type: "daily" }                              // 每天
  | { type: "everyNDays"; days: number }           // 每 N 天
  | { type: "weekly"; weekdays: number[] }         // 每周（选星期几）
  | { type: "everyNWeeks"; weeks: number; weekday: number } // 每 N 周的星期几
  | { type: "monthly"; dayOfMonth: number }        // 每月 X 号
  | { type: "seasonal"; springSummer: number; autumnWinter: number } // 季节适应
```

### 向后兼容策略

- 数据库保留现有 `scheduleMode` / `intervalDays` / `weeklyDays` / `seasonalIntervals` 字段
- 新增 `repeatRule` 字段（JSON 对象），新数据写入此字段
- 读取时优先取 `repeatRule`，fallback 到旧字段组合
- 迁移脚本将旧数据统一转写为 `repeatRule` 格式

### 显示文案规则

| 规则 | 显示文案 |
|------|---------|
| 每天 | 每天 |
| 每 7 天 | 每 7 天 |
| 每周一、三、五 | 每周一、三、五 |
| 每周全选 | 每天 |
| 周一到周五 | 每个工作日 |
| 周六周日 | 每个周末 |
| 每 2 周的周一 | 每两周（周一） |
| 每月 15 号 | 每月 15 号 |
| 季节适应 | 春夏 5 天 / 秋冬 14 天 |

## 参考竞品

- **iOS 提醒事项**：每天/每周/每两周/每月/自定义，周选可多选星期几
- **Todoist**：自然语言输入 + 结构化选择器，支持"每个工作日""每3天"
- **滴答清单**：每天/每周/每月/每年/自定义，自定义可设间隔+星期几
- **Planta（植物App）**：固定天数 + 季节调整系数，自动随季节变化

## 风险与注意事项

- 排期算法 `computeNextDueAtByMode` 需要同步重构以支持新规则类型
- Cron 扫描逻辑需兼容新旧两套数据格式
- 需要确保已有任务的到期时间在迁移前后保持一致
- "季节适应"可作为高级选项保留，但降低展示优先级
